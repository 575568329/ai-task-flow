# AI 协作开发通用原则（从澳门国际化二期复盘提炼）

> 来源：rpp-web 澳门国际化二期复盘（2026-06-15~06-30），7个会话约500条真实交互
> 目的：沉淀 AI 开发的**通病**和**普适性原则**，指导所有项目的 AI 协作

---

## 一、AI 的结构性盲区（不是偶发失误，是设计特性）

### 1. AI 只改被指到的文件，不会自动补全"模块全集"

**表现**：
- 让 AI "改造个人中心"，给了 3 个文件，AI 只改这 3 个
- 同模块下的其他 2 个页面（my-benefits / my-homepage）完全不进入视野
- 菜单标签批量翻译了，制造"已完成"假象，但页面内部整页未翻

**根因**：
- AI 是**被动驱动**（你指哪改哪）+ **按文件流**（逐文件处理，无全局视角）
- "边发现边改"必然遗漏——只翻手头碰到的文件，碰不到的永远看不见

**通用原则**：
> **批量改造第一步：先列清单，再执行**
> 
> 让 AI 做"某模块全量改造"（国际化/重构/字段迁移/批量升级等），第一步必须输出**受影响文件清单**，经你确认后再逐项执行。禁止"边发现边改"。
>
> 清单来源优先级：**路由子树** > 目录树 > 引用扫描。
>
> 已写入全局 `CLAUDE.md` 1.8 节。

**案例**：
- 个人中心 5 页只翻 3 页，漏了 2 页
- T6 任务详情路由分析：路由在 task-list，实际页面在 order-detail，AI 多次分析错对象

---

### 2. AI 对"覆盖率"无感知——卡口只管"对不对"，管不了"全不全"

**表现**：
- `i18n-check.js` 卡口能抓 key 错位、漏翻、漏包 `$tm`
- 但对"整页 0 翻译"（无任何 i18n 痕迹）完全失明
- 用户切语言后点进去整页中文，AI 说"卡口通过，没问题"

**根因**：
- 静态卡口只检查"你写的代码对不对"，检查不了"你该写的都写了没"
- 整页漏翻是**范围/计划问题**，属工作流，不属代码正确性

**通用原则**：
> **工具分工：卡口管正确性，覆盖率靠人工清单**
>
> - 卡口 / lint / 自动化测试 → 回答"已写的代码对不对"
> - 覆盖率 / 完整性 → 靠**开工前枚举清单 + 验收时逐项勾选**
> 
> 试图让卡口扫"哪些该做没做"会导致：靠启发式猜测（扫中文/扫注释）、噪声大、越界误报。

**推论**：AI 看不见"没写的代码"。你得告诉它"哪 10 个页面要改"，不能只给它"已改的 3 个文件"然后期待它发现漏了 7 个。

---

### 3. AI 会"过度信任局部信息"——看到部分符号就下结论

**表现**：
- 看到菜单 5 个标签全翻了 → 判断"个人中心已完成"
- 看到 `rppCommonService` 变量名 → 推断接口在 `rpp` 目录，实际 import 自 `commonService`
- 看到 `errorType` 中文判断 → 推断"死代码"，实际是 SDK 返回值

**根因**：
- AI 基于"当前视野内的证据"推理，视野外的矛盾信息它看不见
- 人类会本能警觉"只看到一半"，AI 不会

**通用原则**：
> **关键断言必须看完整依据链**
>
> - 变量来源：看 import 路径，不看变量名
> - 接口契约：看实际返回值/SDK 文档，不看本地注释
> - 死代码判断：看调用链，不看"这段逻辑不合理"
> - 部署事实：看 git log / nginx 配置，不靠代码逻辑推断
>
> AI 说"应该是 X"时，追问"你在哪看到的？给我文件+行号"。

---

### 4. 调参式迭代的天花板——数字剧烈震荡是信号

**表现**：
- `BARE_RENDER` 检测命中数震荡：34 → 14 → 1 → 72
- 反复调正则窗口大小、字符距离阈值
- 每次调完"好像对了"，一跑又冒出新的误报/漏报

**根因**：
- 正则 / 启发式规则做"跨变量数据流追踪"本质脆弱（无作用域概念）
- **命中数剧烈震荡本身就是"工具选型触及天花板"的信号**

**通用原则**：
> **调参 2 轮仍震荡 → 停止调参，诊断架构缺陷**
>
> 震荡说明工具维度不够（如正则查不出"这个 item 来自常量还是后端"）。应该：
> 1. 换判定维度（改按单个 export 常量判定、用 AST、限定作用域）
> 2. 缩小工具职责（只查主流写法，极端写法留给人工）
> 3. 接受"这类问题工具查不全"，配人工审阅
>
> 不要陷入"再调一次参数就能完美"的幻觉。

## 二、AI 的行为模式陷阱

### 5. 过度谨慎绕圈 vs 过度自信臆断

**过度谨慎表现**：
- 改编题型校验数据源：在 getSections / cbb 内部 / 自请求之间反复追问 7~8 轮
- 找按钮归属：label 由 slot 父组件传入的假象，横跳多轮才定位
- 本可以"先试一个方案，错了再改"的场景，非要全部澄清才动手

**过度自信表现**：
- 没查 git log 就断言"nginx 漏了 `/dist/` 路径"
- 看到 4 个兄弟组件都保留中文判断，就说 custom-uploader "死代码"（实际是误改）
- 没 Read 文件就说"冒号都正确"（实际 70 处错）

**通用原则**：
> **给 AI 明确决策边界**
>
> - 低风险（改一个文件、加一个判断）→ 告诉 AI "先试，错了再改"
> - 高风险（删数据、改架构、断言线上行为）→ "给我依据（文件+行号 / git log），我来判断"
>
> AI 追问 3 轮还在绕圈 → 你直接给个 URL / 文件路径 / 具体值，比让它继续推理高效。

---

### 6. 顺手重构 / 过度设计

**表现**：
- DownloadCenter 列定义改 computed（实际切语言会 reload，data() 一次性求值就够）
- Condition 组件加行内样式（破坏公共组件纯净性）
- 删掉看似"无用"的 import（constantApprove，实际其他分支在用）
- Tipo 列手写 tooltip（el-table 自带，多此一举）

**通用原则**：
> **明确"只解决当前问题，不重构周边代码"**
>
> 在任务描述里写：
> - "只翻译 XX 页面，不要动其他文件"
> - "只修这个 bug，看起来可以改进的地方也别动"
>
> AI 说"顺便优化了 YY" → 立即叫停。顺手清理反而留坑（半清理、破坏一致性）。

---

### 7. 子代理结论需人工复核

**表现**：
- 子代理报"冒号都正确"，实际 70 处错
- 子代理说"BookTypeList 应该是 key"，实际存的是中文"同步教材"
- 子代理用"应该是 / 具体值未见"这类猜测语气

**通用原则**：
> **子代理 = 初筛工具，关键断言必须主会话核查**
>
> 子代理适合：大范围扫描、并行重复劳动、低风险判断。
> 
> 主会话必做：关键 bug 确认（Read 文件看真实值）、提交前 diff 复核、架构决策。
>
> 子代理说"没问题"不等于真没问题，尤其当它用"应该 / 可能"时。

## 三、协作纪律（AI 反复犯的错）

### 8. 提交纪律

**二期反复被批评的点**（适用所有项目）：
- ❌ 擅自提交（"我改完顺手提交了"）
- ❌ 提交混入无关文件（config/index.js、package-lock.json、临时脚本）
- ❌ commit message 带 `Co-Authored-By: Claude` / `opus-4.8` 署名
- ❌ 提交前不看 `git diff`，暂存用 `git add .`

**通用原则**：
> **默认：改完不提交，等明确指令**
>
> 提交前三步：
> 1. `git status` 逐文件看，确认都是该提交的
> 2. `git diff` 看改动，确认无误改 / 多改
> 3. 精确暂存（`git add <具体文件>`），不用 `git add .`
>
> 排除项（项目相关，二期是这俩）：`config/index.js`、`package-lock.json`。
>
> commit message 格式按项目约定，不带 AI 工具署名。

---

### 9. 需求反复 / 返工的预防

**二期案例**：
- 跨应用语言传递：做了 Vue↔Angular 双向 `?lang=`，跨 2 仓库 4 文件 → 3 天后"不维护其他应用"全撤回
- 登录页营销区：全译 → 锁中文，与原计划矛盾需回填文档
- 冒号格式：半角 → 半角+空格 → 全角，三轮返工

**通用原则**：
> **跨系统 / 高耦合改动，先确认对方配合意愿**
>
> 问自己：
> - 这个方案依赖其他团队/系统改动吗？（如 Angular 端配合）
> - 对方答应配合了吗？有没有"口头说配合，实际不改"的风险？
> - 撤回成本多大？（跨仓库 / 多处联动的，撤回成本指数级增长）
>
> 宁可前期多确认 2 天，也别做了 3 天再全撤。

---

### 10. 需求澄清：什么时候该追问 vs 什么时候该先试

**该追问的**（追问成本 < 试错成本）：
- 跨系统契约（Angular 端会不会配合？后端返回值格式？）
- 业务规则（这页锁不锁中文？品牌名翻不翻？）
- 用户偏好（冒号用全角还是半角？简化翻译到什么程度？）

**该先试的**（试错成本低、能快速验证）：
- 布局调整（label 宽度多少合适 → 先给 120px，不行再调）
- 数据源定位（getSections 还是 cbb → 先读一个，不对再读另一个）
- 正则优化（窗口大小 → 先试 50，误报多再调）

**通用原则**：
> **高风险 / 跨边界 → 问清楚再做；低风险 / 本地验证 → 先试快速迭代**
>
> 如果 AI 追问超过 3 轮还在"可能是 A 也可能是 B"→ 你直接给答案或让它"先试 A"。

## 四、工具边界认知

### 11. 卡口的五类盲区

从 `i18n-check.js` 推广到所有静态检查工具：

| 能检查 | 检查不了 |
|--------|---------|
| ✅ 路径错位（key 不存在） | ❌ 翻译文案做逻辑判断（值与后端不一致） |
| ✅ 四语言缺 key | ❌ 语言决策逻辑错（该澳门的用了页面语言） |
| ✅ 常量存 key 漏包 `$tm` | ❌ 动态拼接 key（`` `prefix.${type}` ``） |
| — | ❌ 整页未国际化（0 个 `$tm`，无痕迹） |
| — | ❌ 翻译质量（译文不通顺、不准确） |

**通用原则**：
> **每类工具划清"管什么 / 不管什么"**
>
> - 静态卡口：管语法正确性、引用完整性
> - 覆盖率：靠枚举清单
> - 语义正确性：靠人工审阅、单测、真实环境验证
>
> 别指望一个工具解决所有问题。卡口盲区 → 用其他手段补（专项脚本、人工抽查、E2E）。

---

### 12. 子代理 / 并行的使用边界

**适合子代理的**：
- 大范围扫描（57 个提交逐个看 diff）
- 并行重复劳动（3 个子代理各审查一类问题）
- 低风险判断（这个文件改了国际化吗？列出 key 数量）

**不适合子代理的**：
- 架构决策（语言决策集中到哪个函数？）
- Bug 确认（这是不是真 bug？读文件看实际值）
- 提交操作（git diff / add / commit，必须主会话人工复核）

**通用原则**：
> **子代理 = 扩展视野的工具，不是替代你判断的大脑**
>
> 派子代理时带"把结果给我，我来判断"的预期，不是"你帮我决策"。
>
> 子代理报告里的数字/结论，关键的要主会话抽查核实。

---

## 五、总结：给 AI 的三个"不要指望"

1. **不要指望 AI 自动补全"全集"** → 你得给清单
2. **不要指望 AI 发现"没写的代码"** → 覆盖率靠你枚举
3. **不要指望工具解决所有问题** → 卡口 + 人工 + 流程，三层防御

---

## 附：本原则清单的使用场景

- ✅ 开工前：提醒自己"先列清单"、"明确卡口边界"
- ✅ AI 绕圈时：判断是过度谨慎（给具体值）还是工具天花板（换维度）
- ✅ 验收时：检查"覆盖率清单勾完没"、"卡口盲区人工查了没"
- ✅ 复盘时：对照"这次踩的坑，哪条原则能预防"
