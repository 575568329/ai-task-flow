## 🔧 可复用的技术方案

### 静态 i18n 卡口（基线增量策略）
- **适用场景**：存量项目引入 i18n 检查，但历史遗留问题多，不能一次性卡死全员提交。
- **做法**：卡口只校验"代码引用的 key 在基准语言存在 + 四语言齐全"，存量问题冻结进 baseline（`count=0` 表示最严），只拦新增。
- **效果**：渐进收紧，新代码不留债，老债不阻塞。

### 语言决策集中到单一汇聚点
- **适用场景**：多租户/多区域（如大陆 vs 澳门）需要不同语言策略。
- **做法**：所有语言判断收敛到一个函数（`resolveBizLocale`），按路由类型分流；工作室路由 + 工作室语言为空时**固定回退基准语言**，防止错误区域显示多语言。
- **效果**：审计时只需核汇聚点，不必逐调用点排查。

### 翻译长度导致的布局适配
- **适用场景**：中文→英/葡的固定宽度容器（label、表格列、tag、按钮）。
- **做法**：优先精简译文（教育语境可缩写）> `min-width`/`min-height`+`word-break`+字号缩小；样式用 `.lang-en-US`/`.lang-pt-BR` 类只作用目标语言，中文不动。容器宽度判断的语言源要与 `$tm` 渲染的语言源**同源**。
- **效果**：葡文（常比中文长 2~4 倍）不撑破布局，且不影响中文。

### 表头文本省略 + hover tooltip（Element UI）
- **做法**：mixin + `:render-header`，而非每列手写 slot 或全局 directive。注意 `slot="header"` 优先级高于 `render-header`，已有 slot 的列不会被覆盖。

## ⚠️ 避坑清单

- **当用翻译文案做逻辑判断时**（`变量 === $tm(...)`、`indexOf($tm(...))`）→ 查变量来源，若来自后端/SDK（不随语言变）则切语言必失效。判断一律用语言无关的 code/英文常量，`$tm` 只负责显示。**卡口查不出这类**。

- **当枚举/常量把 i18n key 存进 name 字段时** → 模板引用必须 `{{ $tm(item.name) }}`，直接 `{{ item.name }}` 会显示 key 字符串。这类是卡口 BARE_RENDER 检测的对象，但仍应改造时就注意。

- **当做"整个模块全量改造"（国际化/重构/迁移）时** → 第一步先按**路由子树**枚举全部页面列 checklist（不是靠"手头碰到的文件"），逐页勾选。AI 只改被指到的文件、不会自动补全模块全集，边发现边改必遗漏。

- **当菜单标签翻译完成时** → 不等于页面内容翻译完成。菜单和页面内容是两拨工作，菜单全翻会制造"已完成"假象，务必逐页验收页面内部。

- **当在 JS 文件里写 i18n 时** → 用 `i18n.t()`，不要传 `$tm` 方法引用进普通函数（会丢 `this` 导致 `this.$i18n` undefined）。

- **当用 `new Vue({...})` 创建独立弹窗实例时** → 必须手动 `new Vue({ i18n, ... })` 注入，否则 `$tm` 崩。

- **当组件 props 的 default 需要翻译时** → default 里拿不到 `this`，不能写 `this.$tm()`。用空字符串 default + template 里 `|| $tm(...)` 兜底。

- **当页面出现无限 loading 时** → 给整条 async 链路加统一前缀日志，最后一条日志即卡点。警惕 `async` + `.catch` 吞异常造成的"静默卡死"；在数据可能缺失处（树遍历 `for...of`、findRootNode 等）加空值保护。

- **当 label 后想加冒号间距时** → 用全角冒号 `：`，不要半角冒号+空格（HTML 会压缩连续空格）。

- **当占位符（`{type}` 等）翻译时** → 四语言必须都保留占位符，缺一个则传参被丢。葡文注意单复数性数一致（`{type} Disponíveis` 这类动态拼接易错），可改用动词不定式绕开性数变化。

- **当编辑 JSON 里的 i18n key 时** → 用解析（node 读 key 数）验证 section 归属，别凭肉眼数花括号边界，容易写错 section。

- **当推断部署路径/线上行为时** → 查提交历史（`git log -S`、看 HEAD 版本），不要基于代码逻辑臆断。部署事实用户/运维掌握。

- **当接口变量名和 import 不一致时** → 追接口看真实 import 路径，不看变量名（如 `rppCommonService` 实际 import 自 `commonService`）。

- **当遇到公式/特殊图片时** → `data-latex` 图片是题目内容不是装饰，删图=毁题。图片校验区分"无宽高"和"加载失败"两个条件。

## 📐 约定与规范

- **翻译出口**：Vue 组件用 `$tm()`，JS 文件用 `i18n.t()`，禁用 `$t()`。
- **不翻译的内容**：品牌名、符号（冒号/分隔符/括号/顿号）、后端返回的业务字段值、逻辑判断标识符、注释。
- **配置页锁中文**：复杂度高、ROI 低的配置/新建页，`$tm` 加 `{locale:'zh-CN'}` 锁定。
- **namespace 规划**：>100key 或引用>50次才独立 JSON，否则进 `_rest.json` 平铺。
- **提交纪律**：默认不提交，等用户明确指令；提交前 `git status` 逐文件核实、精确暂存自己改的；排除 `config/index.js` 和 `package-lock.json`；提交格式 `feat:[国际化二期]xxx`，不带 Claude/Co-Authored-By 署名。
- **不重构周边**：只解决当前问题，顺手清理（删 import、加行内样式）反而留坑。
- **编辑前先 Read**：全角/半角符号、行号变化、并行改动都会导致 old_string 不匹配。

## 🧭 方法论：卡口 vs 覆盖率 vs 工具选型

- **卡口管正确性，覆盖率靠枚举清单**：静态卡口回答"你写的代码对不对"，回答不了"该写的都写了没"。整页未国际化是范围/计划问题，属工作流而非脚本——塞进卡口会靠扫中文猜测、噪声大、越界。三层防御：①开工前按路由子树枚举列 checklist ②独立扫描器 ③覆盖率清单作交付依据。

- **i18n-check 卡口的五类盲区**：能查①路径错位②四语言缺 key；查不出③翻译文案做判断④语言决策错⑤动态拼接 key。后三类需 eslint、专项扫描脚本、人工审阅补位。

- **正则做跨变量数据流追踪本质脆弱**：正则无"作用域"概念，区分不了"这个 item 来自 key 常量"还是"那个 item 来自后端数组"。**命中数剧烈震荡（如 34→14→1→72）本身就是工具选型触及天花板的信号**——应停止调参式迭代，诊断架构缺陷或换判定维度，而非反复调参数。

- **子代理结论需人工复核**：子代理用"应该是/具体值未见"这类猜测语气下的结论不可信；关键断言（如"某常量存的是 key 还是中文"）必须查定义证实/证伪，不能采信推测。

- **AI 批量改造是"被动驱动、按文件流"**：只翻被指到的文件、不会自动补全模块全集。这是遗漏的结构性根因，不是偶发失误。对策是"先列清单再执行"。
