# 看板查看与交互 Claude 对话 — 完整技术调研

> 目标：仅做调研，供权衡选型，不进入实施。
> 调研时间：2026-07-23。关键事实已联网核实（来源见末尾），但开源项目细节动手前需再对照其仓库 README。
> 阅读顺序：先看「现状」→「四方案详评」→「横向对比矩阵」→「风险与决策点」。

---

## 一、本项目现状（探查结论）

| 维度 | 现状 | 是否可复用 |
|---|---|---|
| 派发模型 | MCP 拉模型：backend 不 spawn claude，Claude Code 启动后经 stdio MCP 反查 `get_task`/`record_result` | ✅ 方案 A/B/D 可沿用 |
| 启动 Claude | `TerminalLauncher` 用 `child_process.exec` 跑 `start cmd /k claude`，**fire-and-forget，无法注入消息** | ❌ 交互方案需替换 |
| 会话扫描 | `ClaudeSessionScanner` 已扫描 `~/.claude/projects/**/*.jsonl`，**只取元信息**（title/messageCount/mtime） | ✅ 方案 A 直接扩展 |
| 实时推送 | `InMemoryEventBus` + SSE(`/api/events`)，**只推任务 CRUD** | ✅ 改造后可推对话事件 |
| WorktreeManager | 已实现 create/destroy/getDiff，**未接入派发流程** | ✅ 方案 B/C 可启用 |
| 前端 | `OpenClaudeDialog` 只展示会话列表元信息，无对话正文 | ✅ 加抽屉即可 |

**判断**：基础设施（jsonl 扫描、SSE、worktree）雏形齐备，缺的是「对话正文解析 + 实时回流 + 交互下发」链路。所有方案都能在现有架构上生长，无需推翻重来。

## 二、四个方案详评

### 方案 A：读取 transcript jsonl（只读查看）

**原理**：Claude Code 把每个会话以 JSON Lines 写到磁盘，路径 `~/.claude/projects/<编码后的项目路径>/<session-uuid>.jsonl`，每行一个事件（user/assistant 消息、tool_use、tool_result）。本项目 `pathCodec.ts` 已实现路径编解码。

**工时预估**
- 解析正文 + messages 接口：0.5 天
- 前端对话抽屉渲染：0.5 天
- fs.watch 准实时 + SSE 推送：0.5 天
- 防御式解析 + 单测：0.5 天
- **合计约 2 天**

**能达到的效果**
- 基线：点开任务 → 看到该任务对应 Claude 会话的完整对话历史（用户消息、Claude 回复、调用了哪些工具、工具返回）。
- **最佳效果**：准实时（秒级延迟）跟随 Claude 推进，前端像聊天记录一样滚动渲染；支持折叠 tool_call、代码块高亮、按工具筛选。只读但「看得见全过程」。
- 做不到：发消息给 Claude、干预执行。

**技术栈影响**
- 新增依赖：可选 `chokidar`（跨平台文件监听，比原生 `fs.watch` 稳）。
- 后端：扩展 `ClaudeSessionScanner` + 新增 1 个路由 + 1 个 SSE 通道。
- 前端：新增对话渲染组件（建议直接用现成 markdown + 代码高亮库）。
- **无破坏性改动**，完全增量。

**对项目的长期影响**
- 正面：补齐「可观测性」，看板从「任务管理」升级为「任务 + 执行轨迹」。
- 风险：jsonl schema 是 Claude Code 内部格式，**非官方稳定契约**，升级后可能破坏解析（需防御式解析 + 版本探测）。
- 不引入计费、不改变原生交互体验，**风险最低**。

---

### 方案 B：Claude Agent SDK / `claude -p` headless（可读可交互）

**原理**：派发任务时，backend 用 Agent SDK 的 `query()` 直接驱动一个 headless claude 进程（`claude -p --output-format stream-json` 输出 NDJSON 事件流），把事件转 DomainEvent → SSE → 前端；前端输入框回写 → 注入同一会话（`--resume <sessionId>`）。

**工时预估**
- `ClaudeRunner`（封装 SDK、生命周期、超时、错误恢复）：1.5 天
- stream 事件 → DomainEvent → SSE → 前端渲染：1 天
- 前端交互输入框 + 会话状态机：1 天
- worktree 隔离集成 + 并发控制：1 天
- 鉴权/权限回调（`canUseTool`）、单测、联调：1.5 天
- **合计约 6 天**

**能达到的效果**
- 基线：看板上点「派发」→ backend 自动在 worktree 里跑 claude → 实时看到对话和工具调用 → 可在输入框追加指令、批准/拒绝工具调用、中止任务。
- **最佳效果**：完全网页驱动的多任务并行 AI 编排（接近 Vibe Kanban 形态），每张卡一个隔离 worktree，diff 自动回流，结果自动 record_result，全程不离开浏览器。
- 做不到/代价：见下方计费与体验。

**技术栈影响**
- 新增依赖：`@anthropic-ai/claude-agent-sdk`（或直接 spawn `claude` CLI）、并发队列。
- 后端：新增 `ClaudeRunner` 服务、改造派发流程（从「开终端」变「spawn 进程」）、worktree 接入派发。
- 前端：从「会话列表」升级为「对话 + 输入 + 工具审批」交互面板。
- **破坏性中等**：改变现有 MCP 拉模型的派发语义，需保留两条路径或灰度。

**对项目的长期影响**
- 正面：产品形态跃升到「网页原生编排 Claude」，竞争力最强。
- ⚠️ **计费（2025-06-15 起的政策，重要）**：`claude -p`/Agent SDK **按 API 零售价计费**，但**已并入订阅**——付费 Claude 计划每月含 Agent SDK 额度：Pro $20、Max5x $100、Max20x $200、Team $100/席，按计费周期重置，**超额才需 API key**。即：轻度/中度多任务并行可能**不额外花钱**（吃订阅额度），重度并行才产生增量成本。比早期「完全独立、不共享订阅」的模型友好得多，但额度用尽后按 API 价计费仍可能比交互式订阅贵。
- ⚠️ **放弃原生体验**：本项目 v3 明确「保留原生 Claude Code 体验」，方案 B 用 headless 替代交互式，原生斜杠命令/TUI 体验消失。与既有产品定位冲突，需重新对齐。
- 长期对官方 SDK 强依赖（SDK API 变动需跟进）。

---

### 方案 C：PTY 终端流（保留原生体验）

**原理**：backend 用 node-pty spawn 真正的交互式 `claude`（带 TTY），stdout 经 SSE 推前端，前端用 xterm.js 渲染终端画面，输入回写 stdin。

**工时预估**
- node-pty 集成 + Windows/WSL 编译踩坑：1.5 天
- 多 PTY 生命周期管理 + SSE 流：1.5 天
- 前端 xterm.js 接入 + 输入：1 天
- 跨平台（Win cmd/pwsh/wsl）兼容：1.5 天
- 联调 + 边界（断线重连、resize）：1 天
- **合计约 6.5 天**（Windows 下风险偏高，可能超期）

**能达到的效果**
- 基线：看板里嵌一个真终端，跑的就是原生 Claude Code，所见即所得。
- **最佳效果**：100% 原生 TUI 体验（斜杠命令、vim、进度条、颜色全保留），网页里操作和本地终端无异。
- 做不到：难做结构化高亮（单独定位某次 tool_call、抽取代码 diff 需另外解析转义码）。

**技术栈影响**
- 新增依赖：`node-pty`（**原生模块，需编译**，Windows/WSL 环境是已知坑点）、`xterm.js` + addons。
- 后端：新增 PTY 管理服务、改造 `TerminalLauncher`。
- 前端：嵌入 xterm.js 终端组件。
- **破坏性低-中**：与现有「开终端窗口」并行，可共存。

**对项目的长期影响**
- 正面：原生体验零损失，最贴近本项目「保留原生 Claude Code」定位。
- 风险：node-pty 原生编译增加部署/CI 复杂度；终端转义码解析脆弱；多会话资源占用；Windows 兼容性长期维护成本。
- 计费：仍是交互式订阅额度，**不走 API 计费**（与方案 B 的关键区别）。

---

### 方案 D：Claude Code Hooks 进度时间线（轻量补充）

**原理**：用 Claude Code Hooks（`UserPromptSubmit`、`PreToolUse`/`PostToolUse`、`Stop`、`SubagentStop`）在关键节点写记录到本地或回调 backend，前端展示「任务执行时间线」。

**工时预估**
- Hooks 脚本 + 回调接口：0.5 天
- 前端时间线组件：0.5 天
- **合计约 1 天**

**能达到的效果**
- 基线：看到任务当前在跑什么工具、是否结束、耗时。
- **最佳效果**：结构化进度面板（工具调用链、耗时、失败点），但**不是完整对话正文**。
- 定位：方案 A 的轻量补充，补「实时性」，不替代完整对话视图。

**技术栈影响**
- 几乎无新依赖（Hooks 是 Claude Code 配置项，回调走现有 HTTP）。
- **纯增量，零破坏**。

**对项目的长期影响**
- 正面：低成本补实时可观测性。
- 局限：信息粒度粗，不能替代对话查看。
- 与方案 A 组合性价比高。

## 四、组合可能性

- **A + D**（约 3 天）：低成本组合，先做到「完整只读对话 + 实时进度」，零计费、零原生体验损失。**性价比最高的起步**。
- **A → C**（2 + 6.5 天）：先只读，再升级到网页内嵌原生终端，保留体验、不加计费。
- **A → B**（2 + 6 天）：先只读，再升级到完全网页驱动编排——能力最强，代价是放弃原生体验 + 计费从「交互式订阅」切到「SDK 额度（订阅内含月度额度，按 API 零售价，重度并行可能超额）」。
- **B 单独**：若产品方向就是「网页 AI 编排平台」，直接做 B，跳过 A。

## 五、关键风险与待决策点（需你权衡）

1. **计费模式**：方案 B 走 Agent SDK 额度——订阅内含月度额度（Pro $20 / Max5x $100 / Max20x $200 / Team $100），按 API 零售价计、周期重置，**超额另付 API key**。问题不是「能不能接受 API 计费」，而是「**预期并行任务量是否吃得消订阅额度**」：轻度并行可能零增量成本，重度并行会超额。方案 A/C/D 不涉及此问题（C 仍是交互式订阅额度）。
2. **原生体验底线**：本项目 v3 定位「保留原生 Claude Code 体验」。方案 B（headless）与该定位冲突；方案 C（PTY）符合。交互方案选 B 还是 C，本质是这个产品定位问题。
3. **jsonl schema 稳定性**：方案 A/D 依赖 Claude Code 内部文件格式，非官方契约，升级有破坏风险（可缓解，不可消除）。
4. **Windows + WSL 双环境**：所有方案都要处理 jsonl 路径映射、PTY 在哪侧跑、spawn 跨平台问题。本项目已有 `pathCodec.ts` + `ClaudeSessionScanner` 的 Win/WSL 经验可复用，但 PTY 路线仍需额外验证。
5. **维护成本**：方案 C 的 node-pty 原生模块、方案 B 的 SDK 跟进，都是长期维护负担。
6. **Vibe Kanban 状态**：最对标的开源参考已 sunsetting（转社区维护），可参考源码但不宜直接依赖。
