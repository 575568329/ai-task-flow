# 组件驱动开发 (Component-Driven Development, CDD)

> 更新时间:2026-07-09
> 关联技术:React · shadcn/ui · Radix UI · Tailwind CSS · Storybook
> 适用场景:任何现代组件化前端(React/Vue/Svelte)

---

## 一句话定位

**先造零件,再拼整车。** UI 从「最小可复用组件」开始,自底向上逐层组合成页面。每个组件独立开发、独立预览、独立测试,边界清晰到可以单独丢进组件库。

> 💡 与「页面先行」相反:CDD 不从「画一个完整页面」开始,而是先抽象出按钮、输入框、卡片这些原子,再组装。页面只是组件的「一种排列组合」。

---

## 二、为什么是现代前端的核心密码

### 本质契合

React/Vue/Svelte 的运行时单位就是组件。框架已经替你把「组件化」变成了第一公民,CDD 只是把同样的思维用在**设计和工程**层面。

### 设计系统天然落地

Tailwind + shadcn/ui + Radix 这套组合,把「设计 Token(颜色/间距/圆角)」和「无障碍 + 交互逻辑」分层抽离。CDD 让你顺着这套分层,自然产出一致、可维护的 UI。

### AI 编码助手的最佳拍档

这是 CDD 在 AI 时代的新价值:**组件边界越清晰,AI 越能精准干活**。当你对 Claude Code 说「重构这个 Card 组件的悬停态」,因为组件是独立文件、Props 契约明确,AI 只动这一个文件,不会把整个页面改乱。反之,逻辑全堆在一个巨型页面组件里,AI 很难「局部手术」。

---

## 三、核心思想:Atomic Design

源自 Brad Frost 2013 年提出的原子设计方法论,把 UI 类比化学元素分层:

| 层级 | 含义 | 前端实例 |
|------|------|----------|
| **Atoms 原子** | 最小、不可再分的 UI 元素 | `<Button>` `<Input>` `<Badge>` |
| **Molecules 分子** | 几个原子组合的功能单元 | `<SearchBar>`(Input + Button) |
| **Organisms 有机体** | 分子组合成独立区块 | `<Header>` `<TaskCard>` |
| **Templates 模板** | 页面骨架,放占位组件 | 看板布局(不含真实数据) |
| **Pages 页面** | 模板注入真实数据 | 看板页面 |

> ⚠️ 别教条:**不强制五个层级都建文件夹**。实战中通常 `components/ui/`(原子)+ 业务组件两层就够用,Atomic Design 是「思维粒度」,不是「目录死规定」。

## 四、组件设计三原则

### 1. 单一职责(Single Responsibility)

一个组件只做一件事。`<TaskCard>` 只负责「展示一个任务」,「删除任务」的逻辑不该藏在它里面。

### 2. 单向数据流(One-Way Data Flow)

- 数据**自上而下**通过 Props 流入
- 事件**自下而上**通过回调(`onChange`/`onSubmit`)冒泡
- 子组件不直接改父组件的状态,只「报告」发生了什么

### 3. 高内聚低耦合

- **内聚**:一个组件的样式、逻辑、类型定义放在一起(co-location)
- **耦合**:组件之间只通过「明确的 Props 契约」通信,不互相窥探内部

## 五、现代工具链(2026 主流组合)

| 层 | 角色 | 代表 |
|----|------|------|
| **样式层** | 设计 Token + 原子类 | Tailwind CSS v4 |
| **无头组件(headless)** | 只管交互逻辑和无障碍,不带样式 | Radix UI / Headless UI |
| **复制式组件库** | 给 headless 套上美观样式,代码进你的仓库 | **shadcn/ui** |
| **隔离开发工作台** | 独立预览/测试每个组件的各种状态 | Storybook / Ladle |

### 为什么是 shadcn/ui 而不是 Material UI?

shadcn/ui 是 **copy-paste 而非 npm install**:组件源码直接生成进你的 `components/ui/`,你拥有完全控制权。它基于 Radix(headless,处理焦点/键盘/无障碍)+ Tailwind(样式),既不掉进「覆盖第三方样式地狱」,又不丢掉无障碍。

```tsx
// shadcn/ui 生成的 Button —— 代码就在你仓库里,随便改
import { cva, type VariantProps } from "class-variance-authority";

const buttonVariants = cva("inline-flex items-center justify-center rounded-md", {
  variants: {
    variant: { default: "bg-primary text-primary-foreground", secondary: "bg-secondary" },
    size: { default: "h-10 px-4", sm: "h-9 px-3" },
  },
  defaultVariants: { variant: "default", size: "default" },
});
```

## 六、⚠️ 重要避坑:「容器/展示组件」已被作者本人收回

这是 React 圈流传最广、却也**最容易被照抄成陈旧建议**的模式。

### 这个模式是什么

Dan Abramov 2015 年提出把组件分两类:

| 类型 | 展示组件(Presentational) | 容器组件(Container) |
|------|--------------------------|----------------------|
| 职责 | 渲染 UI | 获取数据、管理状态 |
| 数据来源 | 从 Props 接收 | 自己发请求/连 store |
| 示例 | `<TaskList tasks={...} />` | `<TaskListContainer />`(内部 fetch) |

### 现状:hooks 时代不再必要

**2019 年 Dan Abramov 自己更新了那篇文章**,明确写道:随着 React Hooks 出现,他不再觉得这个区分有用了——逻辑可以用自定义 hook 抽离,UI 组件直接调用 hook 即可,不必硬拆成两类组件。

> ✅ **现代写法**:用 `useXxx` hook 承载逻辑,组件就是组件,不强分容器/展示。

```tsx
// ❌ 老派(2015):硬拆两个组件
function TaskListPresentational({ tasks }) { /* 渲染 */ }
function TaskListContainer() {
  const [tasks, setTasks] = useState([]);
  useEffect(() => fetchTasks().then(setTasks), []);
  return <TaskListPresentational tasks={tasks} />;
}

// ✅ 现代(2019+):hook 抽逻辑,一个组件搞定
function TaskList() {
  const { tasks, loading } = useTasks(); // 逻辑进 hook
  if (loading) return <Spinner />;
  return tasks.map((t) => <TaskCard key={t.id} task={t} />);
}
```

> 🧠 **记结论**:「关注点分离」「易于测试和复用」这两个**目标**永远成立(本项目 CLAUDE.md 也强调「UI 组件与业务逻辑分离」);但「容器 vs 展示」这个**具体手段**已过时,现在用 **自定义 hook** 实现同样的分离。

## 七、最小落地目录结构

```
src/
├── components/
│   ├── ui/            # 原子层(来自 shadcn/ui,纯展示、无业务)
│   │   ├── Button.tsx
│   │   ├── Input.tsx
│   │   └── Dialog.tsx
│   ├── board/         # 业务组件(组合 ui/ 原子,含业务语义)
│   │   ├── TaskCard.tsx
│   │   └── KanbanColumn.tsx
│   └── chat/          # 另一个业务域
│       └── MessageList.tsx
├── hooks/             # 逻辑抽离(useTasks / useChatStream)
│   └── useTasks.ts
├── stores/            # 全局状态(Zustand,按领域拆)
│   └── taskStore.ts
└── views/             # 页面(组件的「一种排列」)
    └── BoardView.tsx
```

> 对照本项目(ai-task-flow):`frontend/src/components/ui/`(shadcn 原子层)+ `components/board|chat/`(业务组件)+ `views/`(页面)+ `stores/`(Zustand)——正是 CDD + 现代 hook 分离的标准形态。CLAUDE.md 要求「能封装的都封装,最好都能抽进组件库」,这就是 CDD 的落地体现。

## 八、Storybook:组件的隔离工作台

Storybook 让你在**脱离业务路由**的环境里,单独预览一个组件的各种状态(default / loading / error / empty / disabled),还能做视觉回归和可访问性测试。

```tsx
// TaskCard.stories.tsx —— 一个状态一个 story
import type { Meta, StoryObj } from "@storybook/react";
import { TaskCard } from "./TaskCard";

const meta: Meta<typeof TaskCard> = {
  component: TaskCard,
  tags: ["autodocs"],
};
export default meta;

export const Default: StoryObj = {
  args: { task: { id: "WS-001", title: "示例", status: "todo" } },
};
export const Loading: StoryObj = { args: { task: null, loading: true } };
export const Empty: StoryObj = { args: { task: null } };
```

> YAGNI 提醒:小项目不必上 Storybook,维护一套清晰的 `components/ui/` 目录 + 良好的 Props 类型已经能拿到 CDD 80% 的收益。Storybook 是团队/设计系统级别的投入。

## 九、常见误区

| ❌ 反例 | ✅ 正解 |
|--------|--------|
| 一个页面组件 800 行,状态/请求/UI 全塞里面 | 拆:UI 进组件、逻辑进 `useXxx` hook、状态进 store |
| Props 传了 8 层(prop drilling) | 该用 Context 或状态库就别硬传 |
| 原子组件里写死业务逻辑(`<Button onClick={deleteTask}>`) | 原子只管 UI,业务回调用 `onClick` Props 上抛 |
| 还在硬拆容器/展示组件 | 改用自定义 hook 抽逻辑 |
| 组件内部直接发请求 | 数据获取进 hook,组件保持纯展示,易于测试和复用 |
| 列表用数组 index 当 `key` | 用业务唯一 id(本项目 CLAUDE.md 明确禁止 index 当 key) |

## 十、什么时候别过度 CDD

- **一次性落地页/营销页**:组件不复用,过度抽象反而增加心智负担
- **极简管理后台**:三五个表单,直接堆即可
- **原型阶段**:先用「能跑」验证想法,稳定后再抽象组件(但别拖太久)

> 判断标准:**同一个 UI 片段出现第二次时**,才把它抽成组件(YAGNI)。第一次出现就抽象,是过早设计。

## 十一、速查清单

- [ ] UI 从最小原子开始,自底向上组合
- [ ] `components/ui/`(原子,无业务)+ 业务组件 两层起步
- [ ] 逻辑用 `useXxx` hook 抽离,不强分容器/展示组件
- [ ] 单一职责 + 单向数据流 + 明确 Props 契约
- [ ] 列表用业务 id 当 key,不用数组 index
- [ ] 团队级投入再上 Storybook,个人项目保持目录清晰即可
- [ ] 组件边界清晰,让 AI 能「局部手术」式重构
