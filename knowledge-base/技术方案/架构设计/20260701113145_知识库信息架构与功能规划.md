# 知识库信息架构与功能规划

> 规划日期:2026-07-01
> 讨论问题:
> 1. 文档该"每个讨论单独文件夹"还是"按类型分区"?
> 2. 文档之间能否关联?关联点击后能预览、能回退、回退后保留原浏览位置吗?
> 3. 整体判断项目要用哪些功能,业界有无成熟项目可参考改进?

---

## 一、核心结论

**业界已有直接对标项目:Quartz**(开源、静态、TypeScript)。它就是"Obsidian Publish 的免费开源版",原生支持:双向链接、反向链接(backlinks)、关系图谱(graph view)、全文搜索、Obsidian 语法兼容。多个 2026 年评测都把它列为"个人知识库 / 数字花园"的首选静态方案。

**但是否要迁移到 Quartz 需要权衡**(详见第五节):Quartz 需要 Node + 构建流程,而本项目当前的核心优势是"零构建、双击即用 + 本地增删"。**更务实的路径是:借鉴 Quartz 的功能设计,在现有架构上增量实现,而非推倒重来。**

---

## 二、问题 1:文件夹组织 —— 按类型分区 vs 每讨论单独文件夹

### 业界共识:文件夹、标签、链接是三个正交维度

来自 Obsidian/Logseq 社区的成熟结论(Jorge Arango、Steph Ango 等 PKM 专家反复印证):

| 维度 | 表达什么 | 关系 | 特点 |
|-----|---------|------|------|
| **文件夹 (Folder)** | "存在哪" | 一对一(一个文件只能在一个文件夹) | 物理存储,最弱的语义关系 |
| **标签 (Tag)** | "有什么属性" | 多对多 | 扁平、灵活、横向关联 |
| **链接 (Link)** | "指向谁" | 多对多 | 语义关联,知识网络的骨架 |

**关键洞察**:不要用文件夹承担过多语义。文件夹只管"物理归档",真正的知识关联靠**链接**和**标签**。

### 对本项目的判断

当前你已经是"**按类型分区**"(技术方案/物料测试/项目复盘/示例方案),这个方向**是对的**,建议保持。理由:

- **"每讨论单独文件夹"的问题**:讨论会越来越多,每个建文件夹会导致目录树极度膨胀,且很多讨论只有 1 个文件,文件夹变成累赘
- **"按类型分区"的优势**:类型数量有限(技术方案、复盘、测试...),目录树稳定;同类内容集中,好找
- **补充建议**:
  - 文件夹粗分类(按"类型/领域")即可,不要分太细
  - 细粒度的关联交给**标签**(md frontmatter 里写 `tags: [MCP, 图片生成]`)和**双向链接**
  - 命名继续用 `YYYYMMDDHHMMSS_标题`,天然按时间排序

### 推荐结构

```
docs/
├── 技术方案/          ← 按类型(现状,保持)
├── 项目复盘/
├── 物料测试/
├── 示例方案/
└── (未来可加) 索引/    ← 放 MOC(Map of Content)聚合页
```

> **MOC(内容地图)**:一种业界实践,用一篇"目录页"文档,手动链接聚合某主题下的所有相关文档。比文件夹更灵活,是"人工策展的入口"。

## 三、问题 2:文档关联 + 预览 + 回退 + 保留位置

拆成 4 个子能力,逐一判断可行性:

### 2.1 文档之间能否关联? ✅ 完全可做

**方案**:支持 Markdown 的 wiki 链接语法或标准链接,渲染时转成可点击的站内跳转。

- **标准 md 链接**:`[方案对比](../技术方案/xxx.md)` — marked.js 已支持,只需拦截点击转为 hash 路由
- **wiki 链接**(Obsidian 风格):`[[20260630_AI原型方案]]` — 需在渲染前用正则替换成 `<a>`,匹配 manifest 里的文件
- **实现难度**:低。在 `viewers.js` 的 renderMarkdown 里加链接处理即可

### 2.2 关联点击后能预览? ✅ 已经能

现在点目录树就能预览,关联链接本质是"程序化触发同样的路由"。改 `location.hash` 即可复用现有预览逻辑。

### 2.3 能回退吗? ✅ 已经能(浏览器原生)

当前用 `location.hash` 做路由,**浏览器的前进/后退按钮天然可用**。点链接跳转会进历史栈,点后退就回到上一篇。可再加个页内"← 返回"按钮调 `history.back()`。

### 2.4 回退后保留原浏览位置? ⚠️ 需要额外实现

**这是唯一需要专门做的**。现状:回退后文档从顶部重新渲染,滚动位置丢失。

**解决方案**:
- 监听内容区滚动,把每篇文档的 `scrollTop` 存进一个 Map(key = 文档路径)
- 路由切换/回退时,渲染完成后恢复该文档记录的滚动位置
- 配合 `history.state` 或 sessionStorage 持久化
- **实现难度**:中等,约 30~50 行代码

**小结**:四个能力里三个几乎白送,只有"保留滚动位置"要专门写,但不难。

## 四、问题 3:整体功能判断 —— 该有哪些,业界怎么做

对照 Quartz / Obsidian Publish / MkDocs 这些成熟项目,列出功能清单,标注本项目现状:

### 已有 ✅

| 功能 | 现状 |
|-----|------|
| 目录树导航 | ✅ 自动扫描 docs/ 生成 |
| 多格式预览 | ✅ md/pdf/docx/html/img |
| 全文件名搜索 | ✅ 侧边栏搜索框 |
| 亮/暗主题 | ✅ |
| 本地增删文件 | ✅ 上传/删除 + SSE 实时刷新 |
| 双模式(本地全功能/线上只读) | ✅ |
| 响应式(移动端抽屉) | ✅ |

### 建议新增(按优先级)

| 优先级 | 功能 | 业界参考 | 实现难度 | 说明 |
|:---:|-----|---------|:---:|------|
| **P0** | **文档双向链接** | Quartz/Obsidian | 低 | wiki 链接 `[[xxx]]` + 标准链接站内跳转 |
| **P0** | **回退保留滚动位置** | 通用 Web | 中 | 存 scrollTop,渲染后恢复 |
| **P1** | **反向链接 (Backlinks)** | Quartz/Obsidian | 中 | 文档底部显示"哪些文档链接到我" |
| **P1** | **正文全文搜索** | Quartz(FlexSearch) | 中 | 现在只搜文件名,升级为搜正文内容 |
| **P1** | **标签系统** | Obsidian | 中 | md frontmatter 写 tags,侧边栏按标签筛选 |
| **P2** | **关系图谱 (Graph View)** | Quartz | 高 | 可视化文档间链接网络,炫但非刚需 |
| **P2** | **文档大纲 (TOC)** | 多数文档站 | 低 | 右侧显示当前文档的 H1-H3 目录锚点 |
| **P2** | **面包屑增强** | 通用 | 低 | 当前已有基础面包屑,可加可点击跳转 |
| **P3** | **MOC 索引页** | PKM 社区 | 低(内容) | 手写聚合页,不需代码,建个 docs/索引/ |

### 明确不建议做

- **在线编辑器**:VS Code / Typora 更强,知识库专注"看"
- **多用户/权限**:个人知识库不需要,会引入后端复杂度
- **数据库**:纯文件 + manifest 已足够,上数据库违背"clone 即用"

## 五、要不要迁移到 Quartz?

这是最关键的战略决策,做个诚实对比:

| 维度 | 现有自研项目 | 迁移到 Quartz |
|-----|------------|--------------|
| **构建** | 零构建,clone 即用 | 需 Node + `npx quartz build` |
| **本地增删文件** | ✅ 有(SSE 实时) | ❌ 无(纯发布工具) |
| **双向链接/反向链接** | ❌ 需自己加 | ✅ 原生 |
| **关系图谱** | ❌ 需自己加(难) | ✅ 原生 |
| **全文搜索** | ❌ 只有文件名 | ✅ 原生 FlexSearch |
| **多格式预览(pdf/docx)** | ✅ 有 | ⚠️ 主要面向 md,pdf/docx 需自己配 |
| **定制自由度** | ✅ 完全可控 | ⚠️ 受框架约束 |
| **维护成本** | 自己维护全部代码 | 跟随上游更新,但要学它的体系 |

### 判断

**不建议整体迁移,建议"借鉴 + 增量自研"**,理由:

1. **本项目有 Quartz 没有的核心能力**:本地增删文件 + SSE 实时刷新 + pdf/docx/html 多格式预览。迁到 Quartz 反而丢功能。
2. **Quartz 的优势(链接/反链/搜索/图谱)都可以增量实现**,前三个难度都不高。
3. **迁移成本高**:要重学 Quartz 体系、放弃已有的双模式架构和多格式预览。
4. **唯一值得纠结的是"关系图谱"**:自己实现较难。但它是"锦上添花"非刚需,可放到最后,或需要时单独接个 d3.js 力导向图。

**结论**:把 Quartz 当"功能参照物",逐个把它的好功能借鉴过来,保留本项目"零构建 + 本地全功能 + 多格式"的独特优势。

## 六、建议的实施路线图

### 第一阶段:关联能力(P0,最高价值)

1. Markdown 渲染时支持 `[[wiki链接]]` 和站内相对链接,点击走 hash 路由
2. 回退保留滚动位置(存 scrollTop Map,渲染后恢复)

### 第二阶段:发现能力(P1)

3. 反向链接:构建"谁链接了谁"的索引(生成 manifest 时顺带扫描),文档底部显示
4. 正文全文搜索:引入轻量前端搜索库(如 FlexSearch/MiniSearch),索引 md 正文
5. 标签系统:解析 md frontmatter 的 tags,侧边栏加标签筛选

### 第三阶段:增强体验(P2/P3)

6. 文档右侧 TOC 大纲(锚点跳转)
7. MOC 索引页(纯内容,建 docs/索引/)
8. (可选)关系图谱 —— 需要时再做

## 八、一句话总结

- **文件夹**:保持"按类型分区",别搞"每讨论一文件夹"(会膨胀)。细关联交给链接和标签。
- **关联/预览/回退**:三个白送(hash 路由 + 浏览器历史),只有"保留滚动位置"要专门写,不难。
- **整体功能**:业界对标是 **Quartz**,但**不迁移**,而是借鉴它的双向链接、反向链接、全文搜索,增量加到现有项目上,保住"零构建 + 本地全功能 + 多格式预览"的独特优势。
