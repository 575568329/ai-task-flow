# 规格驱动设计 (Spec-Driven Design)

> 更新时间:2026-07-09
> 规范版本:OpenAPI 3.1(对齐 JSON Schema 2020-12)
> 关联技术:OpenAPI · openapi-typescript · Prism/MSW · LLM Function Calling
> 适用场景:前后端协作、Mock 并行联调、AI Agent 自动调用 API

---

## 一句话定位

**先写接口规格(契约),再写前后端代码。** 规格是唯一的真相源(single source of truth),前端的 TS 类型、Mock 数据、接口文档、甚至 AI Agent 的工具描述,全部从它自动生成——人只维护一份 spec,机器推导一切。

> 💡 区别于「代码先行」:传统流程是后端先写 Controller/路由,前端等接口出来再对;Spec-Driven 是两边一起盯着同一份 `openapi.yaml`,各自往下推。

---

## 二、为什么是 AI 时代的绝对核心

### 前端视角:消灭字段拼错

用 `openapi-typescript` 把 `openapi.yaml` 一键编译成 TS 类型,前端调用接口时全程强类型提示。字段名拼错、可空字段漏判、枚举值写飞——这些低级错误在编译期就被拦下。

### 后端视角:契约即护栏

后端实现偏离 spec 时(比如偷偷多返回/少返回字段),契约测试(contract test)立刻报红。spec 成为前后端的**强制共识**,而不是 wiki 上一份没人看的过期文档。

### AI / Agent 视角(本篇重点)

大模型怎么知道该调哪个外部接口、传什么参数?**答案就是把 OpenAPI 规格喂给它。** 2026 年的成熟做法是:OpenAPI → 自动转成 function/tool schema → 注册进 LLM 的 function calling(也叫 tool use)。模型读懂每个接口的参数和语义后,自主决定调用并生成合法请求体。

> 🌐 趋势:业界正走向 **OpenAPI-to-tool-schema 管道**——你维护一份 spec,既能给前端出类型,又能给 AI 出工具,还能跑 Mock 和契约测试。一份投入,四处产出。

## 三、OpenAPI 规范速览

### 历史脉络(别把 Swagger 和 OpenAPI 搞混)

| 时间 | 事件 |
|------|------|
| 2011 | Wordnik 的 Tony Tam 创建 **Swagger**,作为 API 文档+调试工具 |
| 2015 | Swagger 规范捐给 **Linux Foundation**,更名 **OpenAPI Initiative** |
| 2017 | OpenAPI 3.0 发布(从 Swagger 2.0 演进) |
| 2021 | **OpenAPI 3.1** 发布,与 JSON Schema 全面对齐 |

> 一句话:**Swagger 是工具品牌**(Swagger Editor/UI),**OpenAPI 是规范本身**。日常说「写 Swagger 文件」其实指「写 OpenAPI 规格文件」。

### OpenAPI 3.0 vs 3.1 关键差异(高频踩坑)

新项目**直接上 3.1**,并记住这几条 breaking change:

| 特性 | OpenAPI 3.0 | OpenAPI 3.1 |
|------|-------------|-------------|
| **JSON Schema 对齐** | 「扩展子集」(自定义变体) | **完整 JSON Schema 2020-12 方言** |
| **表示可空** | `nullable: true` | ❌ 已移除 → 用 type 数组 `["string", "null"]` |
| **Webhooks** | ❌ 不支持 | ✅ 一等公民 `webhooks` 字段 |
| **`exclusiveMinimum/Maximum`** | 布尔值 | 数值(遵循 JSON Schema 2020-12) |
| **`$ref` 同级字段** | 不允许与其他 key 共存 | ✅ 允许 sibling keys 共存 |
| **示例字段** | `example`(单数) | `examples`(复数) |
| **版本号策略** | 语义化版本(SemVer) | 放弃 SemVer,只为追赶 JSON Schema |

> ⚠️ 最容易写错的就是「可空」。3.1 里**别再写 `nullable: true`**,要么 `"type": ["string", "null"]`,要么用 `oneOf`。

### 一份最小可用 spec(3.1)

```yaml
# openapi.yaml
openapi: 3.1.0
info:
  title: Task Flow API
  version: "1.0.0"
paths:
  /api/tasks:
    get:
      summary: 获取所有任务
      operationId: listTasks
      parameters:
        - name: status
          in: query
          schema:
            $ref: "#/components/schemas/TaskStatus"
      responses:
        "200":
          description: 任务列表
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: "#/components/schemas/Task"
components:
  schemas:
    TaskStatus:
      type: string
      enum: [planning, todo, dispatched, review, done]
    Task:
      type: object
      required: [id, title, status]
      properties:
        id:
          type: string
        title:
          type: string
        status:
          $ref: "#/components/schemas/TaskStatus"
        assignee:
          # 3.1 写法:可空用 type 数组,不是 nullable
          type: ["string", "null"]
          description: 负责人,可能为空
```

## 四、工具链(2026 现状)

| 环节 | 推荐工具 | 产出 |
|------|----------|------|
| **类型生成**(前端) | `openapi-typescript` | 从 spec → `types/api.d.ts`,零运行时 |
| **Client 生成** | `openapi-fetch` / `orval` / `hey-api` | 强类型请求函数,带自动重试/拦截器 |
| **全量 Client** | `openapi-generator` | 多语言 SDK(Python/Go/Java...) |
| **Mock 服务** | `Prism`(Stoplight)/ `MSW` | 前端不等后端,直接用 spec 跑假数据 |
| **契约测试** | `Prism` / `Dredd` | 后端实现 vs spec,偏离即报红 |
| **可视化文档** | `Swagger UI` / `Redoc` / `Scalar` | 自动生成可调试的 API 文档站 |

> 本项目(ai-task-flow)目前的接口是后端 Fastify 路由直接定义、前端手写 `api/*.ts` 封装,**没有走 spec 生成**。若要升级为 Spec-Driven,第一步就是用 `@fastify/swagger` 自动生成 openapi.json,再让前端 `openapi-typescript` 消费。

## 五、最小落地流(Next.js + Node.js)

### Step 1:维护一份 `openapi.yaml`(或用代码注解自动生成)

```bash
npm i -D openapi-typescript openapi-fetch
```

### Step 2:编译出类型

```bash
# 把 spec 编译成 TS 类型,提交进仓库或 CI 生成
npx openapi-typescript ./openapi.yaml -o ./src/types/api.d.ts
```

### Step 3:前端用强类型 client 调接口

```ts
// src/api/client.ts
import createClient from "openapi-fetch";
import type { paths } from "@/types/api"; // 自动生成的类型

const client = createClient<paths>({ baseUrl: "/api" });

// 调用处:method/path/params 全部有类型提示,response.data 推断为 Task[]
const { data, error } = await client.GET("/tasks", {
  params: { query: { status: "review" } }, // status 枚举自动校验
});
```

### Step 4:Mock 先行,前后端并行

```ts
// 用 MSW(Mock Service Worker)拦截请求,按 spec 返回假数据
import { http, HttpResponse } from "msw";

export const handlers = [
  http.get("/api/tasks", () =>
    HttpResponse.json([{ id: "WS-001", title: "示例任务", status: "todo" }]),
  ),
];
```

后端没写完?没关系——前端对着 spec + Mock 就能把页面做完,接口出来后把 Mock 关掉即可。

## 六、AI Agent 视角:Spec 即工具(本篇灵魂)

### Function Calling / Tool Use 的本质

主流大模型(Claude、GPT、GLM 等)都支持 **function calling**(也叫 tool use):你给模型一组「工具描述」(工具名 + 参数 schema),模型在对话中自主判断要不要调、调哪个、传什么参数,然后**你的代码**真正执行并把结果回灌给模型。

闭环是:`模型建议调用 → 你的环境执行 → 结果回灌 → 模型继续推理`。

### OpenAPI → Tool 的转换

OpenAPI 的 `operationId` + `parameters` + `requestBody` 几乎就是 function calling 所需的 `{ name, description, input_schema }`。所以**一份 spec 可以机械地转成一组 tool 定义**:

```ts
// 伪代码:把 OpenAPI 的一个 operation 翻译成 LLM tool 定义
function operationToTool(op: OpenApiOperation) {
  return {
    name: op.operationId,            // listTasks
    description: op.summary,          // 获取所有任务
    input_schema: op.parameters?.[0]?.schema, // 直接复用 JSON Schema
  };
}

// 注册给 Claude / GPT / GLM
const tools = spec.paths
  .flatMap((path, method) => operationToTool(path[method]));
```

### 给大模型读 Spec 的两种姿势

| 姿势 | 做法 | 适用 |
|------|------|------|
| **转换注册** | 把 spec 转成 tool schema,经 function calling 调用 | 接口固定、要稳定调用 |
| **直接喂文本** | 把整份 openapi.yaml 塞进 system prompt,让模型自行解析 | 接口很多、按需探索(如 ChatGPT GPTs 的 Actions) |

> 💡 现实例子:OpenAI 的 **GPTs Actions**、Anthropic 的 **MCP(Model Context Protocol)**、各种 Agent 框架(LangChain 的 `OpenAPIToolkit`)背后都是这套——**spec 就是 AI 与外部世界的接线图**。本项目自己的 MCP Server(`list_pending_tasks` 等工具)本质上也是「给 Claude Code 一组带 schema 的工具」,只是手写的,而非从 OpenAPI 自动生成。

## 七、常见误区(反例)

| ❌ 反例 | ✅ 正解 |
|--------|--------|
| spec 写完就扔,实现一变 spec 不更新 | spec 进 git,改动走 PR,CI 跑契约测试守护栏 |
| 前端手抄接口字段类型 | 用 `openapi-typescript` 生成,杜绝手抄漂移 |
| 把业务校验逻辑塞进 spec 的 `description` 里 | `description` 只描述语义,校验用 `pattern`/`minLength` 等 schema 约束 |
| 一个超大 `openapi.yaml` 几千行 | 用 `$ref` 拆分到 `components/`,或用代码注解(`@fastify/swagger`/NestJS `@ApiProperty`)反向生成 |
| 3.1 还在用 `nullable: true` | 改用 type 数组 `["string", "null"]` |
| spec 和真实响应字段对不上还不报警 | 上 Prism/Dredd 契约测试,偏离即红 |

## 八、什么时候**别**用 Spec-Driven

KISS & YAGNI 提醒——不是所有项目都值得:

- **一次性脚本 / 内部小工具**:接口就两三个,手写类型更快
- **纯前端展示页**:没有自己的后端 API
- **接口剧烈变动期**:spec 改得比代码还勤,维护成本 > 收益,等稳定再补
- **单人全栈小项目**:你自己就是契约,spec 的「沟通」价值归零(但「生成类型 + AI 工具」价值仍在)

> 判断标准:**只要出现「前端等后端接口」「字段又对不上了」「想让 AI 调我的 API」这三种情况之一**,Spec-Driven 的收益就立刻为正。
