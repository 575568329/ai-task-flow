## 一、输入与发送

### 1.1 发送快捷键 = Enter(行业默认)
- **multica**:`packages/core/shortcuts/definitions.ts` → `{ id: "send", defaultShortcut: primary("Enter") }`。注意 `primary("Enter")` = **Ctrl/Cmd+Enter**(multica 的主快捷键),但 `isShortcutAllowedForAction` 显式把**裸 Enter 也列为合法例外**——所以 multica 实际两种都能发。
- **Cursor / Cline / ChatGPT / Claude.ai**:默认就是**裸 Enter 发送、Shift+Enter 换行**(Cline Issue #10342 有用户抱怨,但行业默认如此)。
- **结论(我们)**:采用裸 Enter 发送 + Shift+Enter 换行——对齐 Cursor/Cline/ChatGPT/Claude.ai 的主流约定(而非 multica 的 Ctrl/Cmd+Enter),最符合用户直觉。

### 1.2 发送键运行时变「停止」(SubmitButton 变形)
multica `packages/ui/.../submit-button.tsx`:
```tsx
if (running) return <Button onClick={onStop}><Square className="fill-current" /></Button>;  // 停止键
return <Button onClick={onClick} disabled={...}>
  {loading ? <Loader2 className="animate-spin" /> : <ArrowUp />}  // 发送键(箭头,非纸飞机)
</Button>;
```
- 图标用 **ArrowUp**(ChatGPT/Claude.ai 同款),不是纸飞机。
- `running`(任务在跑)优先级最高 → 渲染停止键(Square 实心方块)。
- `loading`(提交中)/`busy`(上传中)→ spinner + disabled。
- **结论**:流式时发送键变停止键,用户可随时中断。

### 1.3 发送后 blur 编辑器
multica `chat-input.tsx` handleSend → `commitInput`:发送后 `editorRef.current?.blur()` —— 不让光标停在即将出现的流式回复下方;完成后**不自动 refocus**(会打断用户选中回复文本)。

## 二、自动滚动(只在靠近底部时跟随)

multica `chat-message-list.tsx`(用 react-virtuoso):
```tsx
<Virtuoso
  initialTopMostItemIndex={{ index: "LAST", align: "end" }}  // 打开定位到最新
  atBottomThreshold={120}
  atBottomStateChange={setIsNearBottom}                        // 跟踪是否在底部
  followOutput={() => (!isFetchingOlder && isNearBottom ? "smooth" : false)}
/>
```
- **核心**:`followOutput` 只在 `isNearBottom` 时返回 "smooth",否则 `false` —— **用户上翻看历史时,新内容不会把视图拉回底部**。
- `atBottomThreshold={120}`:120px 内算「在底部」。
- 无 virtuoso 的实现:监听 scroll,`scrollHeight - scrollTop - clientHeight < threshold` 判断 nearBottom,只在 nearBottom 时 `scrollTop = scrollHeight`。

## 三、工具/思考展示 —— Conductor 三段切分 + 外层 process fold(最值得借鉴)

multica `chat-message-list.tsx` 的 `TimelineView` + `splitTimeline`(`lib/copy-text.ts`):

把一轮 assistant 的 timeline 按「非文本项(thinking/tool_use/tool_result)」切成三段:
- **preface**:第一个非文本项之前的文字(开场白)
- **middle**:第一个 → 最后一个非文本项(含中间夹的文字),即「过程」
- **final**:最后一个非文本项之后的文字(最终答案)

渲染:
```tsx
{preface && <RichContent ... />}                          // 开场白:正常 prose
{middle && <OuterProcessFold items={middle} />}           // 过程:折叠成「N steps」
{final && <RichContent ... />}                            // 答案:正常 prose(突出)
```

`OuterProcessFold`(外层折叠):
```tsx
const [open, setOpen] = useState(!!isStreaming);  // 流式时展开,完成后自动收起
useEffect(() => { if (wasStreaming && !isStreaming) setOpen(false); }, [isStreaming]);
// 触发器:"▶ N steps"
```
- **流式中展开**(看进度)→ **完成后自动折叠**成「N steps」,把最终答案顶到显眼位置。
- 内层每个 tool_use/thinking/tool_result 各自独立折叠,点开看详情。

**对比我的实现**:我把 tool_use 全部平铺、思考单独卡。multica 的「过程整体折叠」更克制、答案更突出(Cursor/Claude.ai 同款)。

### 3.1 工具摘要(getToolSummary)
按优先级取字段:`query > file_path > path > pattern > description > command > prompt > skill`,过长截断(command/prompt 100 字符,路径取后两段 `.../a/b`)。我的 ToolUseCard.summarize 思路一致 ✓。

### 3.2 路径缩短
`shortenPath`:>3 段取后两段 `.../dir/file`。

### 3.3 tool_result 单独成行(不嵌进 tool_use)
multica 把 tool_use 和 tool_result 作为**时间线相邻行**各自折叠,而非把 result 嵌进 use 卡片。线性更易读。结果预览 120 字符,完整最多 4000 字符截断。

### 3.4 thinking 行
`ThinkingRow`:Brain 图标 + 斜体 muted 预览(150 字符),点开全文。流式中也是预览态。

## 四、中断/停止机制

- **multica**:daemon 架构,stop → `api.cancelTaskById(taskId)`(server 持有 agent 进程)。还有「延迟取消」——任务 id 还没从 server 拿到时,先记 `stopRequestedBeforeTaskRef`,id 一到就取消。
- **ai-task-flow(我们)**:无 daemon,后端直接 spawn claude。停止 = 前端 `AbortController.abort()` → fetch 断开 → 后端 `request.raw.on('close')` 触发 `child.kill()`。比 multica 简单,UX 一致。
- **关键**:必须 kill 子进程,否则 claude 跑到 max-turns/timeout(默认 50 轮 / 10 分钟)才停 —— 资源浪费 + 用户以为已停其实没停。

## 五、历史记录架构(对话持久化)

### 5.1 数据源选择
Claude CLI 把会话转录写在 `~/.claude/projects/<编码后的cwd>/<sessionId>.jsonl`,按仓库分目录 —— 天然对应「当前仓库下的对话历史」。后端已有 `ClaudeSessionScanner` 读这些文件。

**方案 A(推荐)**:后端读 claude jsonl → 解析成 turns 回放。单一数据源,能看到本仓库所有 claude 会话(不止本工具起的),续接复用 `--resume <sessionId>`。
**方案 B**:后端自存 `<dataDir>/task-chats/<taskId>.json`。简单但与 jsonl 双写易不一致。

### 5.2 multica 的会话列表(chat-thread-list)
- 左侧/侧栏列出 chat_session,每条带未读数、完成态(Check)、运行态(Loader2 spinner)。
- `key={activeSessionId}` 重挂载消息列表(切会话重置滚动)。
- 「新建会话」按钮 + 焦点请求(focusRequest nonce bump 让输入框立即聚焦)。

### 5.3 草稿持久化
multica 按 session 存草稿(`inputDrafts[draftKey]`),切会话不丢未发送文字。我们任务对话量小,可暂不做。

## 六、其他细节

- **MessageFooter**「Replied in 38s · [Copy]」:耗时 + 复制按钮在回复下方,流式中禁用复制。
- **FailureBubble**:失败不用 destructive alert,用「图标 + muted 文案 + 可折叠原始错误」,信息性而非报错性。
- **memo 消息行**:流式每条事件重渲整列表,`memo` 跳过未变行。
- **live→persisted 同 key 切换**:流式行和持久化行用同 key(`task:<id>`),Mermaid/iframe 不重挂载、保留 pan-zoom。
- **空回复处理**:agent 跑完无文字 → 显示「无文字回复」提示,不渲染空 markdown 块。
- **大消息体**:Virtuoso 虚拟滚动;`increaseViewportBy={{top:400,bottom:600}}` 预渲染。
