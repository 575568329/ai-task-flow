## 一句话总结

要从 Claude Code 的开发对话中提炼项目经验,**最优解不是"用工具把对话导出来读",而是让 Claude 直接读取清洗后的对话做智能分析** —— 脚本负责机械去噪,Claude 负责理解归纳,并用"归纳+审阅"分角色制衡保证质量。最终沉淀为 `project-retrospective` skill。

## 探索历程(为什么是这个方案)

### 阶段 1:验证可行性
Claude 能读取自己的对话历史。存储在 `C:\Users\<用户>\.claude\projects\<项目编码>\<会话ID>.jsonl`。
→ 详见 [Claude对话历史分析方案](20260701100000_Claude对话历史分析方案.md)

### 阶段 2:调研现有工具
对比了三个方案,结论是**它们都只做"读取+格式转换",不做智能分析**:

| 工具 | 本质 | 局限 |
|------|------|------|
| Simon Willison / claude-code-transcripts | JSONL → 精美 HTML | 只美化展示,无分析 |
| 我们的 PowerShell 脚本(早期) | JSONL → Markdown+JSON | 只统计,无理解 |
| ChatLab | 通用聊天分析平台 | 重量级、非专用 |

→ 详见 [SimonWillison方案详解](20260701_SimonWillison方案详解.md)、[ChatLab方案对比](20260701_ChatLab方案对比.md)

### 阶段 3:关键洞察
用户提出:"这些工具只是把聊天读出来,不如直接让 Claude 自己读并分析。"
业界证据支持这个直觉:
- Cursor/Claude Code 都用 grep+直接读取,而非 RAG 向量搜索
- LLM 长文本总结最佳实践:Map-Reduce 分块、混合式提取
→ 详见 [对话分析业界方案研究](20260701_对话分析业界方案研究.md)

### 阶段 4:落地为 Skill
把方案固化成可复用的 `project-retrospective` skill。

## 最终方案架构

```
用户触发 skill(提供:项目路径 / 日期范围 / 补充要求)
   │
   ▼
[脚本] discover_sessions.ps1 — 按消息级时间戳发现会话
   │
   ▼
[脚本] extract_clean.ps1 — 剔除噪音,提取干净对话
   │  (只留 typed 用户输入 + assistant 文本,剥离 think 块)
   ▼
[Claude 分角色分析]
   ├─ 归纳者:Map-Reduce 总结,每条标来源
   └─ 审阅者:独立子代理挑错(查遗漏/偏差/臆测)
   │  ↑↓ 最多三轮 → 交用户审批
   ▼
[三路产出]
   ├─ 复盘文档 → docs/项目复盘/
   ├─ 记忆沉淀 → memory/ + MEMORY.md
   └─ 开发经验 → 独立文件(反哺后续开发)
```

## 核心设计决策(踩坑换来的)

| 决策 | 为什么 |
|------|--------|
| **脚本去噪 + Claude 分析** 分工 | 原始 JSONL 90% 是噪音,单文件达 18MB,不能直接 Read |
| **按消息级 timestamp 筛选** | 单会话可能跨多天(实测 6/23→6/26),文件时间会误判 |
| **用 `promptSource:"typed"` 过滤** | 精确标记用户真实输入,是去噪的关键锚点 |
| **剥离 `<think>` 标签** | 部分模型把思考塞进 text 块,混入会干扰归纳 |
| **归纳+审阅分角色** | 单一视角会"自我确认",独立审阅能发现盲区 |
| **最多三轮交人审批** | 防止 AI 反复纠结"再优化一点"陷入循环 |

---

## 验证结果

用真实项目 `rpp-web` 验证:
- **会话发现**:6/23-6/26 范围命中 10 个会话,正确识别跨天会话
- **去噪效果**:12MB 大会话 → 452KB(3.6%);小会话 28KB 无压力通读
- **分角色实测**:审阅者成功抓到归纳者的一处臆测(把 login/get 误定性为"贴错")+ 两处遗漏 → 证明制衡机制有效
