# 后端 spawn Claude CLI 通用方案(Windows + WSL)

> 任何「后端进程主动拉起 Claude Code CLI(headless)、双向通信、流式回传」的应用通用。蒸馏自 ai-task-flow 落地经验 + multica(`D:\study\multica`)源码。本项目具体实现见 `docs/` 实现报告。

## 一、核心结论

```
claude -p --output-format stream-json --input-format stream-json --verbose \
       --permission-mode bypassPermissions --max-turns N --settings <clean.json>
       [--resume <sessionId>]
```

- **prompt 经 stdin pipe 写入**(stream-json `user` 帧),绝不用命令行、绝不用 `< file`
- stdin 写完首帧**保持打开**,收到 `result`(或 abort)才 `stdin.end()`
- stdout 按行读 stream-json,逐事件处理

违反任一条,最典型故障是 **claude 静默 exit 0 无输出**,极难排查。

## 二、根因:为什么必须 stdin pipe + 保持打开

`--input-format stream-json` 把 stdin 当**持续双向流**:claude 处理中可能发 `control_request`,**要求父进程通过同一根 stdin 回 `control_response`**。

| 方案 | 结果 |
|---|---|
| stdin pipe,保持打开 | ✅ 正常 |
| `< file` 重定向(文件读完立即 EOF) | ❌ claude 误判会话结束 → **静默 exit 0,无输出** |
| prompt 写命令行 | ❌ 转义地狱、长度限制、泄露进程列表 |

> 经验:WSL 侧曾用 `< file`,调试三天。间歇成功(碰巧那轮没发 control_request)。一度误判「wsl.exe 不转发 stdin」(实测是测试没给子进程启动时间)。

## 三、Windows 侧

```ts
const child = spawn(CLAUDE_EXECUTABLE || 'claude',
  ['-p','--output-format','stream-json','--input-format','stream-json','--verbose',
   '--permission-mode','bypassPermissions','--max-turns',String(n),'--settings',cleanPath,
   ...(resume ? ['--resume', sid] : [])],
  { cwd: workDir, shell: true, stdio: ['pipe','pipe','pipe'] });
child.stdin.write(JSON.stringify({type:'user',message:{role:'user',content:[{type:'text',text:prompt}]}}) + '\n');
// 按行读 stdout...收到 result 才 child.stdin.end()
```

## 四、WSL 侧(从 Windows 后端调 WSL 内 claude)

**wsl.exe 硬事实(实测)**:

| 事实 | 说明 |
|---|---|
| 转发 stdin pipe | `wsl.exe -- bash` 下 Node 写进 stdin 的数据 WSL 内进程能读到(cat 实测) |
| `--` 后参数逐个 argv | 但带空格值(尤其 `bash -c "..."`)被空格拆坏 → **直接 `-- claude <独立参数>`,不走 `bash -c`** |
| `--cd` 在 `--` 前 | `wsl.exe --cd <mnt> -- claude ...`;路径用 `wslpath` 转 `/mnt/c/...` |

```ts
spawn('wsl.exe',
  ['--cd', toWslPath(workDir), '--', 'claude', ...args, '--settings', toWslPath(cleanPath)],
  { shell: false, stdio: ['pipe','pipe','pipe'] });  // ← 同样 stdin pipe
```

**❌ 弯路**:`bash -c "claude..."`(argv 被拆)、`< file`(EOF 静默退出)、WSLENV 桥接数据目录(Windows Terminal 覆盖,见 [WSL↔Windows 通用方案](./20260724095424_WSL与Windows环境变量桥接通用方案.md))。

## 五、配套要点

| 要点 | 做法 |
|---|---|
| 隔离用户配置 | `--settings clean.json` 清 hooks/permissions(否则 SessionStart hook 注入,单轮 input 2.5k→35k)。注意是合并非替换,清不掉用户级 MCP/skills |
| 会话续接 | `--resume <session_id>`(来自 result 事件)。**按 claude 侧分开存**:Windows/WSL 是不同 home、不同 session 池,跨侧 resume 报 `No conversation found` |
| stream-json 事件 | `system(init)`/`assistant`(text/thinking/tool_use)/`user`(tool_result 按 id 回填)/`result`(收尾);其余 log/hook_* 噪音过滤 |
| control_request | bypassPermissions 下罕见,但 MCP 工具等仍可能发。不回 control_response 会挂到 timeout。**正解(对齐 multica):自动 approve tool_use**;最低限度要 warn 留痕 |

## 六、错误处理与进程清理

| 场景 | 处理 |
|---|---|
| claude 不存在 / WSL 没起 | error message 拼 stderr 末尾(否则只显示 exit code) |
| 用户中断 / 客户端断开 | AbortController → `child.kill('SIGTERM')` |
| **WSL 孤儿进程** | `kill wsl.exe` 不保证杀 WSL 内 claude → finally **先 destroy stdio**(断 stdin 让 claude 收 EOF 自退)再 kill |
| SSE 流已 end | 写前 `if (!reply.writableEnded)` 卫语句 |
| sessionId 路径穿越 | URL 来的 sid 拼 `${sid}.jsonl` 前校验 `/^[A-Za-z0-9-]+$/` |

```ts
} finally {
  clearTimeout(timer);
  try { child.stdin?.destroy(); child.stdout?.destroy(); child.stderr?.destroy(); } catch {}
  if (!child.killed && child.exitCode === null) child.kill();
}
```

## 七、决策一览

| 决策 | 选 | 理由 |
|---|---|---|
| prompt 传递 | stdin pipe | 避免转义 + 避免 EOF 静默退出 |
| stdin 生命周期 | 收到 result 才 end | stream-json 是持续流 |
| WSL 调用 | `wsl.exe -- claude` + pipe | argv 不被拆 + stdin 可转发 |
| 权限 | bypassPermissions | headless 无 UI |
| hook 隔离 | --settings clean.json | 省 token |
| session 存储 | 按侧 {windows,wsl} | 两套独立 session 池 |
| 进程清理 | destroy stdio + kill | 防 WSL 孤儿 |
| 多 CLI 扩展 | 适配器模式(每 CLI 一文件 + 接口) | 借鉴 multica `pkg/agent/` |

## 八、避坑清单

| 坑 | 解法 |
|---|---|
| WSL 静默 exit 0 无输出 | 别用 `< file`,改 stdin pipe |
| 误判 wsl.exe 不转发 stdin | 其实转发,测试给子进程启动时间 |
| `bash -c` 在 wsl.exe 下失效 | 直接 `-- claude <args>` |
| resume `No conversation found` | sessionId 按侧存取 |
| input token 暴涨 | --settings clean.json 清 hooks |
| 停止后 claude 还在跑 | AbortController + finally destroy stdio |
| error 只显示 exit code | message 拼 stderr |
| control_request 挂 timeout | 实现 auto-approve |

## 九、multica 借鉴要点(`server/pkg/agent/claude.go`)

- prompt 经 stdin pipe 写 stream-json 帧,写完保持打开,收 result 才 close
- 协议级参数(`-p`/`--output-format`/`--input-format`/`--permission-mode`)黑名单锁死,用户 custom_args 不可覆盖
- `handleControlRequest` 自动 approve 所有 tool_use;`resumeWasRejected` 检测拒绝并丢 sid 重试
- `pkg/agent/` 每 CLI 一文件实现 `Backend` 接口,工厂分发(适配器模式)
- Windows 用 `CREATE_NEW_CONSOLE + HideWindow`(非 CREATE_NO_WINDOW,否则孙子进程弹窗风暴)
- multica **不做 WSL 分支**(宿主机一侧直接 spawn 本机 claude)
