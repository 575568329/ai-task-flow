# WSL ↔ Windows 跨 home 数据目录问题:代码自探测解法(WSLENV 在 Windows Terminal 下失效)

> 适用场景:任何存在「Windows 进程 + WSL 进程」混合拓扑、且用 `~`/`os.homedir()` 定位数据目录的应用。不限于 Node.js。

> ⚠️ **2026-07-24 重大修订**:原方案用 WSLENV 桥接,实测在 **Windows Terminal**(WSL 的主流启动方式)下被进程级 WSLENV 覆盖而**完全失效**——`wsl --shutdown` 重启多少次都没用。**最终改用「代码运行时自探测」**(见第三节),不再依赖任何外部环境变量注入。原 WSLENV 内容降为第六节附录(仅非 WT 启动场景可用)。

## 一、问题本质(什么场景会遇到)

当一个应用同时有跑在 Windows 和跑在 WSL 的进程,用 `~` 或 `os.homedir()` 定位数据/配置目录的代码必然踩坑——两边的 home 是**两个不同的物理目录**。

| 进程侧 | `os.homedir()` / `$HOME` | 物理目录 |
|---|---|---|
| Windows 原生(node 跑在 Windows) | `%USERPROFILE%` | `C:\Users\<user>` |
| WSL(node 跑在 WSL) | `$HOME` | `/home/<user>` |

`/mnt/c/Users/<user>` 是 WSL 看 Windows 文件系统的窗口,但 **`~` 不会自动指向它**。

**典型症状**:
- WSL 里的工具(MCP / CLI)读到空数据,而 Windows 侧明明有数据
- 配置文件两边各写一份,不同步
- 缓存/日志/上传落到错位置,前端找不到

**真实案例**(ai-task-flow):后端跑在 Windows,任务数据写 `C:\Users\fjyu9\.ai-task-flow\tasks.json`(10 个任务 / 2 个待办);但 Claude Code + MCP 跑在 WSL,`os.homedir()` 解析到 `/home/fjyu9`,读 `~/.ai-task-flow/tasks.json`(不存在)→ MCP `list_pending_tasks` 返回空。

## 二、根因

- WSL 与 Windows 是两套独立的用户环境(home、环境变量、PATH 各自独立)
- 代码里散落的 `os.homedir()` / `process.env.HOME` / `process.env.USERPROFILE` 在不同进程侧返回不同值
- 没有「单一解析入口」+「跨环境对齐」时,数据目录必然对不上

## 三、推荐解法:代码运行时自探测(resolveDataDir 内做 WSL 兜底)

**核心思想**:不依赖外部环境变量注入(WSLENV / env var),而是在「数据目录解析函数」内部检测运行环境——WSL 下主动探测 Windows 用户目录,转成 WSL 路径,指向同一份数据。

```ts
// resolveDataDir:单一解析入口,优先级 显式 > env > WSL 自探测 > 默认 home
import path from 'node:path';
import os from 'node:os';
import { execFileSync } from 'node:child_process';
import { readFileSync } from 'node:fs';

const IS_WSL: boolean = (() => {
  if (process.env.WSL_DISTRO_NAME) return true;
  try { return readFileSync('/proc/version', 'utf8').toLowerCase().includes('microsoft'); }
  catch { return false; }
})();

let cachedWin: string | null | undefined; // undefined=未探测 / null=失败 / string=结果
function detectWindowsHome(): string | null {
  if (cachedWin !== undefined) return cachedWin; // 模块级缓存,只 spawn 一次
  cachedWin = null;
  try {
    // appendWindowsPath=false 下 cmd.exe 不在 WSL PATH,用绝对路径
    const win = execFileSync('/mnt/c/Windows/System32/cmd.exe', ['/c', 'echo %USERPROFILE%'],
      { encoding: 'utf8', stdio: ['ignore', 'pipe', 'ignore'] }).trim();
    if (win) {
      // wslpath:C:\Users\<user> → /mnt/c/Users/<user>
      cachedWin = execFileSync('wslpath', [win],
        { encoding: 'utf8', stdio: ['ignore', 'pipe', 'ignore'] }).trim() || null;
    }
  } catch { /* 探测失败(非标准 WSL / cmd.exe 缺失):回退默认 home */ }
  return cachedWin;
}

export function resolveDataDir(explicit?: string): string {
  if (explicit?.trim()) return path.resolve(explicit.trim());
  const fromEnv = process.env.APP_DATA_DIR?.trim();
  if (fromEnv) return path.resolve(fromEnv);
  if (IS_WSL) {
    const win = detectWindowsHome();
    if (win) return path.join(win, '.app-data'); // ← WSL 进程读到 Windows 侧数据
  }
  return path.join(os.homedir(), '.app-data'); // 默认(各侧 home)
}
```

要点:
- **单一解析入口 + 优先级链**:显式 > env > WSL 自探测 > 默认 home,层层兜底,任一层失效自动落到下一层。
- **模块级缓存**:cmd.exe + wslpath 只在首次解析时 spawn 一次,无性能负担。
- **失败优雅降级**:探测失败(非标准 WSL)回退默认 home,不崩。
- **多设备通用**:`%USERPROFILE%` 自动适配本机用户名,零硬编码。
- **零外部配置**:clone 即用,不写注册表、不依赖 WSLENV、不需要 `wsl --shutdown`。

## 四、为什么不用 WSLENV(关键踩坑)

WSLENV 是微软官方的 Win↔WSL 环境变量桥接机制,理论上完美。**实际部署栽在 Windows Terminal 上**:

- Windows Terminal 启动 wsl.exe 时,会用自己的**进程级 WSLENV**(只含 `WT_SESSION/u`、`WT_PROFILE_ID/u`,用于把这俩变量传进 WSL)。
- 进程级 WSLENV **覆盖(而非合并)注册表 User 级 WSLENV** → 写在注册表的 `AI_TASK_FLOW_DATA_DIR/up` **永远进不来**。
- 现象:`wsl --shutdown` 重启多少次都没用——只要 Claude Code/MCP 是从 Windows Terminal 启动(主流场景),桥接变量就是空,WSLENV 那套「一键配置」形同虚设。
- **诊断线索**:WSL 内 `$WSLENV` 只剩 `WT_SESSION:WT_PROFILE_ID`,但 `[Environment]::GetEnvironmentVariable('WSLENV','User')` 明明返回 `AI_TASK_FLOW_DATA_DIR/up`;父进程链 `claude ← bash ← Relay ← SessionLeader` 证实从 WT 启动。手动 `export AI_TASK_FLOW_DATA_DIR=/mnt/c/...` 后立刻能读到数据,反证问题纯在 env 注入层。

> **结论**:WSLENV 桥接「依赖宿主启动器不覆盖」,这个前提在主流终端(WT)上不成立。**环境注入类方案天然脆弱**(受宿主行为左右,且静默失效极难排查);代码自探测则自洽,不受启动方式影响。

## 五、决策原则:运行时自探测 > 外部环境注入

| 维度 | 外部环境注入(WSLENV/env + 配置脚本 + UI) | 代码运行时自探测 |
|---|---|---|
| 可靠性 | 依赖「宿主不覆盖 + 用户正确配置 + 重启时机」 | 代码自己探测,行为确定 |
| 零配置 | 需「一键配置」按钮 + `wsl --shutdown` | clone 即用 |
| 可移植 | 每台设备都要配一遍 | 一份代码全平台 |
| 维护面 | 脚本 + Service + API + UI + 文档 | 一个解析函数 |
| 失败模式 | **静默失效**(MCP 拉空,极难排查) | 探测失败优雅降级 |

**原则一句话**:**优先让代码在运行时自探测所需的环境信息,而非要求外部(用户 / 终端 / WSL 启动器)预先配置好环境变量。** 自探测是自洽的(self-contained)——行为可控、可移植、零配置;环境注入依赖一长串外部前提,任一环失效就静默崩(本次 MCP 拉空任务,前后查了「数据目录 / WSLENV 标志 / wsl --shutdown」好几轮,才定位到是 WT 覆盖了 WSLENV)。

**适用判断**:
- 需要「跨进程边界共享同一物理路径」、且进程可能跑在不同 home → **优先代码自探测**。
- 仅当自探测代价过高(需凭据、需用户抉择、需业务参数)时,才退而求其次走「检测→提示→修复」的环境配置 UI,并接受其脆弱性。

## 六、附录:WSLENV 方案(仅非 Windows Terminal 启动场景可用)

> ⚠️ 本项目已弃用此方案。仅当 WSL **不经 Windows Terminal** 启动(从 `cmd` / Win+R / VS Code 集成终端 / 纯 `wsl.exe`)时,WSLENV 桥接才有效。以下保留供那些场景参考。

**原理**:Windows 侧设置的环境变量,通过 `WSLENV` 声明,在 WSL 启动时自动注入到 WSL 进程,并可选做路径格式转换。

### 三步落地

1. Windows User 级设业务变量(值用 `%USERPROFILE%` 自动适配本机用户名)
2. `WSLENV` 追加 `VARNAME/up`(`/u` 方向 + `/p` 路径转换)
3. `wsl --shutdown` 重启 WSL 生效(WSLENV 只在 WSL 启动时注入,不热加载)

### WSLENV 标志含义

| 标志 | 含义 |
|---|---|
| `/u` | 仅 Win32→WSL 方向传递 |
| `/w` | 仅 WSL→Win32 方向传递 |
| `/p` | 路径转换(`C:\Users\x` ↔ `/mnt/c/Users/x`) |
| `/l` | 路径列表转换(分号 ↔ 冒号) |

最常用组合 `/up`:Windows→WSL 单向传递 + 自动路径转换。**只用 `/u` 不加 `/p` 会拿到未转换的 Windows 路径,必须 `/up`。**

### PowerShell 桥接脚本(幂等模板)

```powershell
$ErrorActionPreference = "Stop"
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8
$VAR_NAME = "APP_DATA_DIR"
$dataDir = Join-Path $env:USERPROFILE ".app-data"
[Environment]::SetEnvironmentVariable($VAR_NAME, $dataDir, "User")
$cur = [Environment]::GetEnvironmentVariable("WSLENV", "User")
$parts = if ($cur) { $cur -split ";" } else { @() }
$parts = $parts | Where-Object { $_ -and ($_ -split "/")[0] -ne $VAR_NAME }
$parts += "$VAR_NAME/up"
[Environment]::SetEnvironmentVariable("WSLENV", ($parts -join ";"), "User")
```

### 检测 → 提示 → 修复(doctor --fix)

业界(VSCode Remote-WSL、Docker Desktop、conda doctor)惯例:不静默改环境变量,而是「检测状态 → 提示用户 → 一键修复」。落地为 `status` + `setup` 端点对 + 设置面板开关。

## 七、避坑清单

| 坑 | 解法 |
|---|---|
| **WSLENV 被 Windows Terminal 进程级覆盖(从 WT 启的 WSL 拿不到桥接变量,重启也没用)** | **改用代码自探测**(第三节);若坚持 WSLENV,需从非 WT 启动 WSL |
| 散用 `os.homedir()`/`~`/`$HOME` 定位数据目录 | 走 `resolveDataDir` 单一入口,WSL 下自探测 Windows home |
| WSLENV 只写 `/u` | 用 `/up`(路径转换)——仅非 WT 场景 |
| `setx` 不展开 `%VAR%` | 用 PowerShell `[Environment]::SetEnvironmentVariable()` + `$env:USERPROFILE` |
| `appendWindowsPath=false` | WSL 里调 cmd/powershell 用绝对路径 `/mnt/c/Windows/System32/...` |
| 含中文 `.ps1` 不加 BOM | UTF-8 with BOM(`EF BB BF`),否则 PS 5.1 按 GBK 误读报 `TerminatorExpectedAtEndOfString` |
| WSL 下跑 `vite build` | node_modules 在 Windows 装的,缺 `@rollup/rollup-linux-x64-gnu`;在 Windows 跑构建,WSL 只做 `tsc` |
| 改完环境变量静默不生效 | 别假设生效,要回读校验;代码自探测从根上避开此问题 |

## 九、本质教训(脱离 WSL 的通用准则)

本次踩坑最值得带走的不是 WSLENV 细节,而是两条可复用的决策准则:

1. **别把正确性寄托在你管不了的外部环境上。** 代码要能跑,却依赖「环境变量被正确注入 / 启动器不覆盖 / 用户正确配置 / 重启时机」这一长串你无法控制的前提——宿主只要做一件你预料不到的事(本案:Windows Terminal 用进程级 WSLENV 覆盖注册表),整套就静默失效,且极难排查。
2. **能代码运行时探测的,就别让用户/终端去配置。** 凡是代码能在运行时自己问出来的信息(本机 home、平台、路径),就别外置成环境变量、配置文件或「一键配置」UI。代码探测是自洽的(self-contained):行为可控、零配置、可移植、失败可优雅降级;外部配置则把维护成本和失效风险转嫁给了用户与环境。

> **推论**:选方案时问一句——「如果这步依赖的外部前提全部失效,我的代码还会对吗?」不会、且能用代码兜底,那就兜。
