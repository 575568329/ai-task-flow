# AI 执行任务的脚本化策略:什么该沉淀,存哪,如何复用

> 讨论日期:2026-07-01
> 核心问题:AI 执行任务时是否应尽量写脚本来完成?脚本可复用、准确率高、省 token,但劣势是什么?脚本存哪里?能否封装重复脚本?

## 一、结论先行

**业界共识**:2026 年 Claude Code / Codex 社区已形成明确的黄金法则:

> **"Anything done twice should become a skill."**  
> —— 引导 AI 走过同一流程超过一次,就该沉淀成 skill/脚本。  
> (来源:Armin Ronacher《Agentic Coding Recommendations》,Zyte 团队博客均有相同表述)

**但脚本化不是全场景通吃**。需要按任务类型判断,并且**关键要让 AI 知道你有什么脚本**——否则脚本库形同虚设。

---

## 二、脚本化的优势

| 优势 | 具体表现 |
|-----|---------|
| **Token 省** | 脚本不进上下文,AI 只需 `bash script.sh` 一句话。相比 AI 逐步生成命令,能省 10~20 倍上下文消耗 |
| **准确率高** | 脚本是**确定性**执行,AI 每次重新生成可能出细微差别(变量名、参数顺序、边界处理) |
| **可复用** | 复杂逻辑一次调好,以后所有类似任务都调它 |
| **可测试** | 脚本能单独跑、单元测试,AI 每次生成的一次性代码难以测试 |
| **可审计** | 脚本进 git 有版本历史,能追溯谁改的、为什么;AI 每次生成的代码没这层追踪 |
| **跨会话稳定** | 会话切换、模型升级都不影响脚本行为,AI 的每次生成结果会漂移 |

---

## 三、脚本化的劣势(容易忽略但很重要)

### 1. 上下文遗忘 (Context Rot)

脚本存在磁盘上,但 **AI 不知道它存在**,除非:
- `CLAUDE.md` 里明确列了
- 或者当次任务里让 AI 先 `ls scripts/` 搜索

**结果**:你有一堆好脚本,AI 却重新写了个功能类似的新脚本。这是**脚本库最大的失败模式**。

### 2. 僵化 (Rigidity)

脚本一旦定稿,只解决**它被写出来的那个场景**。稍微换个需求(比如从"扫描 md"变成"扫描 md 和 pdf"),AI 要么改脚本(有风险破坏其他调用者)、要么写新的(违背复用初衷)。

### 3. 抽象层泄漏

复杂脚本一旦封装成"黑盒调用",出错时 AI 不知道内部逻辑,debug 变难。反倒是"AI 每次生成简单代码"更透明可读。

### 4. 维护成本

脚本多了要维护:测试、文档、废弃标记。**5 个不用的脚本比 50 个混乱脚本更危险**,因为 AI 可能挑到过时的用。

### 5. 失去灵活性

原生 AI 生成的价值之一是**灵活适配上下文**,脚本化会牺牲这点。比如"生成 manifest"这种确定性任务适合脚本,"根据这次对话内容整理文档"就不适合。

---

## 四、判断表:什么该脚本化,什么不该

| 场景特征 | 该脚本化? | 例子 |
|---------|:---:|------|
| **确定性输入输出**,规则明确 | ✅ | 生成 manifest、下载图片、部署脚本 |
| **副作用敏感**(改文件、调 API) | ✅ | 数据库迁移、批量重命名 |
| **重复次数 ≥ 3** | ✅ | 每次都跑的 lint/build/test |
| **要多人/多机器一致** | ✅ | CI/CD、初始化脚本 |
| **一次性任务** | ❌ | "帮我看下这个 bug" |
| **需要根据上下文调整** | ❌ | "根据这次讨论整理文档" |
| **探索/调试性任务** | ❌ | "试试能不能优化性能" |
| **需要人在环判断** | ❌ | 代码审查、方案讨论 |

## 五、存哪里:两套方案

### 方案 1:项目内 `scripts/` 目录 + CLAUDE.md 索引

```
项目/
├── scripts/
│   ├── generate-manifest.ps1
│   └── serve.js
└── CLAUDE.md  ← 关键:必须在这里索引脚本清单
```

- **优点**:跟项目走,git 版本化,协作者 clone 就有
- **关键**:CLAUDE.md **必须**列出脚本清单和用途,否则 AI 找不到 = 不存在

### 方案 2:Claude Code Skills(2026 官方标准)

```
项目/.claude/skills/generate-docs/
├── SKILL.md          ← 元数据 + 触发条件 + 使用说明
├── scripts/          ← 实际脚本
│   └── generate.ps1
└── references/       ← 参考资料(文档、模板)
```

或全局位置 `~/.claude/skills/`(所有项目共享)。

**核心机制**:SKILL.md 的 `description` 字段被 Claude 用来判断**什么任务该触发这个 skill**——直接解决"AI 不知道脚本存在"的问题。

### 生态规模数据(2026)

- GitHub `alirezarezvani/claude-skills`:337 个 skills + 30+ 插件
- `Antigravity Awesome Skills`:1234+ 社区维护的 skill
- Anthropic 官方 skills(PDF、Excel、PPT 操作等)
- 统一 SKILL.md 格式,同一份 skill 可跨 Claude Code / Cursor / Codex CLI / Gemini CLI 通用

## 六、当前项目已有脚本盘点

知识库项目 `scripts/` 目录当前脚本清单:

### `generate-manifest.ps1`

**用途**:扫描 `docs/` 目录递归生成 `manifest.json`(前端目录树的数据源)。

**触发时机**:
- 添加、删除、重命名 `docs/` 内文档后
- `启动.bat` 启动时会自动调用一次
- `scripts/serve.js` 检测到 fs.watch 事件时自动调用
- GitHub Actions 部署前也会调用(通过 `.github/workflows/pages.yml`)

**技术要点**:
- 支持格式:md、markdown、pdf、docx、html、htm、png/jpg/jpeg/gif/webp/svg/bmp/ico
- 自动去掉排序前缀(`01_`、`02-`)作为展示标题
- 输出 UTF-8 无 BOM(避免 fetch JSON 解析问题)
- **脚本本身必须存为 UTF-8 with BOM**,否则 PowerShell 5.1 中文注释乱码导致解析报错

**调用**:`powershell -File scripts/generate-manifest.ps1`

### `serve.js`

**用途**:Node 零依赖静态服务器 + 本地增强 API。

**能力**:
- 静态资源服务(所有文件)
- `GET /api/status` — 前端探测本地模式
- `GET /api/events` — SSE 推送 docs/ 变化事件
- `POST /api/upload` — multipart 上传文件到 docs/
- `DELETE /api/file` — 删除 docs/ 内文件
- fs.watch 监听 docs/ 目录,变化自动重生成 manifest 并广播 SSE

**触发时机**:双击 `启动.bat` 会自动调 `node scripts/serve.js 8850`

### `analyze-conversation.ps1` 和 `analyze-conversation-v2.ps1`

**用途**:分析 Claude Code 的会话 JSONL 文件,提取问题、方案、决策和经验,生成结构化的复盘文档。

**输入**:Claude Code 存的对话 JSONL 文件,通常在 `C:\Users\<user>\.claude\projects\<项目slug>\<uuid>.jsonl`

**输出**:`docs/项目复盘/` 下的 md 复盘文档

**v1 vs v2 区别**:
- v1:原版,提取所有对话内容
- **v2 是优化版**(推荐用),智能过滤:
  - 只提取用户的真实消息(过滤掉工具结果/系统提示)
  - 只提取助手的文本回复(过滤掉工具调用的 JSON)
  - 额外统计工具使用情况

**调用示例**:
```powershell
.\analyze-conversation-v2.ps1 `
  -SessionFile "C:\Users\fjyu9\.claude\projects\D--Study-KnowledgeBase\<uuid>.jsonl" `
  -ProjectName "KnowledgeBase"
```

**典型用途**:一次长对话结束后,生成对话复盘存进知识库,方便以后查阅这次讨论了什么、决策了什么。

### `serve.ps1`(已弃用)

**状态**:早期版本,基于 .NET HttpListener 的纯 PowerShell 静态服务器。因请求循环有 bug 返回 500,已被 `serve.js` 取代。保留仅供参考,不应再调用。

## 七、封装重复脚本可行吗?

**可行,但要有原则**。这是软件工程的经典 DRY 问题,不是 AI 特有:

### ✅ 值得做

- **参数化**:两个脚本除了一两个参数不同 → 合并 + 加参数
- **提炼公共函数**:PowerShell 模块 / Node module,多个脚本 import
- **v1/v2 迭代**:像 `analyze-conversation-v2.ps1` 这种,新版稳定后删掉 v1

### ⚠️ 慎重

- **不要过早抽象**:两个脚本表面像但语义不同(比如都叫 "download",一个下载图片一个下载依赖),不要合并
- **给 AI 用的脚本要"扁平"**:少嵌套调用,AI 才好理解和调试
- **接口稳定性**:一旦 AI 学会调某个脚本,轻易改参数会引起混乱

## 八、当前项目改进建议

对本知识库项目,建议按以下优先级落地:

### 短期(马上做)

在**项目根** CLAUDE.md 或 `.claude/CLAUDE.md` 加一段**脚本地图**:

```markdown
## 项目脚本清单
优先复用这些脚本,不要重复造轮子。

- `scripts/generate-manifest.ps1` — 扫描 docs/ 生成清单,添加/删除文档后调用
- `scripts/serve.js` — Node 静态服务器 + 本地 API,双击启动.bat 会自动调
- `scripts/analyze-conversation-v2.ps1` — 分析 Claude 会话 JSONL 生成复盘文档
- ~~scripts/analyze-conversation.ps1~~ — 老版本,已被 v2 取代
- ~~scripts/serve.ps1~~ — 已弃用,由 serve.js 取代

**规则**:实现新功能前,先看这里有没有可复用的脚本。
```

这样 AI 每次会话都能读到,不会重复造轮子。

### 中期(项目稳定后)

考虑把常用的封装成 `.claude/skills/`:

- `kb-add-doc` — 添加文档 + 重生 manifest 的完整流程
- `kb-review-session` — 一键生成本次会话复盘

### 长期(通用工作流)

反复用到的**跨项目**脚本(不特定于知识库),放全局 `~/.claude/skills/`。比如"下载 CDN 库到 vendor 目录"这种通用逻辑。

## 九、一句话总结

> **"两次以上就沉淀,但记得让 AI 知道你沉淀了什么"**

这是 2026 年 agentic coding 的黄金法则。本项目已经有 `scripts/` 目录,现在最缺的关键动作是**在 CLAUDE.md 里给脚本建索引**,让每次会话的 AI 都能找到并复用。
