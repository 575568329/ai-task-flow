# 调研文档：调研对话 Agent 化 & Electron 桌面化评估

> 日期：2026-07-23
> 性质：可行性 + 性价比调研，供权衡决策，**非实施方案**。
> 信息来源：项目源码（ChatService / KnowledgeService / KnowledgeScanner / 各路由）、GitHub API 与原始 README 核实（第二轮验证）、网络最佳实践检索。

## 背景与两个问题

- **问题 1**：网页版"资料调研"对话（`research/ChatService`，类 Perplexica）目前只做"分类→web检索→流式生成→存聊天记录"，**既不读本地项目文件/已有笔记，也不把结论归档进知识库**。期望：让调研像 agent 一样能读写项目资料、自动整理进知识库。
- **问题 2**：当前是 `backend(Node+Fastify+MCP) + frontend(React19) + 浏览器扩展(剪藏) + CLI + 双层存储(~/.ai-task-flow/ + 项目 worktree)` 的 Web 形态。评估改成 **Electron 桌面端** 是否更方便。

# 第一部分：调研对话 Agent 化（问题 1）

## 1. 现状（已核实）

| 组件 | 能力 | 是否接入调研链路 |
|---|---|---|
| `ChatService.handleChat` | 分类→改写→web检索→RAG流式→引用校验→存聊天 | ✅ 主链路，但止步于此 |
| `SearchOrchestrator.search()` | 仅 web + arxiv 检索 | ❌ 无本地源 |
| `KnowledgeService.createDoc({title,content,tags,dir})`（MCP `save_to_knowledge` 用）/ `saveDoc(path,content)`（HTTP `/save` 用） | 两条写入入口均已就绪 | ❌ 均未接入调研 |
| `KnowledgeScanner.scanKnowledge()` | 列出知识库文档元数据 | ❌ 无语义检索 |
| `save_to_knowledge` MCP 工具 | 同 saveDoc 内核，暴露给 Claude Code | ❌ 仅 MCP 客户端可用 |

**结论**：零件齐全，缺的是"把调研接进本地读写闭环"。这是**功能补全**，不是推倒重来。

## 2. 效果阶梯（最小可用 → 理想最佳）

| 等级 | 效果 | 实现复杂度 |
|---|---|---|
| **L1 最小可用** | 调研可勾选"参考本地资料"，回答能引用 knowledge-base 文档片段；assistant 消息可一键归档进知识库 | 低 |
| **L2 增强** | 本地检索升级为语义检索（向量）；归档支持自动摘要、自动归类、去重 | 中 |
| **L3 理想最佳（Agent 化）** | 调研变成 agent loop，模型自主决定何时读文件/检索/写知识库；支持多轮工具调用；可跨知识库+源码语义检索；调研报告自动结构化入知识库并更新 MOC 索引 | 高 |

## 3. 技术路线对比

| 路线 | 做法 | 新增依赖 | 时间 | 效果上限 |
|---|---|---|---|---|
| **A1 轻量** | ChatService 加"关键词检索本地文档→喂 context" + 前端归档按钮复用 `createDoc` | 无 | **2~3 天** | L1 |
| **A2 完整 RAG** | 给知识库加 embedding 向量索引，语义检索 | embedding 模型 + 向量存储 | 5~7 天 | L2 |
| **B 接入外部 agent** | 让 `open_deep_research`（原生支持 MCP）挂载本项目 MCP，白嫖读写知识库能力 | **引入 Python / LangChain**（技术栈分叉） | 3~5 天集成 | L3 |
| **C 自研 agent loop** | 在 backend 里实现工具调用循环（读文件/检索/写库都做成工具） | 无强依赖，工作量大 | 1~2 周 | L3 |

## 4. 时间估算（人天）

| 路线 | 最小可用 | 到效果上限 |
|---|---|---|
| A1 | 2~3 天 | 同（上限即 L1） |
| A2 | — | 5~7 天（L2） |
| B | 3~5 天（集成+联调） | 后续迭代至 L3 |
| C | 1 周 | 2 周（L3） |

## 5. 技术栈 / 项目影响

- **A1**：零影响。纯 TypeScript，复用现有 `KnowledgeScanner`/`KnowledgeService`，符合项目 KISS/YAGNI。
- **A2**：新增 embedding 依赖（本地 `transformers.js`，或调用已配置的 LLM embedding API）+ 向量索引文件。项目仍单语言 TS，但多一层向量维护（文档变更要重索引）。
- **B**：**技术栈分叉**——Python/LangChain 跑在外部进程，与本项目 Node 后端通过 MCP 通信。运维复杂度上升（需装 Python、管两套进程），但能直接复用 open_deep_research 的成熟 agent 能力，长期演进空间最大。
- **C**：自研 agent loop，技术栈不变但代码量与维护成本显著增加，且要自己处理工具调用编排、错误恢复、上下文管理。

## 6. 风险与权衡

| 风险/权衡 | 说明 |
|---|---|
| 关键词检索召回不准（A1） | 知识库体量小时够用；预留检索接口，未来可热替换为 A2 向量版，不动上层 |
| 本地文档过大拖慢首字 | 检索设超时、只取片段，不全量入 context |
| 归档质量参差 | L1 采用"用户手动归档+可编辑标题/目录"，可控；L3 自动归档需配去重/校验 |
| 技术栈分叉（B） | 引入 Python 后，部署/打包/CI 都要兼顾两套；与未来 Electron 化叠加更复杂 |
| 与现有写入能力重复 | MCP 走 `createDoc`、HTTP 走 `saveDoc`，是两个不同方法；A1/C 归档应统一复用 `createDoc`（带 title/tags/dir，语义更贴合"归档"），避免造第三个写入入口 |

## 7. 适用场景判断

- 知识库规模不大、想最快见效 → **A1**。
- 已有 embedding 基础设施、追求检索精度 → **A2**。
- 想要最强 agent 自主性、能接受 Python 栈 → **B**。
- 不想引入外部依赖、又要 L3 → **C**（成本最高）。

# 第二部分：Electron 桌面化（问题 2）

## 1. 现状与动机

当前 Web 形态痛点：
- 需用户自备 Node/npm 环境，`npm install`/源/lock 屡屡踩坑（科大讯飞内网源等）。
- 前后端分离，dev 要按顺序起 shared→backend→frontend。
- 浏览器扩展访问 localhost 受 PNA/CORS 限制，需 `text/plain` 绕过。
- 前端靠文件轮询刷新看板，非实时。

## 2. 能升级什么 / 会丢失什么

### ✅ 会升级
| 项 | 说明 |
|---|---|
| 免环境 | 双击安装即用，绕开 npm/源/lock 全部坑 |
| 原生文件系统 | 知识库扫描/监听更快，可 watch 实时刷新（告别轮询） |
| 消除 CORS/PNA hack | renderer↔main 走 IPC，localhost 跨域问题消失 |
| 系统集成 | 托盘、全局快捷键、原生通知、开机自启、单实例 |
| MCP 内置 | main 进程直接 host MCP server，Claude Code 连本地更顺 |
| git worktree | 直接系统 git，更稳 |

### ❌ 会丢失或需重做
| 项 | 处理 |
|---|---|
| **浏览器扩展（剪藏）** | 最大流失点。Electron 无浏览器。建议：扩展继续独立存在，桌面端仍开 HTTP server 接收剪藏（改动最小） |
| **CLI / npm 包定位** | 桌面版里 CLI 意义减弱，但 **MCP server 必须保留**（Claude Code 终端依赖它），故 backend 建议仍作为独立可执行 |
| **存储路径** | `~/.ai-task-flow/` 需决策：用 `app.getPath('userData')` 或显式保持原路径兼容老数据 |
| **多终端 worktree 链路** | 不受影响（worktree 在磁盘、Claude Code 在终端），但需防 Electron 抢占 git/端口 |

## 3. Electron vs Tauri 对比（关键决策）

| 维度 | Electron | Tauri v2 + Node sidecar |
|---|---|---|
| 体积 | 重（Chromium+Node 全打包） | 核心 <600KB，但 Node sidecar 仍 +~84MB |
| 内存 | 高 | 比 Electron 低约 50% |
| Node 集成 | main 进程原生跑，深度集成简单 | Node 作为 sidecar 隔离进程，IPC 有开销 |
| 对本项目适配 | **优**：Fastify/MCP stdio/git worktree 直接 main 跑 | sidecar 跑 worktree/MCP stdio 更繁琐 |
| 现成参照 | **Cherry Studio（48.9k★，同栈+MCP+知识库）可直接抄** | 无这么贴切的参照 |
| 成熟度 | 非常成熟（VS Code/Slack） | v2 仍在成熟，sidecar 工作流演进中 |

**倾向**：本项目后端是 Node 且深度依赖 git/MCP，**Electron 更顺**；Tauri 适合"未来想瘦身"时再评估迁移。

## 4. 时间估算（阶梯）

| 阶段 | 范围 | 估时 |
|---|---|---|
| 最小可用壳 | Electron 加载现有 React build + 内嵌 backend 子进程 + 去掉 CORS hack | **1~2 周** |
| 功能对齐 | 系统托盘、文件监听、知识库原生集成、单实例 | +1~2 周 |
| 浏览器扩展方案 | 保留扩展+HTTP 通信（推荐） | +3~5 天 |
| 打包发布 | electron-builder + 自动更新 + 签名 | +3~5 天 |
| **到稳定桌面版** | | **合计 4~6 周** |

## 5. 技术栈 / 代码改动影响

- **新增**：`desktop/`（electron main + preload + 打包配置）；`electron-builder` 或 `electron-forge`。
- **frontend**：baseURL/端口发现改为 preload 注入；删除 PNA/CORS 绕过逻辑；其余 React 代码基本不动。
- **backend**：增加"被内嵌启动"模式（指定端口、关闭部分 standalone 行为）；**建议保留独立可执行**，供 Claude Code 终端链路使用。
- **构建链**：多一套打包/签名/自动更新流程，CI 复杂度上升。
- **发布形态**：从 npm 包 → 多平台安装包（Win/Mac/Linux），分发渠道变化。
- **与问题 1 的叠加**：若同时走 B（Python agent），桌面打包还要塞 Python 运行时，体积与复杂度暴涨——**两条路若都选最重，需谨慎**。

## 6. 参考项目（已核实真实性与能力）

| 项目 | 规模 | 价值 | 核实结论 |
|---|---|---|---|
| **CherryHQ/cherry-studio** | 48.9k★ | Electron+React+TS+MCP+知识库，几乎完美对应 | ✅ README 确认 MCP Server + Knowledge Management，今日仍更新 |
| **chatboxai/chatbox** | 41k★ | 轻量 Electron AI 客户端 | ✅ 真实（正确路径 chatboxai/chatbox） |
| langchain-ai/open_deep_research | 12.4k★ | 路线 B 的 agent 主体 | ✅ README 确认 full MCP compatibility + 写报告 |
| zilliztech/deep-searcher | 8k★ | 本地向量库 deep research | ⚠️ 真实但半年未更新（2025-11） |
| Electron 迁移实践 | — | Slack/LogRocket/Rolique 文章 | ✅ 普遍结论：纯 SPA 包壳几小时；内嵌后端显著加复杂度 |

## 7. 风险与权衡

| 风险/权衡 | 说明 |
|---|---|
| 工作量被低估 | 内嵌 Node 后端 + 打包签名是主要耗时点，非"套个壳"那么简单 |
| 体积/内存 | Electron 通病；若在意，Tauri 是替代但适配成本更高 |
| 浏览器扩展去留 | 砍掉则失剪藏能力；保留则桌面端仍需开 HTTP server，未完全脱离 Web 形态 |
| 双形态维护 | 若保留 Web + 桌面两套，需共享代码（参考 Slack 文章），否则分叉 |
| backend 定位 | 必须保留独立进程给 Claude Code，不能"全塞进 Electron" |
| 与现有 npm 发布冲突 | 桌面包与 npm CLI 两条分发线并存，需理清 |

## 8. 适用场景判断

- **值得做**：希望降低用户安装门槛、想要系统集成（托盘/快捷键/通知）、想干掉 CORS hack 与轮询。
- **暂缓做**：若核心价值仍是"Claude Code 终端 + 看板"的工作流，Web 形态已够用；桌面化是体验升级而非功能必需。
- **不建议**：为了"纯桌面"砍掉浏览器扩展或 backend 独立进程——会丢核心能力。

# 总览：两条路线的性价比

| 维度 | 问题 1（Agent 化） | 问题 2（Electron 化） |
|---|---|---|
| 性质 | 功能补全 | 形态升级 |
| 最小见效 | 2~3 天（A1） | 1~2 周（最小壳） |
| 效果上限 | L3 全自动 agent | 稳定桌面版（4~6 周） |
| 技术栈影响 | A1/A2/C 无；B 引入 Python | 引入 Electron 构建链/打包 |
| 对现有架构冲击 | 小（增量） | 中~大（新增 desktop 层 + 发布形态变化） |
| 可逆性 | 高（增量功能） | 中（一旦双形态需持续维护） |
| 推荐起步 | A1（低风险快见效） | 先搭最小壳验证体验是否值回票价 |

## 决策需权衡的关键问题

1. **问题 1 走哪条**：零依赖快见效（A1） vs 最强但引入 Python（B） vs 自研（C）。
2. **问题 2 是否值得**：体验升级的价值是否抵 4~6 周成本 + 长期双形态维护。
3. **若两者都做**：避免 B（Python）+ Electron 叠加导致打包爆炸；建议问题 1 走 A1，与 Electron 化无冲突。
