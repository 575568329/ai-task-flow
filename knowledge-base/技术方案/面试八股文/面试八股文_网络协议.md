## 一、HTTP 协议

### 1. HTTP 版本演进

| 版本 | 特性 | 性能 |
|------|------|------|
| **HTTP/1.0** | 短连接,每次请求建立 TCP | 慢(握手开销大) |
| **HTTP/1.1** | 持久连接/管道化/分块传输 | 队头阻塞问题 |
| **HTTP/2** | 多路复用/头部压缩/服务端推送 | 快,但 TCP 层仍有队头阻塞 |
| **HTTP/3** | 基于 QUIC(UDP)/多路复用/0-RTT | 最快,无队头阻塞 |

### 2. HTTP/1.1 vs HTTP/2

**HTTP/1.1 问题**:
1. **队头阻塞**:同一 TCP 连接,前面请求未完成阻塞后续
2. **并发限制**:浏览器对同一域名限制 6 个并发连接
3. **头部冗余**:每次请求都发完整头部

**HTTP/2 改进**:

```
单一 TCP 连接
   ├── Stream 1(请求 A)
   ├── Stream 2(请求 B)  ← 并行,不阻塞
   └── Stream 3(请求 C)
```

- **多路复用**:一个连接并发多个请求
- **头部压缩**:HPACK 算法
- **二进制分帧**:更高效解析
- **服务端推送**:主动推送资源

**HTTP/2 示例**:

```
// 传统优化(雪碧图/域名分片)在 HTTP/2 下反而有害
// HTTP/2:多个小文件 > 一个大文件(缓存粒度更细)
```

### 3. HTTP/3(QUIC)

**为什么基于 UDP**:
- TCP 内核实现,难升级
- UDP 用户态,灵活

**QUIC 特性**:
- **0-RTT 连接**:首次 1-RTT,后续 0-RTT
- **无队头阻塞**:流级别独立
- **连接迁移**:网络切换(WiFi→4G)连接不断

### 4. HTTP 方法

| 方法 | 幂等 | 安全 | 作用 |
|------|------|------|------|
| **GET** | ✅ | ✅ | 获取资源 |
| **POST** | ❌ | ❌ | 创建资源 |
| **PUT** | ✅ | ❌ | 更新资源(全量) |
| **PATCH** | ❌ | ❌ | 更新资源(部分) |
| **DELETE** | ✅ | ❌ | 删除资源 |
| **HEAD** | ✅ | ✅ | 获取头部(不含 body) |
| **OPTIONS** | ✅ | ✅ | 查询支持的方法(CORS 预检) |

**幂等**:多次执行结果相同(PUT/DELETE 是,POST 不是)。

### 5. HTTP 状态码

**1xx 信息**:
- `101 Switching Protocols`:WebSocket 升级

**2xx 成功**:
- `200 OK`
- `201 Created`:创建成功
- `204 No Content`:成功但无响应体

**3xx 重定向**:
- `301 Moved Permanently`:永久重定向(SEO 传递权重)
- `302 Found`:临时重定向
- `304 Not Modified`:协商缓存命中

**4xx 客户端错误**:
- `400 Bad Request`:参数错误
- `401 Unauthorized`:未认证
- `403 Forbidden`:无权限
- `404 Not Found`
- `405 Method Not Allowed`:方法不支持

**5xx 服务端错误**:
- `500 Internal Server Error`
- `502 Bad Gateway`:网关错误
- `503 Service Unavailable`:服务不可用

### 6. HTTP 头部(重要)

**通用头部**:
```http
Cache-Control: max-age=3600
Connection: keep-alive
Date: Mon, 02 Jul 2026 12:00:00 GMT
```

**请求头部**:
```http
Accept: application/json
Accept-Encoding: gzip, deflate, br
Accept-Language: zh-CN,zh;q=0.9
Authorization: Bearer <token>
Cookie: session_id=abc123
Host: example.com
Referer: https://google.com
User-Agent: Mozilla/5.0...
```

**响应头部**:
```http
Content-Type: application/json; charset=utf-8
Content-Length: 1234
Content-Encoding: gzip
Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Strict
Access-Control-Allow-Origin: *
ETag: "abc123"
Last-Modified: Mon, 02 Jul 2026 10:00:00 GMT
```

## 二、HTTPS 与安全

### 1. HTTPS 原理

**流程**:

```
客户端                                服务端
  |------- Client Hello -------------->|  (支持的加密套件)
  |<------ Server Hello ---------------|  (选择的套件)
  |<------ Certificate ---------------|  (服务器证书)
  |<------ Server Key Exchange -------|  (公钥)
  |<------ Server Hello Done ---------|
  |                                    |
  |-- Client Key Exchange ----------->|  (用公钥加密的预主密钥)
  |-- Change Cipher Spec ------------>|
  |-- Finished ----------------------->|
  |                                    |
  |<- Change Cipher Spec -------------|
  |<- Finished ------------------------|
  |                                    |
  |====== 对称加密通信 ================|
```

**混合加密**:
1. **非对称加密**(RSA/ECC):握手阶段,交换密钥
2. **对称加密**(AES):通信阶段,加密数据

**为什么不全用非对称**:非对称加密慢(1000倍+)。

### 2. 证书链

```
根证书(CA,预置在系统)
  └─ 中间证书
      └─ 服务器证书
```

**验证流程**:
1. 浏览器验证证书签名(用上级公钥)
2. 递归到根证书
3. 检查证书有效期/域名/吊销状态

### 3. 中间人攻击防御

**攻击**:拦截流量,伪造证书。

**防御**:
- **证书链验证**:只信任 CA 签发的证书
- **HSTS**:强制 HTTPS

```http
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

- **证书固定**(Certificate Pinning):只信任特定证书

## 三、WebSocket

### 1. WebSocket vs HTTP

| 特性 | HTTP | WebSocket |
|------|------|-----------|
| 连接 | 短连接(HTTP/1.0)/长连接(HTTP/1.1) | 持久连接 |
| 通信 | 半双工(一次一个方向) | 全双工(双向同时) |
| 协议 | 无状态 | 有状态 |
| 开销 | 每次请求带头部 | 首次握手后低开销 |
| 实时性 | 轮询/长轮询(延迟高) | 实时推送 |

### 2. 握手流程

**升级请求**:

```http
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
```

**服务端响应**:

```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

**客户端代码**:

```js
const ws = new WebSocket('wss://example.com/chat')

ws.onopen = () => {
  console.log('connected')
  ws.send('Hello')
}

ws.onmessage = event => {
  console.log('received:', event.data)
}

ws.onerror = error => {
  console.error('error:', error)
}

ws.onclose = () => {
  console.log('disconnected')
}
```

### 3. 心跳保活

```js
let heartbeatTimer

ws.onopen = () => {
  heartbeatTimer = setInterval(() => {
    if (ws.readyState === WebSocket.OPEN) {
      ws.send(JSON.stringify({ type: 'ping' }))
    }
  }, 30000)  // 30秒一次
}

ws.onclose = () => {
  clearInterval(heartbeatTimer)
}
```

### 4. 应用场景

- 实时聊天
- 在线协作(文档/白板)
- 游戏
- 股票行情
- 弹幕

## 四、TCP/IP

### 1. 三次握手

```
客户端                      服务端
  |------- SYN ------------->|  (seq=x)
  |<--- SYN + ACK ------------|  (seq=y, ack=x+1)
  |------- ACK ------------->|  (ack=y+1)
  |                          |
  |===== 连接建立 =============|
```

**为什么三次**:
1. 第一次:服务端知道"客户端能发"
2. 第二次:客户端知道"服务端能收能发"
3. 第三次:服务端知道"客户端能收"

**防止旧连接**:第三次确认客户端仍在线(防止已失效请求)。

### 2. 四次挥手

```
客户端                      服务端
  |------- FIN ------------->|  (主动关闭)
  |<------ ACK --------------|  (收到,但可能还有数据要发)
  |                          |
  |<------ FIN --------------|  (数据发完,准备关闭)
  |------- ACK ------------->|
  |                          |
  |=== TIME_WAIT(2MSL) ======|  (等待2MSL防止ACK丢失)
```

**为什么四次**:TCP 全双工,双向都要独立关闭。

**TIME_WAIT**:2倍 MSL(最大报文生存时间,通常 2 分钟)。

### 3. 流量控制 vs 拥塞控制

**流量控制**(接收方处理能力):
- **滑动窗口**:接收方告知可接收大小

**拥塞控制**(网络承载能力):
- **慢启动**:指数增长(1→2→4→8)
- **拥塞避免**:线性增长
- **快重传**:收到3个重复 ACK 立即重传
- **快恢复**:跳过慢启动,从拥塞窗口一半开始

## 五、DNS

### 1. 查询流程

```
浏览器缓存
  ↓ 未命中
系统缓存(hosts / DNS cache)
  ↓
本地 DNS 服务器(ISP)
  ↓ 递归查询
根域名服务器(.)
  ↓
顶级域名服务器(.com)
  ↓
权威域名服务器(example.com)
  ↓
返回 IP 地址
```

**递归 vs 迭代**:
- **递归**:客户端 → 本地 DNS,后者递归查询
- **迭代**:本地 DNS 逐级查询(根 → TLD → 权威)

### 2. DNS 优化

**DNS 预解析**:

```html
<link rel="dns-prefetch" href="https://api.example.com">
```

**DNS-over-HTTPS(DoH)**:加密 DNS 查询。

**CDN + GeoDNS**:根据地理位置返回最近节点。

## 六、跨域与安全

### 1. 同源策略

**同源**:协议 + 域名 + 端口 完全相同。

```
https://example.com:443/page1
https://example.com:443/page2  ✅ 同源
https://api.example.com        ❌ 域名不同
http://example.com             ❌ 协议不同
https://example.com:8080       ❌ 端口不同
```

**限制**:
- AJAX 请求不能跨域
- DOM 不能跨域访问
- Cookie/LocalStorage 不能跨域读取

### 2. CORS(跨域资源共享)

**简单请求**:

```http
// 请求
Origin: https://example.com

// 响应
Access-Control-Allow-Origin: https://example.com
Access-Control-Allow-Credentials: true
```

**预检请求**(Preflight):

```http
// OPTIONS 预检
Origin: https://example.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Content-Type

// 响应
Access-Control-Allow-Origin: https://example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type
Access-Control-Max-Age: 86400  // 预检缓存24小时
```

**触发预检条件**:
- 非 GET/HEAD/POST
- Content-Type 非 `application/x-www-form-urlencoded/multipart/form-data/text/plain`
- 自定义请求头

### 3. JSONP(已过时)

```html
<script src="https://api.example.com/data?callback=handleData"></script>

<script>
function handleData(data) {
  console.log(data)
}
</script>
```

**缺点**:只支持 GET,安全性差。

### 4. 其他跨域方案

- **代理**:同域服务器转发
- **WebSocket**:不受同源策略限制
- **postMessage**:跨窗口通信

```js
// 父窗口
iframe.contentWindow.postMessage('data', 'https://other.com')

// 子窗口
window.addEventListener('message', event => {
  if (event.origin === 'https://trusted.com') {
    console.log(event.data)
  }
})
```

## 七、高频面试题

### Q1: HTTP 和 HTTPS 的区别?

| 特性 | HTTP | HTTPS |
|------|------|-------|
| 端口 | 80 | 443 |
| 安全性 | 明文传输 | 加密传输 |
| 性能 | 快 | 稍慢(TLS 握手) |
| SEO | 差 | 优 |
| 成本 | 低 | 需购买证书 |

### Q2: GET 和 POST 的区别?

| 维度 | GET | POST |
|------|-----|------|
| 语义 | 获取资源 | 提交数据 |
| 幂等 | ✅ | ❌ |
| 缓存 | 默认缓存 | 不缓存 |
| 参数 | URL(有长度限制) | Body(无限制) |
| 安全性 | 参数暴露在 URL | 相对安全 |
| 书签 | 可收藏 | 不可 |

### Q3: 如何优化网络性能?

**减少请求数**:
- 合并资源(雪碧图/打包)
- 懒加载
- 缓存

**减少传输量**:
- 压缩(Gzip/Brotli)
- 图片优化(WebP/压缩)
- Tree-shaking

**提升速度**:
- CDN
- HTTP/2
- DNS 预解析
- Keep-Alive

**优化关键路径**:
- 内联关键 CSS
- defer/async 脚本
- 预加载关键资源

### Q4: Cookie vs Session?

**Cookie**(客户端):
- 存浏览器,4KB 限制
- 每次请求自动携带
- 可设置过期时间

**Session**(服务端):
- 存服务器,大小不限
- 通过 Session ID(Cookie)关联
- 服务器重启丢失(除非持久化)

**JWT**(无状态):
- 不存服务端
- 自包含用户信息
- 无法撤销(除非黑名单)

### Q5: TCP 为什么可靠?

1. **序号机制**:数据包编号,保证顺序
2. **确认应答**:收到发 ACK
3. **超时重传**:未收到 ACK 重发
4. **流量控制**:滑动窗口
5. **拥塞控制**:慢启动/拥塞避免
6. **校验和**:检测数据损坏
