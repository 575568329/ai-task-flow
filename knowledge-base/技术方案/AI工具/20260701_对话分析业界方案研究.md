## 📊 业界实际方案

### 1. Cursor / Claude Code 的技术选择

**核心发现**：这些顶级 AI 编程工具**不使用 RAG 向量搜索**，而是：

#### 他们用什么？
- ✅ **Grep/Ripgrep** - 精确字符串匹配
- ✅ **AST 导航** - 结构化代码分析
- ✅ **直接文件读取** - 选择性加载上下文
- ✅ **大上下文窗口** - 200K+ tokens 直接加载

#### 为什么不用 RAG？
> "Vector embeddings don't capture code structure reliably"
> 
> 来源：[Is RAG Dead? What AI Agents Use Instead](https://www.mindstudio.ai/blog/is-rag-dead-what-ai-agents-use-instead)

**关键理由**：
1. **代码是结构化的**，不是语义文本
2. **向量搜索的灾难性失败** - 找不到定义时会编造不存在的方法
3. **Grep 更精确** - 为精确查找而设计
4. **上下文完整性** - 函数需要完整的签名、类型、导入

**对对话分析的启示**：
- 对话记录是半结构化的（有角色、时间戳、工具调用）
- 精确匹配（如查找某个问题）用 Grep
- 理解和总结用 LLM 直接读取

### 2. LLM 对话总结的生产实践

来源：[Galileo AI - LLM Summarization Production Guide](https://galileo.ai/blog/llm-summarization-production-guide)

#### 核心挑战
> "Each processed token risks your credibility"

**常见失败**：
- ❌ 关键信号在压缩中丢失（如"退款请求"被忽略）
- ❌ 多主题对话中的截止日期遗漏
- ❌ 过度概括导致误导性结论

#### 推荐方案

##### A. 提示工程策略
```
1. 明确定义范围和输出格式（JSON Schema）
2. 链式思考（Chain of Thought）- 让模型追踪上下文
3. 混合方法：先提取关键句子 → 再总结
```

**示例提示**：
```
任务：分析这个项目开发对话，提取：
1. 用户提出的问题（按优先级）
2. 技术决策及理由
3. 遇到的错误和解决方案
4. 可复用的经验教训

输出格式：JSON
{
  "problems": [{...}],
  "decisions": [{...}],
  "solutions": [{...}],
  "lessons": [{...}]
}
```

##### B. 长对话处理模式

**Map-Reduce 模式**：
```
1. 将对话分块（带 10% 重叠）
2. 并行总结每个块（Map）
3. 递归合并总结（Reduce）
4. 生成最终综合报告
```

**滑动窗口模式**：
```
1. 维护持久记忆核心
2. 仅重新总结最近交流
3. 增量更新到记忆系统
```

**两阶段模式**：
```
阶段 1：关键词/实体提取
阶段 2：基于关键点的抽象总结
```

##### C. 防止幻觉

**Decompose-then-verify 流程**：
```
1. 将总结拆分为原子级声明
2. 每个声明与原文验证
3. NLI 模型检查事实一致性
4. 计算总体可信度分数
```

### 3. LLM 总结的三大策略

来源：[Master LLM Summarization Strategies](https://galileo.ai/blog/llm-summarization-strategies)

#### 策略对比

| 策略 | 适用场景 | 幻觉风险 | 适合对话分析吗？ |
|------|----------|----------|----------------|
| **抽取式** | 精确引用（法律/医疗） | ⭐低 | ⭐⭐⭐ 适合提取关键对话 |
| **生成式** | 可读性报告 | ⭐⭐⭐高 | ⭐⭐⭐⭐ 适合经验总结 |
| **混合式** | 多文档综合 | ⭐⭐中 | ⭐⭐⭐⭐⭐ 最适合！ |

**推荐：混合式**
```
步骤 1：抽取关键对话片段（问题、错误、决策点）
步骤 2：生成式总结每个主题
步骤 3：结构化输出到记忆系统
```

#### RAG 集成建议

> "Retrieval grounds summaries in source material"

**对对话分析的应用**：
1. 先语义检索相关对话段
2. 将检索结果作为上下文
3. LLM 基于实际内容总结
4. 减少编造，提高准确性

## 🚀 针对 Claude Code 对话的最佳实践

### 方案 A：直接 LLM 总结（推荐）

**适用**：单个会话 < 200K tokens

```
步骤：
1. 提供 JSONL 文件路径
2. 明确指定提取目标（问题/决策/经验）
3. 要求结构化输出（JSON/Markdown）
4. 验证关键事实
```

**提示模板**：
```markdown
请分析这个项目开发对话：
`C:\Users\xxx\.claude\projects\MyProject\session.jsonl`

提取以下内容：
1. **技术决策**（选型理由、架构设计）
2. **问题解决**（遇到的错误、解决方案、为什么有效）
3. **工具使用**（哪些工具最频繁、为什么）
4. **经验教训**（下次可以改进的地方）

输出为结构化 Markdown，每个主题下列出具体案例。
```

### 方案 B：Map-Reduce 分块总结

**适用**：多个大型会话、跨项目分析

```python
# 伪代码
sessions = glob("~/.claude/projects/MyProject/*.jsonl")

# Map 阶段
chunk_summaries = []
for session in sessions:
    summary = claude_summarize(session, focus="problems+solutions")
    chunk_summaries.append(summary)

# Reduce 阶段
final_report = claude_synthesize(chunk_summaries)
write_to_memory(final_report)
```
