# Token 用量面板:数据从哪来、怎么算、怎么存

> 本文只讲清一件事:**怎么统计 Claude Code 在终端干活的 token 用量**——数据从哪来、长什么样、怎么提取、怎么存、怎么算成本。
>
> 团队成员读完能讲清原理。各家 agent 对照、ccusage 做法、计费模式等通用谈资见文末「扩展」附录。

---

## 一、先纠正一个直觉误区

「统计用量 = 在后端加埋点?」——**对项目内调研 agent 对,对 Claude Code 错。**

| | 项目内调研 agent | Claude Code 终端 |
|---|---|---|
| 谁调 LLM | 我们后端 | Claude Code 自己直连 |
| 我们抓得到吗 | 抓得到(流经后端) | **抓不到**(不经过我们) |
| 数据从哪来 | 流式响应埋点 | **扫它的本地会话日志** |

关键洞察:Claude Code 每次对话都把完整记录写进本地日志(为了断点续传),**而日志里带每次调用的 token 用量**。这是唯一不依赖 Claude Code 配合的观测点——扫日志即可,**零侵入、全本地、零配置**。

---

## 二、日志长什么样(真实样例,本机实测)

### 位置
```
~/.claude/projects/<编码后的项目路径>/<会话ID>.jsonl
```
项目路径编码:把 `/` 换成 `-`。本项目 `/mnt/d/Study/ai-task-flow` → `-mnt-d-Study-ai-task-flow`。
**一个 `.jsonl` 文件 = 一场对话**;一个目录 = 一个项目的全部对话。

### 格式:JSONL(每行一条 JSON 消息)
```
{"type":"system",    ...}   ← 会话开场元信息
{"type":"user",      ...}   ← 你问的
{"type":"assistant", ...}   ← 模型答的(用量在这!)
{"type":"user",      ...}
{"type":"assistant", ...}
```

### 三类行的真实长相

**system 行**——会话元信息(含 cwd / gitBranch,可用于反推项目):
```json
{"type":"system","subtype":"local_command","cwd":"/mnt/d/Study/ai-task-flow",
 "sessionId":"cce64d39-...","gitBranch":"feat/vocab-translate","timestamp":"..."}
```

**user 行**——用户输入:
```json
{"type":"user","message":{"role":"user","content":"拉一下任务"},
 "cwd":"...","sessionId":"...","timestamp":"..."}
```

**assistant 行**——模型回答 + 用量(**我们要的数据全在这**):
```json
{"type":"assistant","message":{
   "model":"glm-5.2",
   "usage":{
     "input_tokens": 24753,              ← 本次读入(问题+历史上下文+系统提示)
     "output_tokens": 106,               ← 本次生成(模型回答)
     "cache_read_input_tokens": 8512,    ← 命中缓存读了多少(input 的子集)
     "cache_creation_input_tokens": 0,   ← 新建缓存总量
     "cache_creation": {                 ← 缓存细分(计费用)
        "ephemeral_5m_input_tokens": 0,  ← 5 分钟缓存 → 1.25×
        "ephemeral_1h_input_tokens": 0   ← 1 小时缓存 → 2×
     },
     "server_tool_use": {"web_search_requests":0,"web_fetch_requests":0}
   },
   "content":[...]}}
```

## 三、怎么提取(代码)

```ts
// 扫一个会话文件,累加所有 assistant 行的 usage
interface SessionUsage {
  input: number; output: number; cacheRead: number; cacheCreation: number; model: string;
}

function scanSession(lines: string[]): SessionUsage {
  const agg: SessionUsage = { input:0, output:0, cacheRead:0, cacheCreation:0, model:'' };
  for (const line of lines) {
    let m: any;
    try { m = JSON.parse(line); } catch { continue; }     // 跳过坏行
    if (m.type !== 'assistant') continue;                 // 只认 assistant 行
    const u = m.message?.usage;
    if (!u) continue;
    agg.input         += u.input_tokens ?? 0;
    agg.output        += u.output_tokens ?? 0;
    agg.cacheRead     += u.cache_read_input_tokens ?? 0;
    agg.cacheCreation += u.cache_creation_input_tokens ?? 0;
    agg.model       ||= m.message.model;                  // 取首个模型名
  }
  return agg;
}
```

**一句话:遍历每行 → `JSON.parse` → 只取 `type==="assistant"` 的 → 累加 `message.usage` 四个字段。**

> 任务关联(`taskId`)从**全会 user 行**(含 `get_task` 的 `tool_result`)提取标记,缓存 5m/1h 细分(`ephemeral_*`)用于成本计算——分别见第五、六节。

## 四、怎么存 + 性能策略

> **核心原则:请求只读缓存,扫描按需低频。绝不在请求路径里全量遍历 jsonl。**

### 关键性质
jsonl **只追加,历史行永不变** → 一个会话扫一次建立基线,之后只管新增行;旧行永不重扫。

### 请求路径三层过滤(把"扫"踢出热路径)

| 层 | 做什么 | 效果 |
|---|---|---|
| **L1 staleness 窗口** | 距上次扫描 < 60s,直接返缓存,连 mtime 都不查 | 高频刷新面板也快 |
| **L2 mtime diff** | 只对源 jsonl 的 mtime 变了的会话重扫(只 `stat`,不读内容) | 没变的会话零成本 |
| **L3 增量行** | 变了的会话,从 `scannedLines` 之后只读新增行 | 不重扫旧行 |

请求路径 = 读缓存 +(可能)几个 mtime stat +(可能)增量几行。**永远不全量遍历。**

### 遍历本身也省:行首粗筛
别每行都 `JSON.parse` 再判 type,先粗筛跳过 user/system 行(占大头):
```ts
for (const line of lines) {
  if (!line.includes('"type":"assistant"')) continue;   // 粗筛,不 parse
  const m = JSON.parse(line);                            // 只 parse assistant 行
  // ...累加 message.usage
}
```

### 缓存文件格式(一行 = 一个会话累计)
`~/.ai-task-flow/usage.jsonl`:
```json
{"sessionId":"cce64d39-...","repoPath":"D:\\Study\\ai-task-flow","source":"wsl","taskId":"TASK-006",
 "model":"glm-5.2","input":24753,"output":106,"cacheRead":8512,"scannedLines":437,
 "firstSeen":"2026-07-24T10:00:00Z","lastSeen":"2026-07-24T18:00:00Z"}
```
> `source`(会话来源环境)与 `taskId`(任务关联)见第五节。

### 写入:原子 rename,无锁
- scanner 单进程跑(HTTP 请求或后台预热),无跨进程竞争;
- 写法:**先写 `usage.json.tmp` → `rename` 成 `usage.json`**(同文件系统 rename 是原子的)→ 读永远拿到完整文件,写永不撕裂。**零锁代码**;
- 真要跨进程(将来 MCP 也写)再上 `proper-lockfile`,现在不需要。

### 按天拆:行级 timestamp
每行都有 `timestamp`。按天聚合时,把每个 assistant 行的 usage 按**它自己的 timestamp** 归天——不是按会话整体归一天(一个会话可能跨多天)。

### sqlite 什么时候换
文件到**万级会话**、或要 `GROUP BY 模型×日期×项目×环境` 多维交叉查询时再换。现在 JSONL 几百行,全读进内存 reduce 即可,别提前上。

### ⚠️ WSL 跨环境 IO(重点,性能成败关键)

部署形态常是:**后端跑 Windows,Claude Code 跑 WSL**。此时 Claude jsonl 在 **WSL 原生文件系统**(`/home/<user>/.claude/`),Windows 后端读它要走 `\\wsl.localhost\<distro>\` 或 `/mnt/...` 映射——这是 **9P 协议跨文件系统 IO,比原生慢一两个数量级**(每文件打开 10~50ms 量级)。

应对(把慢 IO 挡在请求路径外):

| 做法 | 作用 |
|---|---|
| 慢 IO 只在"扫描"时发生 | 扫描被 L1/L2 挡住 → 多数请求 staleness 命中,**只读 Windows 本地缓存,不碰 WSL** |
| 必须扫时只 `stat` mtime | 比 read 内容轻得多 |
| 冷启动首次全扫 | 放**后台预热**,不阻塞首个请求(可接受首次稍慢) |

> **待实测声明**:跨 `\\wsl$` 的 stat/read 实际耗时未在本机实测,按 9P 常见表现推断。落地后在 Windows 侧实测面板刷新耗时;若仍卡,加大 staleness 窗口(如 5min)或上 `fs.watch`/`chokidar` 事件驱动(文件变更才扫,彻底不轮询)。

### 量级估算

| 场景 | 耗时 |
|---|---|
| 请求读缓存(7 会话) | < 1ms |
| 请求 + stat 7 个 mtime(WSL 跨界) | ~几十 ms |
| 请求 + 增量扫 1 个变化的会话(几十行) | ~5–10ms |
| 全量重扫所有会话(最坏,冷启动) | 几百 ms~1s(只一次) |

**结论**:个人用量级(几项目 × 几会话),这套够,WSL 也不卡——慢 IO 被缓存层挡在请求路径外。

## 五、怎么聚合给前端看

主维度(面板直接展示):

| 维度 | 怎么算 |
|---|---|
| **按项目** | 目录名 = 编码后的 repoPath,和看板任务 projectName 天然对应 |
| **按任务** ⭐ | 会话 user 行里的 `<!-- ai-task-flow: task=TASK-xxx -->` 标记(`get_task` 返回时埋,见下) |
| **按模型** | `message.model` |
| **按天** | 行级 timestamp |
| **按会话** | 一个 `sessionId.jsonl` = 一场对话 |

### 按任务关联(核心价值)⭐

「TASK-006 花了多少 token」比「这项目花了多少」更 actionable。

> **现状前提**:派发/worktree 机制已移除,用户自己开 Claude Code、调 MCP `get_task` 拉任务。**任务关联只能靠标记**(无 worktree 路径可兜底)。

**埋标记**:`get_task` 工具返回的任务 markdown 顶部注入 HTML 注释(对 Claude 可读、不渲染、不干扰):
`<!-- ai-task-flow: task=TASK-006 -->`

**可靠性分析**(标记不会被冲掉,但有查不到的场景):

- ✅ **不会被冲掉**:jsonl 只追加、历史行不可变。标记一旦随 `get_task` 的 `tool_result` 写进某 user 行,就永远在那行,Claude 后续对话改不了它。
- ⚠️ **位置不定**:任务 markdown 经 `get_task` 注入,jsonl 里以 `tool_result` 装在 **user 行**(不一定首条)。→ 扫描器遍历**全会所有 user 行**找标记。
- ⚠️ **一会话多任务(现在常见)**:没 worktree 隔离,用户常在一个终端会话里连拉多个任务 → 出多个标记。**必须按「两标记之间的 assistant 消息」切分 token 归属**(不是可选);切分前/后无标记的段落归「未关联」。
- ⚠️ **查不到的归未关联**:没调 `get_task` 的对话(纯聊天/手起 Claude)无标记 → 归「未关联」,不硬塞。
- **正则**:`/ai-task-flow:\s*task=([A-Z0-9-]+)/i`,容错空格/大小写。

> 即便切分是粗粒度估算,「按任务」仍比「按项目」actionable;查不到兜底为「未关联」,不误导。

### 环境标签(内部字段,可选筛选,不做主维度)

会话来源环境作为 `source` 字段记录,不做成顶部主视图:

- `/home/<user>/.claude/projects/...` → `source: 'wsl'`;`C:\Users\<user>\.claude\projects\...` → `source: 'windows'`。
- `ClaudeSessionScanner` 枚举所有 home 时顺手打标(零额外成本)。
- MVP 呈现:会话列表每行挂 WSL/Win 小标签供筛选;**不做**顶部「按环境分布」主视图。
- 语义:按「会话生于哪个环境的 Claude Code」,非后端跑在哪;sessionId 全局唯一,**不会跨环境重复计**。

## 六、成本怎么算

### 单价表:不用自己查
开源项目 [LiteLLM](https://github.com/BerriAI/litellm) 维护一份 `model_prices_and_context_window.json`,**持续更新、免费、覆盖 100+ 供应商**。ccusage 内嵌的就是它。我们构建时锁快照、定期同步即可;未命中的模型按名覆盖。

### 缓存倍率(Anthropic 官方,对应上面真实字段)
| 字段 | 倍率(相对 input 单价) |
|---|---|
| `cache_creation.ephemeral_5m_input_tokens` | **1.25×** |
| `cache_creation.ephemeral_1h_input_tokens` | **2×** |
| `cache_read_input_tokens` | **0.1×**(命中缓存省 90%) |

### 公式
```
cost = input/1e6 × P_in
     + output/1e6 × P_out
     + cache5m/1e6 × P_in × 1.25
     + cache1h/1e6 × P_in × 2
     + cacheRead/1e6 × P_in × 0.1
```

> **字段互斥(重要)**:`input_tokens` / `cache_creation_input_tokens` / `cache_read_input_tokens` **三者独立、不重叠**——别把 cache 当 input 子集重复计。`cache_creation_input_tokens` 是总数,按 `ephemeral_5m`(1.25×)/ `ephemeral_1h`(2×)细分计费。
>
> **⚠️ 实现提醒**:现有 `modelPricing.ts` 的 `estimateCost` 只算了 `input + output + cacheRead`,**漏了 `cache_creation` 的 1.25×/2× 溢价**(且入参 `Pick` 没带 `cacheCreationTokens`)。落地时必须补:入参加 `cache5m/cache1h`,公式加两项,否则缓存首次创建那轮成本被低估。

> GLM 等订阅制套餐本身不计费,单价为占位值,成本视图标注「估算」,实际以账单为准。

## 七、术语速查(讲给别人时用)

| 术语 | 含义 |
|---|---|
| **per 1M tokens** | 每一百万个 token 多少美元。`$3/1M` = 读 100 万 token 收 3 美元 |
| **token** | LLM 计费单位,**≠ 字符**。英文 ≈ 0.75 词 / 4 字符,中文 1 字 ≈ 1~2 token。**所以不能拿文件大小/字符数算,必须用返回的 token 数** |
| **input / output** | input = 喂给模型的全部(问题+历史+系统提示);output = 模型生成的回答。`cache_read` 是 input 的**子集** |
| **provider / model** | provider = 哪家(anthropic/openai/智谱);model = 具体型号(sonnet/glm-5.2/gpt-4o)。**定价按 model 查**(model 名能推出 provider) |
| **cache_read** | 命中缓存的输入 token,计费约为正常 input 的 10% |
| **如何知道是缓存** | 不靠猜,`usage.cache_read_input_tokens > 0` 即本轮有命中。Anthropic 算好填进响应,Claude Code 写进 jsonl,我们直接读 |

## 九、风险

- **WSL 无法实测**:dev 依赖 native binary(Windows 装),交付靠 `tsc` + build→node 单测,**端到端你在 Windows 侧验**(聊几轮 → 看「用量」页 token 增长)。
- **成本是估算**:单价表可能滞后;未命中模型 cost 留空,标注「估算」。
- **按任务关联**:靠 `get_task` 返回 markdown 埋的 `<!-- ai-task-flow: task= -->` 标记,匹配不到归「未关联」;没调 `get_task` 的对话不计入任务,不硬塞。一会话多任务靠标记间切分,**切分是粗粒度估算**(两标记之间的 assistant 消息归前者)。

## 十、为何自建扫描器而非集成 ccusage

审阅时被问到的核心问题:业界已有成熟的 [ccusage](https://github.com/ccusage/ccusage)(支持多 agent、内置 LiteLLM 定价、`--json` 可调用),为何还自建?

| | 自建(扩 ClaudeSessionScanner) | 集成 ccusage 子进程 |
|---|---|---|
| 工作量 | **小**:Scanner 步骤③已写好,加几十行 usage 累加 + taskId 提取 | 中:spawn + 缓存 + 混合环境处理 |
| 混合环境 | `resolveDataDir` 已处理 home 探测(Windows + WSL) | **坑**:ccusage 跑 Windows 读不到 WSL jsonl;跑 WSL 要跨进程调 |
| 准确性 | 自管定价/倍率 | 内置 LiteLLM,更准且持续更新 |
| 实时性 | 按需扫,可控 | `npx` spawn 几秒,只能定时跑 |
| 依赖 | 零外部 | npx + 网络 + 版本漂移 |
| 任务关联 | **可做**(埋标记提取 taskId) | 不支持按任务 |

**结论:自建**。Scanner 基础已有、混合环境 ccusage 反而更折腾、任务关联是 ccusage 给不了的差异化。ccusage 当**参照标杆**(抄它的 LiteLLM 定价快照 + 倍率算法 + 计费模式),不当运行时依赖。

## 十一、扩展:其他 agent 工具与通用谈资(面试可用)

> 本项目只扫 Claude Code,但原理通用。下面是统计**任意 agent 工具**用量的知识与谈资,面试可直接讲。

### 各家 agent 日志格式对照

| agent | 日志路径 | usage 字段 | 坑 |
|---|---|---|---|
| **Claude Code** | `~/.claude/projects/<编码路径>/<sid>.jsonl` | `message.usage.{input,output,cache_read,cache_creation}_tokens` | 最规范,本方案基准 |
| **OpenAI Codex** | `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl` | `payload.type==="token_count"` | **值是累计,需相邻相减**,否则重复计 |
| **Goose** | `~/.local/share/goose/sessions/sessions.db` | SQLite token 字段 | `total_tokens` 有恒 0 bug |

### 业界标杆 ccusage 的工程做法(可吹)
1. **适配器模式**:monorepo,每个 agent 一个解析包,异构日志统一成内部 token 结构——加 agent = 加包,核心不动。
2. **单次扫描多报告**:一遍 IO 同时出 daily / session / monthly / blocks。
3. **LiteLLM 定价快照 + 离线**:构建期锁,内网可用。
4. **本地优先零隐私**:全程不联网上报。

### 计费三模式(auto / calculate / display)
- `auto`:优先用预计算 `costUSD`,无则从 token 算;
- `calculate`:强制从 token 算(跨时间一致,推荐);
- `display`:只显示预计算值。
- **订阅制(Pro/Max)**:cost 是「占配额的等价价值」(机会成本估算),非真实账单。

### 谈资 / 踩坑
- Claude Code `costUSD` 据社区反馈 v1.0.9 后恒为 0 → 必须从 token 重算(用 `calculate` 模式)。
- Codex `token_count` 累计不减会重复计。
- 格式无官方文档 → agent 升级可能破坏解析器(这正是适配器隔离的价值)。

### 附:另一条路径——自己调 LLM 时取 usage(项目内 agent / 直接集成 SDK)

不走日志旁路,而是从 LLM 响应里直接读 usage(适用于 token 流经自己后端的场景,如项目内调研 agent):

- **Anthropic 流式**:`message_start.usage` 为 **null**(别取),取 **`message_delta` 事件的 `usage`**——流结束前最后一次 `message_delta` 即最终累积值。
- **OpenAI 流式**:默认**不返回** usage,必须显式 `stream_options: { include_usage: true }`,usage 在最后一个额外 chunk(`prompt_tokens` / `completion_tokens` / `total_tokens`)。注意很多"OpenAI 兼容"国产网关对流式 `include_usage` 支持参差,接入需实测。
- **非流式**:两家响应体 `usage` 字段直接读,无坑。
- **业界**:LangFuse / Helicone 走请求级记录 + 内置单价表;OpenTelemetry GenAI Semantic Conventions 标准化 LLM 遥测字段,值得对齐命名。
