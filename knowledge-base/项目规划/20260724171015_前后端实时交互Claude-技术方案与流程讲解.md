# 前后端怎么实时交互 Claude:技术方案与流程讲解

> 这是一份**通用技术讲解**,不涉及具体怎么改代码,只讲清一件事:**在网页上和 Claude 实时对话,技术上是怎么做到的**——用什么技术栈、数据怎么流动、关键难点怎么解。
>
> 团队成员读完能讲清原理。**看懂之后,再决定要不要实施、怎么实施。**

---

## 一、目标:用大白话说

现在我们派发任务后,得手动开终端、启动 Claude Code、用 MCP 拉任务,一次盯一个,看不见实时进度,想插话只能回终端敲。

我们想要的是:

- **看板上每个任务点开一个对话框,直接在里面和 Claude 聊**
- Claude 干活的过程(在想什么、在写什么代码、在调什么工具)**实时看得见**
- 我们能**随时插话**,它接着干
- **不用再开终端**

---

## 二、先搞懂几个基本概念

往下看之前,这几个概念必须先建立,否则流程看不懂。

### 1. Claude Code 是个命令行工具(CLI)

它不是网页,是装在你电脑上的一个**命令行程序**,平时在终端里跑。我们在终端里和它聊天,其实就是跟这个程序打交道。

### 2. "headless 无界面"模式

Claude Code 可以用 `claude -p "你的问题"` 这种方式启动——它**不开界面、不等人在键盘上输入**,直接把问题丢给模型、把答案打到标准输出(stdout)、答完就退出。这叫 **headless 模式**。

> **关键**:程序化地调用 Claude(让后端去驱动它),靠的就是这个模式。这是整个方案的地基。

### 3. 流式输出 stream-json

headless 模式加一个参数 `--output-format stream-json`,Claude 就不会"全部想完再一次性吐出来",而是**边想边输出,一行一个 JSON 事件**:

| Claude 在干嘛 | 吐出的事件 |
|---|---|
| 在思考 | `thinking` |
| 在写文字回答 | `text` |
| 要调用工具(读文件/改代码…) | `tool_use` |
| 工具执行完了 | `tool_result` |

这种"边跑边吐"的事件流,就是**我们要在前端实时显示的东西**。不是黑盒等结果,而是看得见全过程。

### 4. 会话 sessionId

Claude Code 每开一次对话,都有一个唯一 ID(sessionId)。下次启动时用 `--resume <sessionId>`,**能接着上次的上下文继续聊**,不用从头讲一遍。

> **关键**:这是我们实现"多轮对话"的钥匙——每轮对话结束存下 sessionId,下一轮带着它续接。

### 5. 为什么浏览器不能直接连 Claude

浏览器里的网页(JavaScript)出于安全限制,**不能启动你电脑上的程序**。Claude Code 是本机程序,浏览器根本碰不到它。

> **关键**:所以中间**必须有一个"后端"当桥**——后端能启动本机程序,前端只负责和后端通信。这就是为什么必须有后端,不能前端直连 Claude。

## 三、整体架构:前端 ↔ 后端 ↔ Claude

```
   [浏览器:对话窗]  ←── HTTP ──→  [后端:Node 服务]  ── spawn ──→  [Claude Code 进程]
         ↑                              ↑                              │
         │                              │                              ▼
     显示流式输出                 启动 / 转发 / 存会话           在 worktree 里干活
```

**后端是中间人**,干三件事:

1. **替前端启动 Claude**(前端说"开聊",后端去启动进程)
2. **把 Claude 的输出转给前端**(Claude 边吐事件,后端边转发给前端)
3. **把前端的话转给 Claude**(前端发新消息,后端喂给 Claude)

> 类比:前端是顾客,后端是传菜员,Claude 是后厨。顾客不能进后厨,只能通过传菜员点单和上菜。

---

## 四、技术栈是什么

| 层 | 用什么 | 为什么 |
|---|---|---|
| **前端** | React + `fetch` 流式读取 + `react-markdown` | 发消息用 fetch,收流用浏览器原生 ReadableStream,渲染 markdown 用现成库 |
| **传输** | HTTP POST + 流式响应(SSE 风格) | **不用 WebSocket**,单向流够用、最简单 |
| **后端** | Node.js + Fastify(已有)+ `child_process.spawn` | 用 Node 自带的 spawn 启动 Claude,不引新依赖 |
| **Claude** | Claude Code CLI 的 headless + stream-json | `-p` 无界面 + `--output-format stream-json` 流式 |
| **隔离** | git worktree(已有机制) | 每个任务一个独立工作目录,Claude 在里面干活,互不干扰 |
| **模型** | 国内模型(用户选定) | 通过"适配器"接入,换模型不改架构 |

> **重点**:除了 Claude CLI 本身,**其他全是现成技术**,不用引入重型依赖。前端要的 markdown 渲染、动画库,项目里已经有了。

## 五、一次对话的完整流程(逐步时序)

### 第一条消息

1. **用户**在对话窗输入"实现登录功能",点发送
2. **前端**发请求 `POST /api/tasks/任务id/chat`,带上这条消息
3. **后端**收到,检查这个任务之前聊过没(有没有存 sessionId)
   - 没有 → 开个新会话
   - 有 → 用 `--resume` 续接
4. **后端启动 Claude**:`claude -p "实现登录功能" --output-format stream-json`,工作目录设成这个任务的 worktree
5. **Claude 开始干活,边干边吐事件**:`thinking → text → tool_use → tool_result → ...`
6. **后端边读边转发**:每读到一行事件,立刻通过 HTTP 响应推给前端(响应一直不关,持续流)
7. **前端边收边渲染**:
   - 收到 `thinking` → 显示"思考中…"
   - 收到 `text` → 用 markdown 渲染出来
   - 收到 `tool_use` → 显示"正在调用工具:读文件 xxx"
8. **Claude 跑完**,最后吐一条带 sessionId 的结束事件
9. **后端**把这个 sessionId 存起来(下次续接用),关闭响应
10. **前端**显示完成

### 第二条消息(续接,等于连续聊天)

同样发一个 POST,后端这次带上 `--resume <上次的 sessionId>`,Claude 就**接着上次记着的东西继续**。对用户来说,这就是连续聊天,中间没断。

## 六、几个关键决策(为什么这么做,而不是那样)

### 决策 1:传输为什么用 HTTP 流式,不用 WebSocket?

- 我们只需要**单向流**:发消息是偶尔的低频动作(POST 完全够用),只有"Claude → 前端"是连续流。
- WebSocket 是双向长连接,能力过剩,还得装库、管心跳、管断线重连。
- HTTP 流式(响应头设成 `text/event-stream`)就是为"服务端单向推流"设计的,**后端已经有 SSE 基建**,够用、最简。

> 一句话:杀鸡不用牛刀。

### 决策 2:Claude 进程是常驻,还是每次新起?

- **选:每次新起(按轮 spawn)**。发一条消息起一个 Claude,跑完退出;下条消息用 `--resume` 续接。
- 优点:**不用管进程的生死、崩溃、超时**,简单稳定,崩了重发就行。
- 代价:每轮重新启动有点开销(但 Claude 有缓存,开销很小)。
- 另一种做法(常驻进程 + 持续注入消息)延迟更低,但要维护一个进程池、管超时崩溃——复杂度高,**先不碰**。

### 决策 3:多个任务同时聊怎么办?

- 每个任务各自的对话窗,**各自起各自的 Claude 进程,天然并行**,不需要调度器。
- 想同时跑 3 个任务?开 3 个对话窗、各发一条消息就行。
- 这就是为什么"自动执行/调度"可以排到最后——**用户开窗发消息,本身就是执行入口**。

### 决策 4:换国内模型怎么办?

- 把"启动 Claude"这一步**抽象成一个统一接口**(暂叫 AgentRunner),Claude 只是其中一个实现。
- 将来换 GLM / Qwen / DeepSeek 等,**只要加一个新实现**,前端、后端路由、整个流程**都不用改**。
- 这是借鉴 Multica 的适配器模式(它支持十几个 CLI,每个一个适配器文件)。

### 决策 5:Claude 在哪个目录干活?

- 每个任务有一个独立的 **git worktree**(项目已有这个机制)。
- 启动 Claude 时把工作目录指向它——**任务之间互不干扰,代码改动干净,干砸了可以把整个 worktree 丢掉**。

### 决策 6:"实时流"到底流的是什么?

- 就是第二节讲的 stream-json 事件:思考、文字、工具调用、工具结果。
- 前端按事件类型分别渲染:
  - 文字 → markdown 渲染(代码高亮、列表等都支持)
  - 工具调用 → 折叠成卡片("调用读文件 / 改代码")
  - 思考 → 折叠成"正在思考…"
- 不是干等一个最终结果,而是**看得见 Claude 的整个工作过程**。

## 七、和现在的系统是什么关系

这块很重要,澄清"不是推倒重来"。

| 模块 | 会变吗 | 说明 |
|---|---|---|
| **看板 / 任务 / 四态状态机** | ❌ 不变 | 全部保留 |
| **git worktree 隔离** | ❌ 不变 | 全部保留,Claude 在里面干活 |
| **MCP 工具层** | 🔧 角色变,接口不变 | 以前是"你终端里的 Claude 用 MCP 拉任务";以后是"后端启动的 Claude 用 MCP 回写状态"。**接口不变,以前的投入不浪费** |
| **新增** | ➕ 一条通道 | 前端 → 后端 → Claude 的实时对话通道 |

> **本质**:在现有系统上**加一条"前端直接对话"的通道**,不是重写。MCP、看板、任务模型、worktree 全部复用。

## 八、哪些先不做(范围划死)

- ❌ **自动执行 / 后台调度**:用户在对话窗发消息就是执行入口,不需要后台自动认领任务。
- ❌ **工具调用的批准 / 拒绝 UI**:先做到"能看见"工具调用,交互式审批后面再做。
- ❌ **计费 / 模型选型**:用国内模型,接口预留好,具体用哪个后面定。

## 九、需要先验证的几个点(实施前要确认)

1. **`--resume` 的 sessionId 怎么从输出里拿到**:headless 模式跑完的结束事件里,sessionId 在哪个字段——先跑一次真实 Claude 抓个样例。
2. **国内模型 CLI 的输出格式**:是不是也支持 stream-json?如果不是,在适配器里做格式转换。
3. **WSL / Windows 混合环境**:后端启动 `claude` 时,这个程序在 Windows 里还是 WSL 里?要能自动定位(项目已有 resolveDataDir 那套自探测思路可借鉴)。
