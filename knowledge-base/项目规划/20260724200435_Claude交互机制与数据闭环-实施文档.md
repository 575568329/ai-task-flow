## 一、全景:一条闭环

```
┌─────────┐   录任务      ┌──────────────────┐
│  看板前端 │ ──HTTP───►   │  HTTP Server 进程  │
│ (React)  │ ◄──SSE────  │  (Fastify, :5678) │
└─────────┘   实时推送    └────────┬─────────┘
     ▲                            │ 读写
     │ 看板刷新                    ▼
     │                   ┌──────────────────┐
     │                   │ ~/.ai-task-flow/  │ ← 单一数据源(JSON)
     │                   │   tasks.json      │
     │                   └────────┬──────────┘
     │                            │ 轮询感知(mtime)
     │              ┌─────────────┴──────────────┐
     │              ▼                            │
     │   ┌──────────────────┐         ┌─────────────────────┐
     │   │  MCP Server 进程  │         │  Claude Code CLI 进程 │
     │   │   (stdio)        │ ◄────►  │  (用户在终端启动)     │
     │   └──────────────────┘  拉任务  └──────────┬──────────┘
     │                                    回写    │ 直连 LLM
     │                                            ▼
     │                                   ~/.claude/projects/
     │                                     <sessionId>.jsonl  ← 会话日志
     │                                            │ 扫描回采(旁路)
     └────────────────── 用量面板 ◄────────────────┘
```

**三件事一眼看穿**:

1. **两个进程,互不相识**:前端连的是 HTTP 进程,Claude Code 拉起的是 MCP 进程,两边没有内存共享、没有直接调用。它们唯一的共同语言是磁盘上的 `tasks.json`。
2. **主回路(MCP)负责「编排放行」**:Claude 经 MCP 拉任务、回写结果/步骤 → 写文件 → HTTP 进程感知 → SSE 推前端。
3. **旁路(回采)负责「事后算账」**:Claude 直连 LLM,不经我们后端 → 我们碰不到它的 token 消耗 → 靠扫它的会话日志(`jsonl`)非侵入地补上用量数据。

## 二、两条进程,一个文件(双进程协作)

这是理解整条链路的认知前提,不讲清这个,后面所有机制都会困惑「它们怎么知道对方干了啥」。

| 进程 | 谁启动它 | 跟谁通信 | 对 `tasks.json` |
|------|---------|---------|-----------------|
| **HTTP Server** | 用户起 `npm run dev`(dev)或看板服务 | 前端(HTTP+SSE) | 读为主 + 写(看板编辑) |
| **MCP Server** | Claude Code 进程按 `.mcp.json` 拉起(stdio) | Claude Code | 读为主 + 写(回写结果/步骤) |

**核心矛盾**:MCP 进程写完 `tasks.json`,HTTP 进程怎么知道?两者没有共享内存、没有 IPC 通道、没有数据库的触发器。

**解法:轮询文件 mtime**。`TaskFileWatcher`(`backend/src/infrastructure/persistence/TaskFileWatcher.ts`)用 `fs.watchFile(tasks.json, { interval: 1000 })` 每秒 stat 文件,发现 mtime/size 变了就读出全文、与上次基线比对——**整文件内容变了才算「外部写入」**(不做任务级 diff),发一个 `TasksExternallyChanged` 事件,HTTP 进程订阅后经 SSE 通知**前端全量重拉**任务列表。

> ⚠️ **别把 `TaskUpdated` 当成跨进程信号**。`TaskUpdated` 是 domain 聚合根在**进程内**产生的事件。MCP 进程里 `recordResult()` 也会产生它,但 MCP 进程是 stdio、**没有连向前端的 EventBus**——**这个事件到不了前端**。前端能刷新,完全且仅靠 HTTP 进程 `TaskFileWatcher` 发出的 `TasksExternallyChanged`(全量重拉)。记牢:**跨进程的前端刷新只认 `TasksExternallyChanged`**。

> **为什么不用 IPC / 文件锁 / 数据库?** KISS。`tasks.json` 是单文件、体量小(几十 KB),1 秒轮询延迟对「人看板看任务」这种低频场景完全够用,且**跨进程零依赖、崩溃后自恢复**(轮询天然幂等)。引入 IPC 或 DB 是为「不是问题的问题」上重型解。

## 三、机制一:任务标记注入 → 用量回采关联(本文最该讲的部分)

这是本轮(TASK-006)新增、`MCP_TOOLS_GUIDE.md` 完全没提、也是整条链路里**最巧妙**的一环。

### 3.1 问题

Claude Code 是**直连 LLM**的——它把请求发到 Anthropic / 国内模型 API,响应直接回到 Claude Code 自己,根本不经过我们的后端。所以我们天然**看不到**:

- 这个会话烧了多少 token?
- 这个会话是为了哪个任务?(Claude 可能一会话聊多个任务)

没有任何主动上报通道,只能**被动观察**。唯一稳定、非侵入的观察点,是 Claude Code 自己写的会话日志:`~/.claude/projects/<encodeProjectPath(cwd)>/<sessionId>.jsonl`(每行一个 JSON 消息)。

### 3.2 方案:种标记 + 扫日志

**两步,缺一不可**:

**① 种标记(写)**:`get_task` 返回的 Markdown 顶部,注入一行 HTML 注释:

```ts
// backend/src/interfaces/mcp/server.ts:372
let markdown = `<!-- ai-task-flow: task=${task.id.value} -->\n` + lines.join('\n');
```

这行随 `get_task` 的返回进入 Claude 的会话上下文,作为 `tool_result` 被写进该会话的 `jsonl`(属于一条 `type: 'user'` 消息)。

**② 扫标记(读)**:`ClaudeSessionScanner.parseSessionMeta`(`backend/src/infrastructure/system/ClaudeSessionScanner.ts`)逐行扫会话日志:

- 在 `user` 行的 `tool_result.content` 里递归提取所有文本(`collectAllText`,处理嵌套 content 数组);
- 正则匹配其中的 `ai-task-flow: task=XXX` 核心(代码 `/ai-task-flow:\s*task=([A-Z0-9-]+)/i`,**不校验两侧 `<!-- -->`**,更宽松,防 Claude 对注释格式微调时漏匹配);
- 一个会话可能多次 `get_task` 不同任务 → **取出现次数最多的(众数)** 作为该会话的主任务;
- 没有任何标记 → `taskId` 为 `undefined`(纯闲聊会话,不归因)。

归因成功后,该会话内**所有 `assistant` 行**的 `message.usage` 就被累加到这个任务名下。

### 3.3 为什么这么做(三个 Why)

| 决策 | 原因 |
|------|------|
| **标记用 HTML 注释 `<!-- -->`** | 前端 Markdown 渲染器忽略它(用户看不到噪音),但它**确确实实写进了 jsonl**,scanner 能扫到。既能传信又不污染视觉。 |
| **标记随 `tool_result`(user 行)而非 assistant 文本** | `get_task` 的返回天然是 tool_result,结构稳定、一定包含完整标记;assistant 自由文本里塞标记会被 Claude「理解」或改写,不可靠。 |
| **多标记取众数而非首个** | Claude 真实工作流里,一会话先做 TASK-006、再帮看一眼 TASK-007 很常见。取众数 = 取「这个会话主要在为谁干活」,比「第一次 get 的」更准。 |

### 3.4 用量拆分:cache 5m vs 1h(计费正确性)

Anthropic 的 prompt cache 计费和基础输入不同,直接拿 `usage` 字段会算错钱:

- `input_tokens` / `cache_creation_input_tokens` / `cache_read_input_tokens` 三者是**互斥桶**(同一段 token 只会落进一个);
- cache write 又分 **5 分钟**(`1.25×`)和 **1 小时**(`2×`)两档,前者在 `usage.cache_creation_input_tokens`,后者藏在 `message.cache_creation.ephemeral_1h_input_tokens`。

所以 `accumulateUsage` 做了拆分:

```
cacheCreation1h = ephemeral_1h_input_tokens
cacheCreation5m = max(0, cache_creation_input_tokens - cacheCreation1h)
```

`estimateCost`(`backend/src/infrastructure/llm/modelPricing.ts`)按模型单价 + 各档系数(读 `0.1×`、写 5m `1.25×`、写 1h `2×`)算成本。这是用量面板数字「可信」的根。

### 3.5 性能:扫描不在请求路径

扫 jsonl 是重 IO(WSL 跨文件系统读 Windows 的 `.claude` 更慢),绝不能在每次打开用量面板时全量扫。`UsageService`(`backend/src/application/usage/UsageService.ts`)两层缓存:

- **L1(全量结果缓存)**:整体扫描结果缓存 60 秒(`STALE_MS`),过期才触发重扫;
- **L2(文件级增量)**:维护 `sessionId → mtime` 表,重扫时只 reparse mtime 变了的会话(用 `listAllSessionFiles` 仅 stat 不读内容),没变的直接复用上次解析结果。

效果:面板首次打开扫一次;60 秒内再开秒回;有新会话活动时只增量解析增量部分。

## 四、机制二:双进程文件感知与实时同步

承接第二节,这条是**主回路**的「回写 → 前端刷新」链。

```
Claude 调 record_result / complete_step
        │
        ▼
MCP Server: task.recordResult() / setStepCompleted()  (domain 层状态机)
        │
        ▼
taskRepository.save(task)  →  写 ~/.ai-task-flow/tasks.json
        │
        ▼ (mtime 变了)
TaskFileWatcher(fs.watchFile, 1s)  →  发 TasksExternallyChanged(整文件变了)
        │
        ▼
HTTP 进程订阅 → SSE 通知前端全量重拉
        │
        ▼
看板任务卡片刷新(状态/进度/结果)
```

> **注意时序**:MCP 回写后,看板**不是瞬间刷新**——要等 TaskFileWatcher 下一轮轮询(≤1s)+ SSE 往返。实测体感「等一两秒」。这是跨进程架构的固有代价,换来的是零依赖、零锁、崩溃可恢复。**别当成 bug 去查「为什么没实时」**。

### 4.1 四态状态机的自动推进

回写不只是存数据,还驱动状态机。状态转换规则见 `TaskStatus.isValidTransition`(`backend/src/domain/workflow/value-objects/TaskStatus.ts`),但真正常用的是两条**自动推进**:

- `complete_step` 勾某步但未全勾 → `TODO → IN_PROGRESS`(只在当前是 TODO 时推进,避免从 DONE/BLOCKED 误抽出);
- 所有步骤全勾 → `* → DONE`(`setStepCompleted` 每次都检测);
- `record_result(done/partial)` → `DONE`,并**兜底全勾所有步骤**(避免「已完成但卡片进度停在 0/N」的视觉错位)。

> 这部分(流转表 + 工具参数)在 GUIDE 第三节有完整版,本文只点机制,不重复表。

## 五、机制三:WSL / Windows 混合环境适配

后端常跑在 Windows,而 Claude Code / MCP 常跑在 WSL(或反之)。两个环境里 `os.homedir()` 是**不同的物理目录**,这是本项目踩过最深的坑(详见 [WSL与Windows环境变量桥接通用方案](../技术方案/工具流程/20260724095424_WSL与Windows环境变量桥接通用方案.md))。Claude 交互链路受其直接影响的有两处:

### 5.1 截图双路径(WSL 访问不到 localhost)

任务步骤里可能贴截图,存为 `http://localhost:5678/api/uploads/xxx.png`。但 **WSL 侧的 Claude Code 访问不到 Windows 的 localhost**(WSL2 NAT),拿到的是死链。

`get_task` 在返回前做替换(`server.ts:366-395`):把每张本服务上传的图,替换成**两种本地路径**,Claude 按自身运行环境 `Read` 其中一个:

```
（截图 1:按你的环境用 Read 读取本地文件）
  - WSL:     /mnt/c/Users/<user>/.ai-task-flow/uploads/xxx.png
  - Windows: C:\Users\<user>\.ai-task-flow\uploads\xxx.png
```

> 为什么给本地路径、而不是塞 base64 进上下文?**省 token**——截图 base64 动辄几万 token,而 Claude 用 `Read` 工具读图只占实际需要的那部分,且能复用缓存。

### 5.2 数据目录定位(resolveDataDir)

MCP 进程要找到 `tasks.json`,但它在 WSL 跑时 `os.homedir()` 指向 `/home/<wsl-user>`,而文件可能在 Windows 侧 `C:\Users\<win-user>\.ai-task-flow\`。`resolveDataDir()`(`backend/src/config/dataDir.ts`)是**唯一入口**,自探测优先级:

**显式参数 > 环境变量 > WSL 自探测 Windows home(`cmd.exe /c echo %USERPROFILE%` + `wslpath` 转换)> 默认 `os.homedir()`**

> 原则(项目红线):**运行时自探测 > 外部环境注入**。曾尝试用 WSLENV 桥接数据目录,但 Windows Terminal 启动 wsl.exe 时用进程级 WSLENV **覆盖**注册表 User 级,从 WT 启动的 WSL 永远拿不到桥接变量(`wsl --shutdown` 也没用)。自探测不依赖任何外部前提,clone 即用,失败优雅降级到默认 home。

## 六、一次完整交互的时序(端到端走一遍)

把前面所有机制串成一条真实时间线。场景:看板上有个 `TASK-006`,要交给 Claude 做。

| 步 | 动作 | 触发的链路 |
|----|------|-----------|
| 1 | 用户在看板录好任务(标题/描述/步骤/截图),状态 `TODO` | 前端 → HTTP → 写 `tasks.json` |
| 2 | 用户在**项目根**打开 Claude Code(自带 `.mcp.json`,自动连 MCP) | Claude 拉起 MCP stdio 进程 |
| 3 | Claude 调 `list_pending_tasks` | MCP 读 `tasks.json`,返回 Markdown 表 |
| 4 | Claude 调 `get_task("TASK-006")` | MCP 读任务,**注入 `<!-- ai-task-flow: task=TASK-006 -->` 标记 + 截图双路径**,返回 |
| 5 | Claude 开始执行(改代码、跑测试),标记随 tool_result 写入会话 `jsonl` | (旁路埋点完成) |
| 6 | Claude 每完成一步调 `complete_step("TASK-006", 1)` | MCP → 写文件 → TaskFileWatcher → SSE → **看板进度更新,卡片 `TODO → IN_PROGRESS`** |
| 7 | Claude 调 `record_result("TASK-006", status:"done", ...)` | MCP → 写文件(状态转 `DONE` + 兜底全勾)→ SSE → **卡片移到「已完成」列** |
| 8 | (任意时刻)用户打开**用量面板** | UsageService:L1 命中则秒回;过期则扫 jsonl → 凭标记把本会话 token 归因到 `TASK-006` → 五维度汇总 |

走完这八步,「看板录入 → Claude 干活 → 看板同步完成 → 用量归因」的环就闭合了。

## 七、关键设计决策(Why 清单)

把散落各处的「为什么」收口成一张表,便于后人理解时不用重新推理:

| # | 决策 | 为什么 |
|---|------|--------|
| 1 | **用 MCP 协议,不嵌入 Agent SDK** | 保留原生 Claude Code 体验;用户在终端、用熟悉的 Claude Code 干活,我们只通过 MCP 暴露工具。v2 试过嵌入,用户反馈「失去原生体验」才改的。 |
| 2 | **扫 jsonl,不代理 Claude 流量** | Claude 直连 LLM,我们碰不到流量;jsonl 是它自己写的稳定日志,非侵入、零耦合。 |
| 3 | **标记走 HTML 注释 + tool_result** | 不渲染(无视觉噪音)+ 永久写进 jsonl(可扫)+ 结构稳定(Claude 不会改写 tool_result)。 |
| 4 | **双进程靠 `fs.watchFile` 轮询,不靠 `fs.watch` / IPC / DB** | `fs.watch` 跨进程监听不可靠(外部进程写入易丢事件/丢句柄,Node #47058;Windows 单次写还多次触发);轮询主动 stat 跨进程 100% 可靠,单文件 1s 开销可忽略,崩溃可恢复。 |
| 5 | **回写不要求 `DISPATCHED`** | 会话化改造后,打开终端不改状态,`TODO` 可直接回写结果。去掉「派发→审核」两段式。 |
| 6 | **用量扫描不在请求路径(L1/L2 缓存)** | WSL 跨文件系统读 Windows 的 `.claude` 很慢;60s 全量缓存 + mtime 增量,把扫描挡在请求路径之外。 |
| 7 | **截图给本地路径,不给 base64** | 省 token + 绕开 WSL 访问不到 localhost 的问题;Claude 用 `Read` 读图天然支持缓存。 |
| 8 | **数据目录运行时自探测,不靠 WSLENV** | 外部环境注入依赖一长串前提,任一环失效就静默崩(WSLENV 被 Windows Terminal 覆盖就是教训);自探测自洽。 |

## 八、与未来「前端实时对话」方案的关系

`20260724171015` 描述的是**另一条还没做的通道**——后端用 `child_process.spawn` 启动 Claude Code(headless + `stream-json`),把它的流式事件转发给前端对话窗。两者要分清:

| | 本文(MCP 通道) | 71015(spawn 通道) |
|---|---|---|
| **谁启动 Claude** | 用户自己在终端启动 | 后端替前端启动 |
| **传输** | MCP stdio(工具调用) | spawn stdout 流式事件 |
| **交互形态** | 任务编排(拉任务/回写) | 实时对话(看得见思考过程) |
| **状态** | ✅ 已落地 | ⏳ 未实施 |

两者**不互斥**:MCP 通道负责「把任务交给 Claude、收结果」,spawn 通道负责「前端直接和 Claude 聊、看实时过程」。未来若做 71015,MCP 这套机制(状态机、回写、回采)全部复用,只是多一条对话通道。

## 附录:术语速查

- **MCP 进程 / HTTP 进程**:两个独立 Node 进程,前者 Claude 拉起(stdio),后者前端连(HTTP+SSE),靠 `tasks.json` 文件协作。
- **标记**:`get_task` 注入的 `<!-- ai-task-flow: task=TASK-xxx -->`,用于把会话用量归因到任务。
- **众数归因**:一会话多任务标记时,取出现最多的作为主任务。
- **回采**:扫 Claude 会话 jsonl 统计 token 用量的旁路链路。
- **双路径**:WSL(`/mnt/c/...`)+ Windows(`C:\...`)两种本地截图路径,绕开 WSL 访问不到 localhost。
