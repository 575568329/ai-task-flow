# 网页内容剪藏插件设计（Web Clip to Task Extension）

> 版本：v0.1（设计评审中） · 日期：2026-06-16 · 关联：AI Task Flow v0.2.0+

## 1. 背景与目标

### 1.1 现状痛点
当前 AI Task Flow 的任务录入完全靠**手动**：粘贴图片 + 手敲描述。典型场景：测试在缺陷管理平台把**一批同类问题**（如 N 个「文本未国际化」点）塞进**一个缺陷**的富文本里，配多张截图（1截图/2截图/3截图），每个截图对应一个步骤（地址 + 现象 + 期待结果）。开发拿到这「一坨」，要手工抠文字、另存截图、逐条录入看板，既慢又易漏。

### 1.2 目标
做一个**通用浏览器插件**：在任意网页（缺陷页/飞书文档/富文本博客）一键抓取内容 + 图片，经 AI 自适应拆解成任务草案，侧边栏内确认后建到 AI Task Flow 看板。

### 1.3 核心约束（与用户已敲定）
| # | 决策 | 说明 |
|---|------|------|
| 1 | 形态 | 浏览器扩展（Manifest V3，Chrome + Edge 通用） |
| 2 | 通用性 | **输入端通用**（任意网页，不写死 CSS 选择器）；**输出端固定**（只往 AI Task Flow 建任务） |
| 3 | 抓取策略 | 划词优先（有 selection 读选中区），否则 Readability 自动提取正文 |
| 4 | 图片 | 通用能力：抓取范围内 `<img>` 统一 fetch→base64，不依赖任何网站鉴权 |
| 5 | 拆解粒度 | AI **自适应**按语义边界拆（步骤/地址/段落/列表项），不预设「这是缺陷」 |
| 6 | 草案确认 | **侧边栏内确认**，不跳转离开当前网页 |
| 7 | MVP 范围 | 标准版：两种抓取 + 拆解 + 草案编辑确认 + 批量建任务 |
| 8 | 合规 | 讯飞内网允许装第三方扩展、读缺陷数据（已确认） |
| 9 | 任务结构 | **复用现有 `Task.steps`**（`TaskStep` 图文块），不为步骤新增字段；拆解产出 `steps`，直接走现有 `CreateTaskRequest` 建任务。粒度=地址/URL 级，同地址多问题点→同任务的多个 steps，通常一个网页只建 1 个任务 |
| 10 | 来源区分 | 任务区分两种来源：**web**（插件剪藏，带 `sourceUrl` 网页地址）vs **manual**（手动新建，带 `repoPath` 项目地址）。Task 新增 `source` + `sourceUrl` 字段；列表/看板支持按来源筛选与区分展示 |
| 11 | 侧边栏形态 | 编辑界面用 **Side Panel（常驻侧边栏）** 而非 popup（popup 瞬态、点页面即关，无法"边选边编辑"）。侧边栏是**多视图宿主**：剪藏(ClipView)为首个视图，对话(ChatView)等后续视图预留注册接口，加功能=注册一个 `SidePanelView`，不改容器 |

## 2. 架构总览（端到端）

```
[任意网页]                 [浏览器扩展 MV3]                  [AI Task Flow :3000]              [侧边栏]
缺陷页/飞书/富文本   →   content script 抓取         →   POST /api/tasks/clip         →   侧边栏内草案确认
(可选先划词)             划词优先 / 否则 Readability     WebClipService                     编辑/勾选/关联图
                         图片 fetch→base64              存图 + 调 LLM 通用拆解             批量 POST /api/tasks
                                                        → N 个任务草案
```

**数据归属**：扩展负责「采集 + 转码」；后端负责「存图 + AI 拆解」；看板前端负责「草案确认 + 建任务」。三者通过 `/api/tasks/clip` 与现有 `/api/tasks` 串联。

## 3. 组件设计

### 3.1 浏览器扩展（新增 `extension/` 目录，MV3）

侧边栏（Side Panel）是**多视图宿主**——剪藏只是首个视图，对话等后续视图预留接口，加功能=注册一个 `SidePanelView`，不动容器。

| 文件 | 职责 |
|------|------|
| `manifest.json` | MV3 配置：`permissions: [sidePanel, activeTab, storage, scripting]`；`side_panel.default_path` 指向侧栏入口；`host_permissions: [http://localhost:3000/*]`（**不声明 `<all_urls>`**，靠 `activeTab` 在点「抓取本页」时获得当前页临时权限，无高权限警告）；`action.onClicked` → 打开/聚焦侧栏；content script 经 `chrome.scripting.executeScript` 动态注入 |
| `background.ts` | service worker：开/聚焦侧栏、转发抓取请求到后端、回传草案 |
| `content/clip.ts` | **核心采集脚本**：抽成 `capturePageContext(opts)`，按需返回 `{url,title,选区文本,正文,图片base64}`。划词优先（有 `getSelection`），否则 Readability。`img`→fetch→base64 |
| `sidepanel/SidePanel.tsx` | **容器**：顶部视图 Tab 切换 + 视图路由 + 底部连接状态；当前 Tab 记忆到 `chrome.storage` |
| `sidepanel/views/registry.ts` | **视图注册表（扩展点）**：`VIEWS: SidePanelView[]`，加功能=push 一项 |
| `sidepanel/views/types.ts` | 视图契约 `interface SidePanelView { id; title; icon; component }` |
| `sidepanel/views/ClipView.tsx` | **剪藏视图（本次实现）**：「抓取本页」→ 草案预览 → 编辑标题/勾选步骤/关联图 → 「建任务」 |
| `sidepanel/views/ChatView.tsx` | **对话视图（未来，预留）**：复用看板 `/api/chat` + 检索；默认带「当前页面上下文」 |
| `sidepanel/lib/pageContext.ts` | **共享页面上下文**：所有视图消费同一份抓取结果（剪藏建任务 / 对话喂 AI） |

**抓取伪代码**（划词优先，否则 Readability；`wantImages` 控制是否捞图）：
```ts
async function capturePageContext(opts: { wantImages?: boolean }): Promise<PageContext> {
  const sel = window.getSelection();
  let text: string, html: string;
  if (sel && sel.toString().trim()) {            // 划词优先：取选区范围（圈住的 img 也会带出）
    const range = sel.getRangeAt(0);
    text = range.toString(); html = rangeToHtml(range);
  } else {                                        // 正文自动：Readability 去噪
    const article = new Readability(document.cloneNode(true) as Document).parse();
    text = article?.textContent ?? document.body.innerText;
    html = article?.content ?? document.body.innerHTML;
  }
  const images = opts.wantImages ? await collectImages(html) : undefined; // img→base64
  return { sourceUrl: location.href, title: document.title, text, html, images };
}
```

### 3.2 后端新增（AI Task Flow）

| 文件 | 职责 |
|------|------|
| `backend/src/interfaces/http/routes/webClip.ts` | 新接口 `POST /api/tasks/clip` |
| `backend/src/application/webclip/WebClipService.ts` | 应用层：① 图片 base64 复用现有上传逻辑落 `uploads/` 拿 `/api/uploads/xxx` url；② 调 `LlmConfigService` provider 跑通用拆解 prompt；③ 产出 `TaskStep[]`（image block 填 url）；④ 返回任务草案（通常 1 个，含 steps） |
| `backend/src/application/webclip/clipPrompt.ts` | 通用拆解 prompt（见 §5） |
| `shared/src/types/webclip.ts` | 前后端共享类型：`ClipRequest / ClipDraft / ClipResponse` |

**领域模型**：
- **steps 复用现有** `TaskStep`（图文块），不为步骤新增字段。
- **新增来源字段**（区分两种来源，是本次唯一需要动模型的地方）：
  - `source: 'web' | 'manual'`（默认 `'manual'`，兼容现有手动任务）
  - `sourceUrl?: string`（仅 web 来源，= 抓取页面 URL）
- 图片 url 指向现有 `/api/uploads/xxx`（复用 `POST /api/upload/image`）。

WebClipService 拆解后产出 `TaskStep[]`，建任务时 `source='web'`、填 `sourceUrl`，经现有 `CreateTaskRequest`（扩展 source/sourceUrl）入库。

> **插件建任务的项目信息**：`repoPath`/`prefix` 可选——剪藏时通常不知道目标仓库，留空则任务以 `planning` 状态入库，用户后续在看板补全项目信息再派发（建 worktree）。

### 3.3 看板前端（基本无需改动）

看板已有 `Task.steps` 的展示（TaskCard / TaskDrawer），插件建出的任务（含 steps 图文块）在看板正常显示，**无需新增组件**。唯一改动：TaskDrawer 的「地址」字段按 `source` 条件渲染（web 显 sourceUrl / manual 显 repoPath，见 §3.4）。草案确认（编辑标题/描述/勾选步骤）全部在**插件侧边栏的 ClipView** 内完成（插件是独立工程，不复用 frontend 代码，避免跨工程构建复杂度）。确认后插件直接调后端 `POST /api/tasks` 建任务，看板经 SSE 自动刷新出现新任务。

### 3.4 来源区分与筛选

任务两种来源需区分，对应不同的「地址」字段：

| 来源 | 标识字段 | 含义 | 谁产生 |
|------|---------|------|--------|
| `web` | `sourceUrl` | 网页地址（抓取时的 `location.href`） | 插件剪藏 |
| `manual` | `repoPath` / `projectName` | 项目地址（代码仓库路径） | 看板手动新建 |

**筛选与展示（卡片 + 详情页统一按来源条件渲染「地址」，二选一不并存）**：
- 列表 API：`GET /api/tasks?source=web|manual` 支持按来源过滤；响应体含 `source` / `sourceUrl`
- **任务详情（TaskDrawer）**：按 `source` 条件渲染「地址」字段——
  - `web` → 显示「网页地址」`sourceUrl`（可点击跳回原页），**不显示**项目地址
  - `manual` → 显示「项目地址」`repoPath` / `projectName`，**不显示**网页地址
- 看板卡片角标：🌐 `web`（显 sourceUrl 域名）/ 📁 `manual`（显 projectName）
- 看板顶部可选加「来源」筛选条

### 3.5 扩展蓝图：侧边栏为多视图宿主（为未来功能留余地）

侧边栏从设计起就是**多视图容器**，不是写死的"剪藏器"。遵循原则：**加功能 = 注册一个 `SidePanelView`，容器逻辑不动**。

**当前视图**：`ClipView`（剪藏）—— 本次实现。

**预留视图（不实现，只留接口与接入路径）**：
- `ChatView`（对话）—— 把看板「资料调研」搬到侧边栏，在任意网页上一边浏览一边问 AI：
  - **复用（零后端改动）**：`/api/chat`、`SearchOrchestrator`（GLM Web Search + arXiv）、LLM provider —— 侧栏只是又一个 chat 前端入口
  - **新增**：侧栏对话 UI + 默认把「当前页面上下文」（选区/正文）作为对话背景（看板 chat 没有的能力）
  - **前端复用决策（未来定）**：从 frontend 抽共享 chat 组件，或侧栏精简实现

**复用核心：共享页面上下文**
`capturePageContext()` 抓一次，产出 `PageContext { sourceUrl, title, text, html?, images? }`：
- 剪藏视图：用 `text + images` 建任务
- 对话视图（未来）：用 `text` 作为"我正在看这段，帮我…"的背景喂给 AI

**实现纪律**：每个视图自包含，只依赖 ① 共享 `PageContext` ② 各自的后端 API，视图之间互不知晓。新功能进来时只需 `VIEWS.push({...})`。

## 4. 数据契约

### 4.1 `POST /api/tasks/clip`（扩展 → 后端）

**Request**
```jsonc
{
  "sourceUrl": "https://缺陷平台/bug/123",
  "title": "【澳门底层题库】文本未国际化问题汇总",
  "content": { "html": "<...>", "text": "1、审核判定页面... 2截图..." },
  "images": [
    { "name": "img-1", "base64": "data:image/png;base64,iVBOR..." }
  ]
}
```

**Response 200**（draft 直接含 `steps: TaskStep[]`，图片内联为 image block）
```jsonc
{
  "drafts": [
    {
      "title": "审核判定页面文本未国际化",
      "description": "页面位置：审核判定页面（AI 从内容提取，可选）",
      "steps": [
        { "blocks": [
          { "type": "text", "content": "现象：未审核时点提交，提示文本未国际化\n期待：提示文本国际化正确" },
          { "type": "image", "url": "/api/uploads/2026-06-16/abc.png" }
        ]}
      ]
    }
  ],
  "sourceUrl": "https://缺陷平台/bug/123"
}
```
> 同一地址的多个问题点 → 同一任务的多个 `steps`；不同地址 → 不同 `drafts`（通常只有 1 个 draft）。

### 4.2 草案确认 → 建任务（侧边栏 → 后端，复用现有接口）
确认后的每个草案 → `POST /api/tasks`，入参：`source: 'web'`、`sourceUrl`、`title`、`description`、`steps: TaskStep[]`（图文块，复用现有）。后端落库触发 SSE，看板刷新。

## 5. 通用拆解 Prompt 设计（决定拆得准不准）

不写死「这是缺陷步骤」。给 LLM 正文文本 + 图片占位，要求自适应拆分：

```
你是任务拆解助手。下面是从网页抓取的内容（含图片占位 [图1][图2]）。
请按语义边界把它拆成若干个独立、可执行的任务。

拆分规则（自适应，不预设内容类型）：
- 若内容含步骤/地址/编号结构（如"1、2、3"或"步骤1"），按步骤拆；
- 若是列表，按列表项拆；
- 若是连续段落且各自独立主题，按主题拆；
- 若整体是一个主题，就返回 1 个任务（不要硬拆）。

**关键：按地址/URL 分组**——同一页面（sourceUrl）的多个问题点归到**同一任务的多个 steps**；不同页面才拆成不同任务（通常只产出 1 个任务，含多个步骤）。

每个任务输出：title、description（整体说明，可含页面位置）、steps（步骤数组，每步是有序图文块）。
image block 用 ref（如 "img-1"）引用抓取阶段的图片占位，后端替换为真实 url。

只输出 JSON：{ "drafts": [ {title, description, steps:[ {blocks:[ {type:"text",content} | {type:"image",ref:"img-1"} ]} ]} ] }

> `sourceUrl` 由插件在请求层提供（`location.href`），不在 AI 输出里；建任务时填入 `Task.sourceUrl`。

----- 网页内容 -----
{content.text}
```

**关键**：用结构化输出（复用 LlmConfigService 的 `generateObject` / JSON 模式），保证草案可解析。

## 6. 错误处理（各自降级）

| 场景 | 处理 |
|------|------|
| 后端不可达（localhost 未启动） | 侧边栏报错「无法连接 AI Task Flow，请确认后端已启动（:3000）」 |
| LLM 拆解失败/超时 | **回退为 1 个整任务**（原文进 description，所有图都关联），不返回空，不让用户白等 |
| 图片 fetch 跨域被 CORS 拦 | 跳过该图，记 warning，继续拆解（不阻断） |
| 抓取为空（无 selection 且 Readability 失败） | 侧边栏提示「未抓到内容，请先在页面上选中内容再点「抓取本页」」 |
| LLM 返回非法 JSON | 走 `extractJsonObject` 兜底解析；仍失败则回退 1 个整任务 |

## 10. 风险与未决

| 风险 | 影响 | 缓解 |
|------|------|------|
| LLM 拆解准确率 | 拆错/漏拆 | 通用 prompt + 回退整任务 + 侧边栏人工确认兜底；上线后用真实样例迭代 prompt |
| 图片跨域 CORS | 部分图抓不到 | content script 同源优先；跨域失败跳过不阻断；后续可加 background 代理 fetch |
| 侧边栏宽度有限 | 复杂草案（宽表格/长图）横向拥挤 | 可拖宽侧栏 + 「新标签页展开」兜底 |
| 缺陷平台改版 | Readability 不依赖选择器，**不受影响**（这是选 Readability 的核心收益） | — |
| 后端无鉴权 + LAN 可达 | 他人可调 `/api/tasks/clip` 建任务 | 与项目整体定位一致（公司内网），已知限制，非本插件引入 |
