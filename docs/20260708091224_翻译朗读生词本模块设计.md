## 一、背景与目标

参考 [Sider](https://sider.ai/extensions/ai-chrome-extension) 的划词翻译 + 朗读,做一套属于本系统的轻量方案:

- 任意网页选中文字 → 翻译(译文 + 词性 + 释义 + 例句)→ 本地朗读
- 一键存为生词 → 同步到 ai-task-flow 后端(`~/.ai-task-flow/vocab.json`)
- 网页前端「生词本」页随时查看 / 搜索 / 标记掌握

砍掉 Sider 的 AI 对话、多模型对比、整页翻译(MVP 不做)。

## 五、技术选型

| 决策 | 选择 | Why |
|---|---|---|
| 翻译引擎 | 后端 LLM(`llmConfigService.getProvider()`) | 零额外 key、译文带词性/释义/例句(结构化)、天然能记生词;扩展不持有 key 更安全。代价:依赖后端在线 |
| 朗读(TTS) | 浏览器原生 `speechSynthesis`([Web Speech API](https://developer.mozilla.org/zh-CN/docs/Web/API/Web_Speech_API)) | 免费、零延迟、离线可用、扩展与网页前端跨端一致、零额外权限。`chrome.ttsEngine` 是给"扩展充当系统 TTS"用的,不适用 |
| 交互 | 侧栏翻译(点图标 → 自动取选区 → 翻译) | 完全踩在现有 sidePanel + activeTab 轨道,不碰 content script、不要新 host 权限,最快闭环 |
| 扩展 React 版本 | 维持 18,不跟前端升 19 | 扩展独立 build,解耦避免连带重构(遵循 CLAUDE.md §1.7) |
| 鉴权 | 本地默认不强鉴权 | 本系统是本地工具;扩展默认连 localhost。内网暴露场景后置加 token |

### 5.1 技术验证结论(2026-07-08 联网核查)

| 决策 | 验证结论 | 来源 |
|---|---|---|
| TTS = `speechSynthesis` | ✅ 成立。`chrome.ttsEngine` 是"扩展充当系统 TTS 引擎"用的,不适用;`speechSynthesis` 是正路。**但** Chrome 有老 bug(Chromium #41084789):长文/中文会"播一半停"或"播一次后再播失败" → 应对:播放前 `cancel()` 清队列 + 每次新建 `Utterance` + 长文本分句 | MDN / Chromium issue |
| 取选区 = `activeTab`+`scripting.executeScript` | ✅ 可行。manifest 已有 `scripting`+`activeTab`,**零权限改动**。注入函数虽在 isolated world,但 DOM 共享,`window.getSelection().toString()` 拿得到 | Chrome scripting API |
| 通信 = `text/plain`+`PROXY_FETCH` | ✅ 架构对。**机制澄清**:Chrome 127+ 对私有网段(含 localhost)发 PNA preflight,但那只针对"网页 context → 私有网络";**扩展 background service worker 发请求不受 PNA/CORS 约束**(走 `chrome-extension://` origin + `host_permissions`)。故真正绕过 PNA 的是 `background PROXY_FETCH` 中转,`text/plain` 只是顺带减少一次普通 CORS preflight。后端 `server.ts` 已内置 onSend 补 `access-control-allow-private-network` + text/plain→JSON parser,vocab 路由**零额外配置** | Chrome PNA blog |
| 翻译 = 后端 LLM 结构化 JSON | ✅ 容错方案标准。**优化**:后端 `OpenAiCompatibleProvider.generateObject()` 已用 `response_format: json_object`(JSON mode),`translate` 优先调它,失败再回退 streamText + 容错解析,比纯 prompt 可靠 | OpenAI Structured Outputs |
| 复用剪藏扩展 + sidePanel | ✅ 符合 Chrome 116+/119+ Side Panel 现代最佳实践(独立 DOM/CSP,权限比 content script 轻) | Chrome sidePanel API |

## 九、翻译 prompt 与结构化解析

`VocabService.translate` 调 LLM,**优先走 JSON mode**(后端 `generateObject()` 已用 `response_format: json_object`,API 层强制合法 JSON),失败再回退 prompt + 容错解析。

**优先路径(JSON mode)**:`llm.generateObject<TranslateResponse>(messages, schema)` —— prompt 仍要求严格 JSON(双保险):

```
You are a translator. Translate the user's text to {{targetLang}}.
Return STRICT JSON only, no markdown fence, no extra text:
{"sourceLang":"...","translation":"...","pos":"...","definition":"...","example":"..."}
- pos: 词性(n./v./phrase 等),仅对单词有意义,句子留空
- definition: 用 {{targetLang}} 简短解释含义
- example: 一个 {{sourceLang}} 例句(可选)
```

**回退路径**(JSON mode 不支持 / 返回仍裹 fence 时,用 `parseLooseJson`):
1. 剥离 ```json ... ``` fence
2. 抽取第一个 `{` 到最后一个 `}` 子串
3. `JSON.parse`,失败则降级 `{ translation: 原始文本 }`,并 `logger.warn`

## 十三、决策记录

1. **翻译走后端 LLM**:零 key、译文结构化、能记生词、扩展不持 key(安全)。代价是依赖后端在线,可接受(本系统本就后端驱动)。
2. **朗读用 `speechSynthesis`**:跨端一致(扩展侧栏 + 网页前端)、零权限、离线。质量受系统语音影响,差则 P5 上后端 TTS。
3. **侧栏翻译而非浮层**:复用现有 sidePanel + activeTab 架构,不碰 content script、不要新 host 权限,最快闭环。浮层作为 P5 体验增强。
4. **扩展 React 维持 18**:与前端 19 升级解耦,不连带重构。
5. **本地不强鉴权**:本地工具默认信任 localhost;内网暴露场景后置加 token。
6. **TTS 补 halting bug 应对**(2026-07-08 验证):Chrome 长文/中文朗读有 Chromium #41084789 卡断 bug,`speech.ts` 封装 `cancel()` + 新建 `Utterance` + 分句三招规避。
7. **翻译优先 JSON mode**(2026-07-08 验证):后端 `generateObject()` 已就绪,优先 `response_format: json_object`,失败回退 streamText + `parseLooseJson`,比纯 prompt + 正则可靠。
8. **生词去重 = word+targetLang 唯一**(2026-07-08 定):重复存储返回 409,生词本保持干净;`reviewCount`/`lastReviewedAt` 字段现在就加(P5 复习预留)。

## 十四、风险与待确认

| 风险 | 应对 |
|---|---|
| LLM 返回非 JSON | 剥离 fence + 子串抽取 + `JSON.parse` 失败降级,`logger.warn` |
| `speechSynthesis` 中文语音质量依赖系统(Windows/Linux 可能差) | MVP 先用;不可接受再上后端 TTS(P5) |
| Chrome 朗读卡断 bug(Chromium #41084789,长文/中文播一半停) | `speech.ts`:`cancel()` 清队列 + 每次新建 `Utterance` + 长文本分句 |
| activeTab 取选区需先点图标 | 文档/提示告知;可加快捷键(P5) |
| 选区在 iframe / closed shadow DOM 取不到 | 边界,MVP 不处理,记入已知限制 |
| 扩展与后端跨域/PNA | 已由剪藏扩展验证。**机制**:靠 background service worker 代理(`host_permissions` 授权)绕 PNA,`text/plain` 只为避免普通 CORS 预检;后端 `server.ts` 已补 PNA 响应头 |
