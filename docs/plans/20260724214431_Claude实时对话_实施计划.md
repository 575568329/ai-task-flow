## 一、spike 验证结论(真实输出契约)

在本机用真实 `claude` 跑了两次(Windows `2.1.210`,`model=glm-5.2`),抠出 stream-json 真实结构。**这是后端解析 + 前端渲染的契约依据,不是猜的。**

### 1.1 claude 可执行文件位置

| 侧 | 路径 | 版本 |
|---|---|---|
| Windows | `C:\Program Files\nodejs\claude`(`/c/Program Files/nodejs/claude`) | 2.1.210 |
| WSL | `/usr/bin/claude` | 2.1.166 |

后端跑 Windows → spawn Windows 侧 `claude`。优先级:**显式/env > PATH 自探测**(`execPath` 找不到给清晰报错,不静默)。

### 1.2 事件序列(一次「读文件并回复」的 26 行流)

```
system/hook_started ×2     ← 项目/用户级 SessionStart hook,前端忽略
system/hook_response
system/hook_progress
system/hook_response
system/init                ← 含 session_id / cwd / model / version(真正初始化)
system/thinking_tokens ×15 ← thinking 进度流,前端忽略(用 assistant 里的 thinking 块渲染)
assistant                  ← turn1:含 thinking block + tool_use block
assistant                  ← (可能分多事件,按 content block 聚合)
user                       ← tool_result block(关联 tool_use_id)
assistant                  ← turn2:最终 text block
result/success             ← 终态:含 session_id + 累计 usage + total_cost_usd + duration_ms + num_turns
```

### 1.3 关键结构(逐字段)

**`result` 事件(终态,最关键)**:
```jsonc
{
  "type": "result", "subtype": "success",
  "session_id": "c6ce4fd9-597f-42ab-a347-c117adedecc6",  // ← 续接用 --resume <这个>
  "is_error": false,
  "result": "ok",                  // 最终文本
  "total_cost_usd": 0.179,
  "duration_ms": 5901, "num_turns": 1,
  "usage": {                       // ← 累计 usage(assistant 单条 usage=0,只能用这个)
    "input_tokens": 8113, "cache_read_input_tokens": 63552,
    "cache_creation_input_tokens": 0, "output_tokens": 42,
    "cache_creation": { "ephemeral_1h_input_tokens": 0, "ephemeral_5m_input_tokens": 0 }
  }
}
```

**`assistant` 事件 — `message.content[]` block 类型**:
```jsonc
[
  { "type": "thinking", "thinking": "The user wants me to read target.txt..." },
  { "type": "tool_use", "id": "call_8c3a36...", "name": "Read",
    "input": { "file_path": "D:\\spike-tmp\\target.txt" } },
  { "type": "text", "text": "文件内容是 ANTIQUE-POTATO" }
]
```

**`user` 事件 — tool_result(关联 tool_use_id)**:
```jsonc
{ "type": "tool_result", "tool_use_id": "call_8c3a36...",
  "content": [ { "type": "text", "text": "1\tANTIQUE-POTATO\n2\t" } ] }
```

### 1.4 验证对照 71015 第九节

| 71015 待确认点 | spike 结论 |
|---|---|
| sessionId 从哪拿 | ✅ `result.session_id`(subtype=success)+ `system/init.session_id`,续接 `--resume <id>` |
| 国内模型 stream-json 兼容 | ✅ GLM(glm-5.2)走 stream-json,usage 字段与 anthropic 同名,结构一致 |

### 1.5 spike 暴露的新问题(实施必须处理)

| # | 问题 | 对策 |
|---|---|---|
| 1 | **hook 污染**:用户/项目级 SessionStart hook(superpowers)注入大量上下文,spike2 `cache_read 63552` token | ✅ **已验证可行**:`--settings clean.json`(`{"hooks":{},"permissions":{"allow":[],"deny":[],"ask":[]}}`)把首轮 input 从 **35812 → 2514**(降 93%),claude 正常回复、认证 OK(GLM 走 env)。注:[Issue #11392](https://github.com/anthropics/claude-code/issues/11392) 确认 `--settings` 对 hooks 是 merge 不 replace、无 `--no-hooks`,但 permissions 清空实测能阻断 SessionStart hook 注入。cwd 需非项目目录才彻底(见下) |
| 2 | **assistant 单条 `message.usage`=0**,累计只在 `result.usage` | 实时 token 显示从 `result.usage` 取;扫 jsonl 回采的 `ClaudeSessionScanner` 若累加 assistant.usage 在 GLM 下可能为 0,**需复核**(可能要改扫 result 行) |
| 3 | **hook/thinking_tokens 事件噪音** | 前端按 `type` 白名单渲染:`assistant`/`user`/`result`;其余忽略 |
| 4 | **path 是 Windows 反斜杠** | 前端展示 tool_use input 时注意,不强转 |

## 二、前端设计(演进 TaskDrawer 栏3)

### 2.1 栏3 改造:tab 切换

```
栏3(原预览):
┌──────────────────────────────────┐
│ [ 对话 | 预览 ]     ← tab 切换     │  ← 新增 tab 头
├──────────────────────────────────┤
│ <对应 tab 内容>                    │
└──────────────────────────────────┘
```
- 「预览」tab:现有 `<MessageContent content={markdown} />` 原样搬进来,**一行不改**。
- 「对话」tab:见 2.2。
- 默认选中「对话」(对话是主场景);tab 状态留在组件内 `useState`(不持久化,关抽屉重置)。

### 2.2 「对话」tab 结构

```
┌──────────────────────────────────┐
│ 消息流(纵向滚动,自动滚到底)        │
│  ┌─ Claude ──────────────────┐   │
│  │ 💭 thinking(折叠,默认收) │   │  ← thinking block
│  │ 🔧 Read  target.txt  ▸   │   │  ← tool_use 卡片(折叠)
│  │   └ 结果:ANTIQUE-POTATO  │   │  ← tool_result(关联 id,嵌套进 tool_use 卡片)
│  │ 文件内容是 ANTIQUE-POTATO │   │  ← text block → MessageContent
│  └───────────────────────────┘   │
│  ┌─ 我 ─────┐                    │
│  │ 再读一遍  │                    │  ← 用户消息
│  └──────────┘                    │
├──────────────────────────────────┤
│ [输入框 Textarea]        [发送]   │  ← 发送 → POST /api/tasks/:id/chat 流式
└──────────────────────────────────┘
```

### 2.3 事件 → 渲染映射(契约表)

| stream-json 事件 | 前端渲染 |
|---|---|
| `assistant.content[].type=text` | 走 `<MessageContent>`(markdown/代码/mermaid 复用) |
| `assistant.content[].type=thinking` | 折叠卡片「💭 思考」,默认收起,点开看全文 |
| `assistant.content[].type=tool_use` | 折叠卡片「🔧 {name} {关键 input 摘要}」,展开看完整 input |
| `user.content[].type=tool_result` | 按 `tool_use_id` **嵌入对应 tool_use 卡片**的「结果」区(`is_error` 标红) |
| `result`(subtype=success) | 流结束;底部显示 token(`input/output/cache_read`)+ 耗时;**存 session_id** 供下次续接 |
| `result`(subtype=error / `is_error`) | 显示错误 + stderr tail |
| `system`(`hook_*`/`thinking_tokens`/`init`) | **忽略** |

> **tool_use ↔ tool_result 关联**:同一轮里,`assistant` 发 tool_use(带 id),`user` 回 tool_result(同 id)。前端维护 `id → 卡片` map,result 到了塞进去。

### 2.4 复用清单(不重写)

| 已有 | 用途 |
|---|---|
| `chat/MessageContent` | text block 的 markdown 渲染(代码高亮/mermaid 都有) |
| `chat/CodeBlock` | 代码块(若有) |
| 现有 `streamChat` 的 fetch 流式读法 | 仿写 `streamTaskChat`(ReadableStream 读 NDJSON) |
| `previewStore`(YARL) | tool_result 里若含图片路径,点开大图 |
| `useTaskStore` / `useUIStore` | 任务选中、任务数据 |

## 三、后端设计

### 3.1 AgentRunner(适配器,仿 multica `pkg/agent`)

`backend/src/application/agent/AgentRunner.ts`(新建):

```ts
// 按轮 spawn:发一条消息起一个 claude,跑完退出;下条 --resume 续接
class AgentRunner {
  run({ prompt, cwd, resumeSessionId, mcpConfigPath }): AsyncGenerator<AgentEvent>
}
```

spawn 参数(spike 验证 + multica 借鉴):
```
claude -p "<prompt>"             // prompt 走命令行:Node spawn 用 args 数组(不经 shell),无转义风险
  --output-format stream-json
  --verbose                       // stream-json 必需(spike 实测:缺则报错)
  --permission-mode bypassPermissions   // autonomous 放行
  --max-turns <N>                 // 防失控(任务对话建议 50)
  --settings <clean.json>         // ✅ 隔离用户级 superpowers hook(spike 实证 input 35k→2.5k)
  [--resume <sessionId>]          // 续接
  [--mcp-config <tmp>]            // 可选:挂本项目 MCP,让 spawn 出的 claude 能回写状态
```
- **clean.json 内容**:`{"hooks":{},"permissions":{"allow":[],"deny":[],"ask":[]}}`(随包内固定文件,不依赖用户环境)。
- **cwd** = 任务的 `worktree.path`(派发后)或 `repoPath`(未派发)。⚠️ 真实 cwd 是项目内目录 → 会加载**项目** `.claude/CLAUDE.md`(对任务有用,保留);`--settings clean.json` 隔离的是**用户级** superpowers。实施后实测一轮,确认 input 在合理范围(应 < 15k = claude system + 项目 CLAUDE.md)。
- **stdin** 重定向 `/dev/null`(spike 实测 claude 会等 stdin 3s 才 proceed,显式关掉避免延迟)。
- **stdout** 逐行读 NDJSON → 解析 → 产出 `AgentEvent`(前端契约的事件)。
- **stderr** tail(失败带最后几 KB,仿 multica `newStderrTail`)。
- **execPath 探测**:显式/env > `claude`(PATH);找不到给清晰错误。
- **进程清理**:AbortController / 超时 kill,防僵尸(spike1 偶发 "Shell cwd was reset" 提示需注意子进程回收)。

### 3.2 路由:`POST /api/tasks/:id/chat`

`backend/src/interfaces/http/routes/taskChatRoutes.ts`(新建):
- 流式响应(`text/event-stream`,沿用现有 SSE 基建)。
- body:`{ message: string }`。
- 取任务 → 取 `worktree.path || repoPath` → 取上次 `sessionId`(存哪见 3.3)→ `AgentRunner.run()` → 每个事件 SSE 推前端 → 结束存新 sessionId。
- **不写 `tasks.json`**(对话是旁路,不污染任务状态);状态机/worktree/MCP 通道全不动。

### 3.3 sessionId 续接存储

- 存哪:`<dataDir>/task-sessions.json`(`{ [taskId]: sessionId }`),仿 `tasks.json`/`chats.json` 同级。或挂任务 JSON 的扩展字段(但 TaskDTO 改动大,倾向独立文件)。
- 读:发消息前查,有就 `--resume`。
- 写:`result` 事件拿到 session_id 后落盘。

### 3.4 复用(不动)

`tasks.json` 文件协作、四态状态机、git worktree、MCP 通道、SSE 基建、`resolveDataDir`。71015 第七节「加一条通道,不重写」。

## 四、该干什么(步骤)

1. **后端 AgentRunner + 路由**:先做,单独可 curl 验证(`POST /api/tasks/:id/chat` → 看 stream-json 透传)。
2. **隔离 hook 验证**(3.1 子点):跑通后量一次 cache_read,确认 < 1k 才算过关。
3. **前端 `streamTaskChat` + store**:仿 `streamChat`。
4. **`TaskDrawer` 栏3 tab 改造**:预览搬 tab,对话 tab 挂 `TaskConversation`。
5. **事件卡片渲染**:text(复用 MessageContent)/ thinking / tool_use / tool_result 关联。
6. **sessionId 续接**:存 `task-sessions.json`,第二轮验证续接成功。
7. 自测 + 联调。

## 五、不该干什么(红线)

1. **不新建「任务对话」独立视图/侧栏项**。对话挂在 `TaskDrawer` 栏3,看板点卡片进抽屉(已定)。
2. **不改栏1 / 栏2 / 底部按钮 / `StepEditor` / `OpenClaudeDialog`**。改动集中在栏3 + 后端 AgentRunner。
3. **不让对话写 `tasks.json` / 转状态机**。对话是旁路通道;要回写状态仍走 MCP(那条通道不动)。
4. **不代理 Claude 流量 / 不自己实现 LLM 调用**。就是 spawn CLI,透传 stream-json。
5. **不先解决 hook 隔离就上**。spike 证明不隔离每轮白烧 60k token,必须先过 3.1 子点。
6. **不把 result 之外的事件当对话内容**。`system/hook_*`、`thinking_tokens` 必须忽略,否则前端全是噪音。
7. **不搞工具审批 UI / 自动调度 / 多 agent**(71015 第八节明确排除)。
8. **prompt 不进命令行**(走 stdin),避开 shell 转义 + 对齐 stream-json 输入协议。

## 七、风险与未决

- **`--settings` 覆盖语义**:已 spike 验证 `clean.json`(空 hooks/permissions)能把用户级 superpowers 注入从 ~30k 压到 ~0;但 cwd 在项目内时仍会加载项目 `.claude/CLAUDE.md`(对任务有用,保留)。实施后实测一轮 input 确认范围。
- **`ClaudeSessionScanner` 在 GLM 下 assistant.usage=0**:实时流用 result.usage 不受影响;但用量面板的 jsonl 回采可能少算,**需复核**(可能改扫 result 行累加)。列为后续子任务。
- **Windows 路径反斜杠** 在 tool_use input 展示:不强转,原文显示。
- **未派发任务(无 worktree)**:cwd 回退 `repoPath`;都没有则禁用对话 tab + 提示先填仓库/派发。
- **并发**:同一任务连点发送 → 前端禁用输入框直到本轮 `result`(按轮 spawn,天然串行)。
