# 前端 React + Tailwind 重构实施计划

> 日期:2026-06-05
> 目标:用 React 18 + Tailwind CSS 重写前端,对齐现有功能 + 补全缺口(拖拽改状态、Diff 审查闭环、明暗主题、前后端共享类型)

---

## 决策汇总(已与用户确认)

| 项 | 决定 |
|----|------|
| 技术栈 | Vite + React 18 + TS + Tailwind v4 + Zustand + 自建组件 |
| 状态管理 | Zustand 包干(不引 TanStack Query) |
| 拖拽 | @dnd-kit |
| Diff 渲染 | react-diff-view |
| 图标 | lucide-react |
| 范围 | 对齐现有 + 补全缺口 |
| 后端 | 允许小改(补 diff / approve / reject 端点 + 事件 + shared 类型) |
| 主题 | 明暗双主题 |
| Diff 审查 | 完整闭环(派发→回写→看 diff→通过/打回) |
| Vue 旧码 | 直接替换(git 可找回) |

---

## 现状核实结论(基于实际代码)

- 现有 Vue 前端:看板展示 + CRUD + SSE **真实可用**;拖拽、Diff **从未实装**(commit 信息夸大)。
- 后端 `Task.approve(mergeStrategy)` / `reject(reason)` 已存在,但**不发领域事件**(MVP 简化)。
- `WorktreeManager.getDiff(ref, baseBranch='main')` 可复用,但 HTTP server **未注入 WorktreeManager**。
- 后端 HTTP **无 diff 端点**(仅 MCP 有 get_task_diff)。
- 无 `shared/` 包,前端自维护一份类型(漂移风险,审查报告 M-5)。

---

## 阶段一:后端小改(支撑前端闭环)

### Task B1:新增 shared 类型包
- 新建 `shared/`(workspaces 已声明需恢复):`package.json` + `tsconfig.json` + `src/types/task.ts`
- 把 TaskStatus / Priority / Task DTO / ExecutionResult / WorktreeRef DTO 定义在此
- 后端 domain 值对象保持不变,但 DTO/接口契约引用 shared
- 前端直接 import `@ai-task-flow/shared`
- 根 `package.json` 恢复 `shared` workspace + build:shared

### Task B2:审查事件补全
- 新增 `TaskApproved`(含 mergeStrategy)、`TaskRejected`(含 reason)领域事件
- `Task.approve()` / `reject()` 内 push 对应事件(对齐 dispatch/recordResult 模式)
- 单元测试覆盖

### Task B3:HTTP 注入 WorktreeManager + 审查端点
- `createHttpServer` / `startHttpServer` / `http-server.ts` 多注入一个 `WorktreeManager`
- 新增端点(taskRoutes.ts):
  - `GET /api/tasks/:id/diff` → 复用 `WorktreeManager.getDiff`,无 worktree 返回 409/空
  - `POST /api/tasks/:id/approve` → `task.approve(mergeStrategy)`,非 review 返回 400
  - `POST /api/tasks/:id/reject` → `task.reject(reason)`,非 review 返回 400
- baseBranch 问题:getDiff 默认 'main',实际可能 'master' → 端点支持 query `?base=` 或自动探测当前分支
- curl E2E 测试补充(tests/curl/)

## 风险与权衡

| 风险 | 应对 |
|------|------|
| Tailwind v4 与 Vite 集成较新 | 若踩坑回退 v3.4(stable) |
| getDiff baseBranch 'main'/'master' 不确定 | 端点自动探测默认分支 |
| 自建组件工作量 | 只建必需的 7~8 个,不追求完备组件库 |
| diff 需要真实 worktree | 审查视图对无 worktree 任务降级提示 |
| 范围偏大 | 严格按 Task 顺序,每个 Task 可独立验证 + commit |
