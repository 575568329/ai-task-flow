# 依赖安装排坑记录

- **日期**：2026-07-23
- **环境**：Windows 11 / Node v22.22.2 / npm 10.9.7 / 公网环境
- **结论**：两个独立问题叠加（混用 pnpm + lock 锁定内网源），报错信息极具误导性

---

## 一、症状（报错链）

依次出现的错误：

1. backend 启动报 `Cannot find package 'gray-matter'`
2. frontend 报缺 `@radix-ui/*`、`@uiw/react-codemirror`、`mermaid`、`rehype-raw` 等一堆依赖
3. shared `tsc` 报 39 个 `TS2688: Cannot find type definition for 'd3' / 'aria-query' / 'chrome' / 'react-syntax-highlighter' / ...`
4. `npm install` 报 `Exit handler never called! This is an error with npm itself`
5. 删 lock 后再装，报 `Cannot read properties of null (reading 'name')`

---

## 二、根因（两个独立问题叠加）

### 根因一：混用 pnpm

操作顺序：`npm i` → 切镜像 → `pnpm i`。pnpm 输出的警告直接挑明了问题：

```
[WARN] The "workspaces" field in package.json is not supported by pnpm.
[WARN] Moving @types/react that was installed by a different package manager to "node_modules/.ignored"
Packages: +32   ← 只装了根目录 3 个 devDependencies
```

pnpm 不认 npm 的 `workspaces` 字段，**shared/backend/frontend 子包依赖一个都没装**。同时 pnpm 把 npm 原本扁平提升到顶层的 `@types/*`（来自 `mermaid→d3`、`react-syntax-highlighter`）打残并塞进 `.ignored`，TS 仍把它们当「隐式类型库」加载却找不到完整声明 → 39 个 `TS2688`。

### 根因二：`package-lock.json` 锁定科大讯飞内网源

`package-lock.json` 中各包的 `resolved` 字段指向科大讯飞内网 Artifactory：

```
https://depend.iflytek.com/artifactory/api/npm/npm-repo/.../xxx.tgz
```

实测 **384 处**。npm install 时优先按 lock 里的 `resolved` URL 下载，公网环境全部 `ENOTFOUND`，重试耗尽后抛出 `Exit handler never called!`。

> 注意：`npm config get registry` 显示的是 `npmmirror`，但 npm **不会因为 registry 不同而自动改写已有 lock 里的 `resolved`**，lock 里的内网地址会覆盖 registry 设置。

中途失败的 install 还会在 `node_modules` 留下损坏的半成品，导致再次安装时 npm 在 `@npmcli/arborist` 的 `buildDeps` 阶段读到 `name` 为 null 的残缺包 → `Cannot read properties of null (reading 'name')`。

---

## 三、错误信息对照表（避坑关键）

报错信息与真因完全不对应，是这次最容易踩的陷阱：

| 报错信息（误导） | 真实原因 |
|------|------|
| `Cannot find 'gray-matter'` / 缺 `@radix-ui/*` | 混用 pnpm，子包依赖没装 |
| shared `TS2688: Cannot find type definition for 'd3'` 等 39 个 | pnpm 污染 node_modules，顶层 `@types/*` 残缺 |
| `Exit handler never called!` | lock 含内网源 `resolved`，公网 ENOTFOUND |
| `Cannot read properties of null (reading 'name')` | 上次失败 install 留下的脏 node_modules 残留 |

---

## 四、标准修复流程（公网环境）

```bash
# 1. 确认 registry 指向淘宝镜像
npm config get registry          # 应为 https://registry.npmmirror.com

# 2. 彻底清理：所有 node_modules / lock / dist
find . -maxdepth 3 -name "node_modules" -type d -prune -exec rm -rf {} +
find . -maxdepth 3 \( -name "package-lock.json" -o -name "pnpm-lock.yaml" \) -not -path "*/node_modules/*" -delete
find . -maxdepth 2 -name "dist" -type d -prune -exec rm -rf {} +

# 3. 清 npm 缓存（清掉脏状态）
npm cache clean --force

# 4. 重装（npm 会用 npmmirror 重新解析，生成干净的 lock）
npm install
```

本次实测：`added 788 packages in 36s`，shared `tsc` 零错误，backend/frontend 依赖齐全。

---

## 五、预防措施

1. **统一用 npm**，禁止 pnpm / cnpm / yarn（详见 `.claude/CLAUDE.md` 第十节）。
2. **非内网环境装前删 lock**：`rm package-lock.json`。
3. **项目根加 `.npmrc`**（内容 `registry=https://registry.npmmirror.com`）并提交，保证多设备 registry 统一。
4. **快速自检**：`grep -c "depend.iflytek.com" package-lock.json`，结果 > 0 即为内网源 lock。

---

## 六、安全提醒

若 `~/.npmrc` 中含明文 npm authToken（`//registry.npmjs.org/:_authToken=...`），注意不要随对话日志或截图外泄；必要时到 [npmjs.com](https://www.npmjs.com) 账户设置撤销并重新 `npm login` 轮换。
