| 项目 | 一句话定位 | 来源 |
|------|-----------|------|
| **AI Task Flow**(本项目) | 个人 AI 任务编排看板 + MCP Server,看板派发 → worktree 隔离 → Claude Code 经 MCP 拉取回写 | — |
| **Vibe Kanban** | 面向多 AI coding agent 的并行编排看板,每任务独立 worktree | [GitHub](https://github.com/BloopAI/vibe-kanban) |

## 三、各竞品要点

### Vibe Kanban(最直接的竞品)
- **定位**:"Get 10X more out of Claude Code, Codex or any coding agent",并行跑多个 agent,每任务一个 worktree,内置 code review + MCP/IDE 集成。([官网](https://vibe-kb.com/))
- **机制差异**:**嵌入式**——由 Vibe Kanban 直接启动/管理 agent 进程(Claude/Cursor/Copilot/Gemini CLI)。"each agent gets its own Git worktree, zero conflicts" 被评测称为 killer feature。([VirtusLab 评测](https://virtuslab.com/blog/ai/vibe-kanban))
- **技术栈**:Rust 后端 + React 前端,SQLite。YC 背书。
- **现状**:母公司 BloopAI 于 **2026-04-10 宣布关停**,项目转为 Apache 2.0 社区维护。([Nimbalyst](https://nimbalyst.com/blog/vibe-kanban-after-bloop-whats-next/))
- **与本项目关系**:机制最像(看板+worktree+多 agent),但**它代起 agent,本项目让用户在自己的终端用原生 Claude Code**——这是 v3 设计刻意的取舍(保留原生体验、用户完全掌控 AI 交互)。

### Claude Task Master
- **定位**:AI 项目经理替代品,把复杂需求/PRD 拆解为结构化任务,供 Cursor/Windsurf/VS Code/Claude Code 等通过 MCP 消费。([npm](https://www.npmjs.com/package/task-master-ai))
- **机制差异**:**专注任务拆解**,无 web 看板、无 worktree 隔离。是"上游"——它解决"任务从哪来",本项目解决"任务如何派发执行+可视化追踪"。
- **互补性**:理论上可组合——Task Master 拆解 → 导入本项目看板 → worktree 派发。

### Backlog.md
- **定位**:git 生态内 人/AI 协作的 markdown 任务管理,终端原生。([GitHub](https://github.com/MrLesk/Backlog.md))
- **机制差异**:任务即 **markdown 文件**(随 git 版本化),CLI 优先 + 内置 web/TUI 看板,无 worktree 隔离。
- **差异点**:存储哲学不同——它把任务存进仓库(git 可追溯),本项目任务存在仓库外(`~/.ai-task-flow/`),用 worktree 隔离代码改动。

## 四、本项目的差异化定位

### 独特卖点
1. **MCP 拉取式 + 原生 Claude Code 体验**
   竞品(尤其 Vibe Kanban)是"工具代起 agent";本项目是"agent(用户的 Claude Code)主动经 MCP 拉取任务"。用户始终在自己的终端、用原生 Claude Code,工具不抢夺 AI 交互控制权。这是 v3 相比 v2(嵌入式 Agent SDK)的核心转向。

2. **worktree 隔离 + 看板可视化 双管齐下**
   Task Master / Backlog.md 都没有 worktree;有 worktree 的 Vibe Kanban 又是嵌入式。本项目是少数把"worktree 隔离 + web 看板 + MCP 拉取"三者结合的。

3. **DDD + 事件驱动的可演进架构**
   EventBus + EventStore + 四层分层,为后续扩展多 Bounded Context 预留。同类工具多为 CLI 优先或单体。

### 劣势 / 差距(诚实评估)
| 短板 | 对标 | 说明 |
|------|------|------|
| 仅支持 Claude Code | Vibe Kanban 多 agent | 当前未接 Codex/Gemini/Cursor |
| 无 AI 任务拆解 | Task Master | 任务靠人工录入 |
| JSON 文件存储 | Vibe Kanban SQLite | 无事务/并发控制(见审查报告 L-3) |
| 成熟度 | 竞品均有社区/融资 | 本项目为 MVP,功能与稳定性待打磨 |
| 多 agent 并行编排 | Vibe Kanban | 暂为单任务派发,无并行调度面板 |

---

## 五、结论

**市场位置**:本项目处在 "Vibe Kanban × 原生体验" 的细分点上——
- 比 **Task Master / Backlog.md** 多了 worktree 隔离 + 真正的 web 看板;
- 比 **Vibe Kanban** 少了多 agent 与成熟度,但换来"不嵌入、保留原生 Claude Code 控制权"的体验。

**最值得借鉴**:Vibe Kanban 的多 agent 抽象与并行编排面板,以及 Task Master 的 AI 任务拆解——二者恰好对应本项目当前两个最大短板(单 agent、人工录入)。

**机会点**:Vibe Kanban 母公司已关停转社区维护,其"嵌入式"路线的用户中,部分人正是因为"想保留原生 agent 体验"而流失——这正是本项目的目标人群。
