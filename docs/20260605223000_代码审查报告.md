## 二、问题清单(按严重程度)

### 🔴 高(建议尽快处理)

无阻断性高危问题。以下一条接近高优,因涉及网络暴露:

**H-1:HTTP 服务默认监听 `0.0.0.0` 且无任何鉴权**
- 位置:`src/http-server.ts:15-16`
- 现象:`host: '0.0.0.0'`(所有网卡)+ 全部 API 无鉴权
- 影响:同一局域网内任何人可读写/删除任务。对"个人本地工具"定位而言是过度暴露
- 建议:默认改为 `127.0.0.1`(仅本机);若需局域网访问再显式开启,并加简单 token

### 🟡 中(建议规划处理)

**M-1:状态机转换表 `isValidTransition` 是死代码**
- 位置:`src/domain/workflow/value-objects/TaskStatus.ts:11`
- 现象:精心设计的 6 状态转换表,**全项目无任何调用方**。聚合根 dispatch/recordResult 用各自硬编码的 `if` 检查
- 影响:状态流转校验逻辑分散且不统一;转换表形同虚设
- 建议:让 dispatch/recordResult/applyUpdate 统一走 `isValidTransition`,或删除该表避免误导

**M-2:`applyUpdate` 绕过状态机,允许任意状态跳转**
- 位置:`src/domain/workflow/entities/Task.ts`(applyUpdate)
- 现象:HTTP PATCH 可把任务从任意状态改到任意状态(如 done → todo),不校验合法性
- 影响:看板拖拽灵活,但可能产生业务上非法的状态(如跳过 review 直接 done)
- 备注:这是有意识的 MVP 权衡(代码注释已标注),非疏漏。记录在案

**M-3:CORS `origin: '*'` 与 `credentials: true` 组合无效**
- 位置:`src/interfaces/http/server.ts:27-29`
- 现象:浏览器规范禁止 `credentials: true` 搭配通配 origin,该组合实际不生效
- 影响:当前无 cookie 鉴权,暂无实际危害,但配置语义错误,埋坑
- 建议:本地工具去掉 credentials,或把 origin 收敛为具体来源

**M-4:HTTP 路由无 schema 校验**
- 位置:`src/interfaces/http/routes/taskRoutes.ts`
- 现象:POST/PATCH body 无 Fastify JSON Schema 校验,title/description 无长度上限
- 影响:可写入超长字段;非法 `prefix`(如小写)触发 TaskId 构造异常 → **500 而非 400**
- 建议:为路由加 Fastify schema(Fastify 原生支持),非法输入返回 400

**M-5:前后端类型定义双份维护(漂移风险)**
- 位置:`backend` 无 `shared/types`;`frontend/src/types/task.ts` 独立定义一份
- 现象:Task/TaskStatus/Priority 等在前后端各写一遍
- 影响:后端改字段,前端不会编译报错,易静默漂移
- 建议:抽到 monorepo 的 `shared/` 包,前后端共享(package.json 已声明 shared workspace)

**M-6:application 层为空,路由直连 repository**
- 位置:`src/application/{workflow,mcp}/` 均为空目录
- 现象:DDD 四层中应用层无任何用例(UseCase),HTTP/MCP 直接操作 repository + 聚合根
- 影响:业务编排逻辑散落在接口层;未来多入口(HTTP+MCP)会重复逻辑
- 建议:把"创建任务""派发任务"等提取为 UseCase,HTTP/MCP 共用

### 🟢 低(可选优化)

**L-1:MCP handler 入参 `args: any`(5 处)**
- 位置:`src/interfaces/mcp/server.ts` 各 handler
- 现象:inputSchema 已声明 JSON Schema,但 handler 仍 `args: any`,schema 与 TS 类型未联动
- 建议:用 zod 定义 schema 并 `z.infer` 出类型,handler 入参强类型化

**L-2:`catch (error: any)`(4 处)**
- 现象:TS 早期惯例,应为 `catch (error: unknown)` + 类型守卫
- 建议:统一改 unknown,收窄后再访问 .message/.code

**L-3:ID 生成是"查最大序号+1",非并发安全**
- 位置:`src/interfaces/http/routes/taskRoutes.ts:59-64`
- 现象:findAll → max+1。并发 POST 可能产生重复序号(JSON 文件存储无事务)
- 影响:单用户本地场景几乎不会触发;高并发会冲突
- 建议:MVP 可接受;未来换 DB 时用自增/UUID

**L-4:前端 bundle 1.2MB 未分包**
- 现象:Element Plus 全量打包,index.js 1.2MB(gzip 393KB)
- 建议:按需引入 Element Plus + 路由懒加载

## 五、优先级建议

若要从 MVP 走向"可分享/可托管",建议顺序:
1. **H-1**:监听地址收敛到 127.0.0.1(一行改动,显著降风险)
2. **M-4**:路由加 schema 校验(消除 500、防超长输入)
3. **M-5**:前后端共享类型(防漂移)
4. **M-1/M-2**:统一状态机校验
5. **M-6**:补 application 层 UseCase

L 级问题可在重构期顺手处理。
