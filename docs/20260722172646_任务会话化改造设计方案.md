## 一、背景与要解决的问题

### 1.1 现状问题

当前「任务派发」是一个重型状态机动作，`POST /api/tasks/:id/dispatch` 会：建 git worktree → 落盘任务 md → 生成 `/rename` 指令 → 自动开终端跑 `claude` → 把任务推入 `TODO→DISPATCHED→REVIEW→DONE` 流水线（含 approve/reject/diff/合并审查）。

它与实际工作流冲突：

1. **resume 不到历史**：用户在 `repoPath` 用 claude 积累了大量历史对话，想「接着聊」，但 `claude --resume <id>` 必须在**会话原启动目录**执行；worktree 是另一个目录，resume 失败。
2. **隔离/审查用不上**：worktree 隔离、approve/reject/合并审查这套重型流程对个人任务编排是过度设计，徒增复杂度。
3. **指令双轨**：后端拼 `claudeCommand`、前端 `buildClaudeCodePrompt` 两套并存，易不一致。
4. **状态机臃肿**：`DISPATCHED/REVIEW` 两个状态只服务于 worktree 审查流，删除审查流后即成死状态。

### 1.2 目标（要做成什么样）

回归 **repoPath 连续会话**模型：

1. **打开对话**：任务关联 `repoPath` 后，枚举该路径下的 Claude 历史会话（cmd/wsl 两套历史自然分离），选一个 resume 进入，或新建。
2. **三 shell 支持**：`cmd.exe` / `wsl.exe` / `pwsh.exe` 均可打开。
3. **指令传递**：打开对话时自动复制执行指令到剪贴板；claude 侧通过本项目自带的 **MCP 工具主动拉取任务**（我们的独有能力）。
4. **状态机瘦身**：收敛为 `TODO / DONE / BLOCKED` 三态。
5. **清除死代码**：删除 worktree 隔离、审查流及其连带的值对象/事件/接口/工具。

### 1.3 本次不做（边界，避免无效编码）

- **内嵌终端**（node-pty + xterm.js）：中重型新增，列为 Phase 2 可选，不在本次。
- **Claude Code hooks 驱动状态自动流转**：列为 Phase 3 探索，本次状态仍由人工/MCP 显式变更。
- **BM25 会话全文搜索**：本次只做枚举 + 标题展示；搜索留待有内嵌终端后一并做。
- **fork / checkpoint / 会话迁移**：不做。
- **任务文档预览复用 KnowledgeViewer 工具栏**：独立遗留项，本次不碰。

## 二、解决方案总览

### 2.1 用什么解决什么

| 要解决的问题 | 用什么解决 | 这个东西是什么 / 能做什么 |
|---|---|---|
| 找到 repoPath 下的历史会话 | **ClaudeSessionScanner**（新增后端模块） | 扫描 `~/.claude/projects/<编码cwd>/*.jsonl`，流式解析出 sessionId/标题/最后活跃时间/消息数。`<编码cwd>` 由 repoPath 经编码规则得出，cmd 与 wsl 自然产生两个不同目录，分别枚举。 |
| 把 Windows 路径编码成 claude 的目录名 | **路径编解码工具**（`encodeProjectPath`/`decodeDirName`） | 规则：`\`→`/`，`/`与`.`→`-`。`C:\proj`→`C--proj`，`/mnt/c/proj`→`-mnt-c-proj`。Scanner 与 TerminalLauncher **必须共用同一套**，否则 resume 找不到。 |
| 选择并打开一个会话（resume / 新建） | **TerminalLauncher 扩展**（新增 openClaude） | 按 env 生成启动命令并唤起新终端窗口：cmd `cd /d C:\p && claude --resume <id>`；wsl `wsl.exe --cd /mnt/c/p -- claude --resume <id>`；pwsh `cd C:\p; claude --resume <id>`。新建不带 `--resume`。 |
| 把任务指令传给 claude 执行 | **MCP 工具主动拉取 + 剪贴板兜底** | 项目自带 MCP server，claude 启动后调 `list_pending_tasks` / `get_task` 主动拉取任务详情与指令；打开对话时同步把指令写入剪贴板，作为未配置 MCP 时的兜底。**不依赖任何进程注入。** |
| 三 shell 选择 | **per-task 的 `env` 字段**（task 级偏好） | 每个任务记录自己偏好的执行环境，TaskDrawer 里选；不在弹窗里每次问。 |
| 状态机臃肿 | **三态状态机**（TODO/DONE/BLOCKED） | 删除只服务于审查流的 DISPATCHED/REVIEW；开终端≠进行中，`IN_PROGRESS` 无可靠触发点，YAGNI。 |
| 派发强绑 worktree | **解耦：保留 worktree 能力，删派发强制耦合** | `WorktreeManager`/`WorktreeRef` 作为可选隔离能力**保留**（已写好可用，删了重写浪费）；删的是「派发即建 worktree」的强耦合 + 审查流。`Task.worktree` 降级为可选关联（与 `sessionLink` 并列）。worktree 仅用于「新建会话隔离」，与 resume 历史（须在 repoPath，cwd 不一致）互斥 |

### 2.2 关键技术选型与能力边界（避免误判）

- **本次不引入 tmux，不引入 node-pty**。MVP 的「打开对话」= 唤起外部终端窗口 + 复制指令 + MCP 拉取，三者均零 native 依赖、纯 Node/TS。
- 「向运行中 claude 主动注入消息」**本次不做**（需 node-pty 或 tmux 才可实现）。当前形态是「claude 主动拉（MCP）」而非「看板主动推（注入）」，足够覆盖核心场景。
- 三 shell 全部支持，但**默认按 task.env**；cmd/pwsh 跑 claude 时 cwd 若为 Windows 路径，历史会与 cmd 落在同一编码目录（共享），wsl 独立一套。

### 2.3 核心决策（拍板项，可否决）

| # | 决策 | 选择 | 理由 |
|---|---|---|---|
| D1 | 状态机 | `TODO / DONE / BLOCKED` 三态 + 可选 `activityState` 运行时字段 | 三态管任务生命周期；活动态（idle/working/waiting）是独立观测维度，由 Phase 3 hooks 填充、驱动卡片角标。不把高频运行时态塞进状态机（活动 ≠ 生命周期） |
| D2 | worktree | **保留能力、解耦强绑**：WorktreeManager/WorktreeRef 不删；删派发强制耦合 + 审查流；`Task.worktree` 降为可选关联 | 能力已写好，删了重写是浪费；worktree 仅用于「新建会话隔离」，与 resume 历史（须在 repoPath，cwd 不一致）互斥。Phase 1 不暴露建 worktree UI，未来接「隔离模式」开关即可复用 |
| D3 | 任务↔会话关联 | Task 增加可选 `sessionLink: {sessionId?, openedAt?}`（env 归 D6 的 task.env，不重复） | 轻量记录「这个任务关联/上次在哪个会话执行」，可选不强制，不构成状态流转 |
| D4 | 指令传递 | MCP 拉取为主 + 剪贴板兜底；**统一一套指令文本**（替代后端 claudeCommand 与前端 buildClaudeCodePrompt 双轨） | 我们是 MCP server，主动拉最优雅；双轨合并消除不一致 |
| D5 | 审查流 | **全删**（事件/方法/接口/按钮/diff 工具） | 依赖 worktree + REVIEW，连锁删除 |
| D6 | 三 shell | per-task `env` 字段（cmd/wsl/pwsh） | 不全局、不每次问；TaskDrawer 选一次记住 |
| D7 | 数据迁移 | `DISPATCHED/REVIEW`→`TODO`，清空 worktree，保留 executionResult；幂等 | 不丢任务，已做结果留作历史 |
| D8 | 实施顺序 | **Phase 1 打开对话（可独立交付）→ Phase 2 删派发** | 核心能力最快可用、可回退；删除是清理 |

## 三、编码与实现指导意见

基于代码现状与 Claude Code 会话机制的实战约束，写代码时遵循：

1. **路径编码必须统一、单一来源**。`encodeProjectPath` / `decodeDirName` 放在 shared 或 backend 的独立纯函数模块，Scanner、TerminalLauncher、迁移逻辑全部引用同一份。**禁止各处自行 replace 字符串**——cmd/wsl 双目录一旦编码不一致，resume 必然找不到会话，且错误不可见。

2. **jsonl 流式解析，early exit**。用 `fs.createReadStream` + `readline`，凑够「首条 user 消息 + 消息数≥5」即可停，**不全量读取**。会话文件可能数十 MB，全量读会卡枚举。

3. **只读 summary 行 / 首条 user，不解析 assistant 正文**。jsonl 是 Claude Code 内部格式、非稳定契约；只取稳定字段（type/message/cwd/timestamp）。标题取首条 user 文本截断。

4. **指令传递优先 MCP，不要去做进程注入**。本项目核心定位是 MCP server，claude 启动后主动调 `get_task` 拉任务是最自然的传递方式。剪贴板是未配 MCP 的兜底。任何「往运行中终端写字」的念头都应先问：能不能让 claude 自己拉？

5. **env→启动命令用策略映射（Map），不用 if-else 链**。三个 shell 的启动命令差异是典型的 `>3 分支`，写成 `{ cmd: ..., wsl: ..., pwsh: ... }` 映射，新增 shell 只加一行。

6. **状态机三态，开终端不改状态**。「打开对话」是动作不是状态——打开后任务仍在 TODO，直到 claude 回写 `record_result`（DONE/BLOCKED）才变。避免「打开=进行中」这种无可靠触发点的伪状态。

7. **数据迁移放反序列化处、幂等**。在 `JsonTaskRepository` 加载 tasks.json 时做一次性迁移，已迁移的不再动；迁移逻辑用卫语句尽早返回。

8. **native 依赖本次零引入**。不装 node-pty / xterm，避免触发「WSL 不跑 Linux node、装包走 Windows」的约束。Phase 2 引入内嵌终端时再单独处理装包。

9. **复用优先，最小改动面**。TaskDrawer 三栏布局、Sheet、MessageContent、StepEditor、systemApi.selectDirectory 全部复用；只改底栏按钮、加一个 env 选择器、加一个会话选择弹窗。不要为本次需求重构抽屉结构。

### 4.4 新增模块设计

#### 4.4.1 ClaudeSessionScanner（新增 `backend/.../system/ClaudeSessionScanner.ts`）

职责：给定 repoPath + env，返回该路径下的 Claude 历史会话列表。

```ts
// 接口签名（设计优先）
export interface ClaudeSessionMeta {
  sessionId: string;
  title: string;          // 首条 user 文本截断
  cwd: string;            // 从 jsonl 解析的真实 cwd
  lastActiveAt: string;   // ISO，取文件 mtime
  messageCount: number;
}

export class ClaudeSessionScanner {
  // env 决定从哪套编码目录读：cmd→Windows profile；wsl→wsl home
  listSessions(repoPath: string, env: 'cmd' | 'wsl' | 'pwsh'): Promise<ClaudeSessionMeta[]>;
}
```

实现要点：
- 用 `encodeProjectPath(repoPath)` 算出编码目录名；env 决定根：cmd/pwsh → `%USERPROFILE%\.claude\projects`；wsl → `\\wsl.localhost\<distro>\home\<user>\.claude\projects`（distro/user 实施时探测或配置）。
- 每个匹配目录下 `readdir` 取 `*.jsonl`，流式解析每文件前若干行，提取 sessionId（文件名）、首条 user 文本、cwd、timestamp；messageCount 计数。
- 照搬 `system.ts` select-directory 的编码安全范式（不引入新坑）。
- 纯函数 `encodeProjectPath`/`decodeDirName` 抽到独立文件，Scanner 与 TerminalLauncher 共用。

#### 4.4.2 TerminalLauncher 扩展（`backend/.../system/TerminalLauncher.ts`）

现状只有 `openAndRunClaude(projectPath)`（win32 `start cmd /k claude`）。扩展为：

```ts
export interface OpenClaudeOptions {
  repoPath: string;
  env: 'cmd' | 'wsl' | 'pwsh';
  sessionId?: string;     // 有则 resume，无则新建
}

export class TerminalLauncher {
  // 按策略映射生成启动命令并唤起新窗口；返回用于剪贴板的 claudeCommand 文本
  static openClaude(opts: OpenClaudeOptions): Promise<{ claudeCommand: string }>;
}
```

策略映射（D6 三 shell）：

```ts
const SHELL_LAUNCHERS = {
  cmd:  (repoPath, resume) => `start "Claude" cmd /k "cd /d "${repoPath}" && claude ${resume ? `--resume ${resume}` : ''}"`,
  wsl:  (repoPath, resume) => `start wsl.exe --cd "${toWslPath(repoPath)}" -- claude ${resume ? `--resume ${resume}` : ''}`,
  pwsh: (repoPath, resume) => `start pwsh.exe -NoExit -Command "cd '${repoPath}'; claude ${resume ? `--resume ${resume}` : ''}"`,
};
```

> 实施时本机实测三条命令的行为再定稿（尤其 wsl `--cd` 与 `start` 组合、pwsh 的引号转义）。

#### 4.4.3 Task 聚合根调整

- 删 `dispatch()`/`approve()`/`reject()` 三方法（派发 + 审查流）；`recordResult` 改轻量版（不校验 DISPATCHED）。
- **保留 `worktree` 字段**（从派发强制副产物 → 降为可选关联，语义见 D2）。

## 五、前端设计（按交付阶段）

> **核心原则**：Phase 1 前端一次性到**终态界面**（用户看到即最终样子）；Phase 2 只清后端死代码，前端界面不再变动。复用现有三栏抽屉与全部 ui 原语，**不重构抽屉结构**。
>
> **样式约束**：颜色全用现有 CSS 变量（`--primary-6` / `--warning-6` / `--success-6` / `--error-6` / `--grey-6`，见 `taskMeta.ts:12-18`），禁止硬编码；间距/圆角复用 Tailwind 默认刻度。

### 5.0 两个交付态总览

| 维度 | Phase 1 交付（前端到终态） | Phase 2 交付（后端清理，前端不动） |
|---|---|---|
| 看板列 | `BOARD_COLUMNS` **4列→3列**（待办/已完成/已阻塞） | 不变 |
| TaskDrawer 底栏 | 删派发+审查三按钮；加「打开对话」+「复制执行指令」 | 不变 |
| env 选择器 + OpenClaudeDialog | 新增 | 不变 |
| shared `TaskStatus` 类型 | **不动**（保留 DISPATCHED/REVIEW，兼容后端返回） | 收敛为 3 态 |
| `STATUS_LABELS/COLORS` 映射 | **保留** DISPATCHED/REVIEW（防 `undefined` 崩渲染） | 删残留 |
| 后端派发/审查流 | **不动**（接口/状态机/事件全保留） | 全删 + 数据迁移 |

> 为什么前端能先到终态：前端删的是「按钮 + API 调用点」，后端 `/dispatch` `/approve` 等接口虽在但无人调用 = 安全死接口，Phase 2 统一清。Phase 1 期间旧的 DISPATCHED/REVIEW 任务不显示在看板列，Phase 2 迁移回 TODO。

### 5.3 OpenClaudeDialog（Phase 1 新增）

点「打开对话」触发。布局：

```
┌─ 选择 Claude 会话 ──────────────────────────┐
│ [🔍 搜索（Phase 2 接 BM25，本次占位禁用）]  │  顶部
├──────────────────────────────────────────────┤

├──────────────────────────────────────────────┤
│ 历史会话（按最后活跃倒序）        ScrollArea  │
│ ┌────────────────────────────────────────┐  │  max-h-[50vh]
│ │ ◉ 修复登录页样式         [wsl] 2分钟前  │  │  RadioGroup 单选
│ │    · 12 条消息  /mnt/c/proj             │  │
│ └────────────────────────────────────────┘  │
│ ┌────────────────────────────────────────┐  │
│ │ ○ 添加导出功能           [cmd] 昨天     │  │
│ │    · 34 条消息  C:\proj                 │  │
│ └────────────────────────────────────────┘  │
│   (cmd/wsl 历史混排，env 徽标区分)          │
├──────────────────────────────────────────────┤
│                       [取消] [打开并复制指令] │  底部
└──────────────────────────────────────────────┘
```

**交互态**（防拉胯的关键，每态都要有明确样式）：

| 态 | 样式与行为 |
|---|---|
| 加载中 | 列表区 `Loader2` 转圈 + 「正在枚举会话…」；调 `systemApi.listClaudeSessions(repoPath, task.env ?? 'cmd')` |
| 空 | 列表区「未找到历史会话，点上方新建」+ 三新建按钮 `ring` 高亮引导 |
| 选中 | RadioGroup 单选，默认不选中（强制选历史或新建，防误开）；选中项 `bg-primary/10` |
| 确认 | 选历史 → `openClaudeSession({sessionId})`；点新建 → `openClaudeSession({env})`；成功 toast「已打开终端，执行指令已复制」 |
| 错误 | 接口失败 → toast 报错，弹窗不关（可重试） |

**token**：env 徽标 `cmd → var(--grey-6)` / `wsl → var(--warning-6)` / `pwsh → var(--primary-6)`（Badge 变体）；弹窗 `max-w-2xl`，列表 `ScrollArea max-h-[50vh]`。

> 复用原语：`Dialog`/`DialogContent`、`RadioGroup`、`ScrollArea`、`Badge`、`Button`、`Loader2`、`toast`。二级浮层注意与图片预览(YARL)的 outside-click 对齐（参考 TaskDrawer 现有 `previewOpen` 守卫）。

### 5.4 交互流程

```
用户点「打开对话」
  → OpenClaudeDialog 加载历史(repoPath, task.env ?? 'cmd')
  → 用户选历史 or 点新建(cmd/wsl/pwsh)
  → POST /api/system/claude-sessions/open
  → 后端 TerminalLauncher.openClaude 唤起新终端(claude --resume 或新建)
  → 返回 claudeCommand → 前端写剪贴板 + toast
  → (可选) 写入 task.sessionLink
  → claude 启动后用户在终端调 MCP get_task 拉任务执行
```

「终端已开着不想新开」→ 直接用「复制执行指令」按钮，去已有对话框粘贴。

### 5.5 状态管理与 API 层（Phase 1）

- `api/task.ts`：**删** `dispatch/getDiff/approve/reject`；`systemApi` **加** `listClaudeSessions(repoPath, env)`、`openClaudeSession(body)`。
- `stores/taskStore.ts`：**删** `dispatch` action 及接口声明。
- SSE 刷新链路不变（`TaskUpdated` 仍驱动）。

### 5.6 设计 token 与样式 checklist（防拉胯）

- **颜色**：全用 CSS 变量（`--primary-6` 等），env 徽标/按钮主次/状态色均不硬编码十六进制。
- **按钮层级**：打开对话(主色 sm) > 复制指令(outline) > 保存(默认) > 删除(ghost destructive)。
- **三态覆盖**：每个异步列表（会话枚举）必须有 加载/空/错误 三态，不留白屏。
- **禁用态**：无 repoPath 时「打开对话」禁用 + tooltip 说明，不能点了无反应。
- **间距/圆角**：复用现有 Tailwind 间距刻度与组件默认圆角，不引入新值。

## 八、风险与对策

| 风险 | 对策 |
|---|---|
| 删派发/审查流链路漏改某处 → 编译错 | 按 §六清单逐项；删完 `tsc --noEmit` 全量查 |
| WorktreeManager 保留但当前无业务调用方（待接线能力） | 加注释标明用途；Phase 2「隔离模式」UI 接入时 resolve 使用；若长期无隔离需求，下个迭代再清理 |
| 旧 `tasks.json` 含 DISPATCHED/worktree → 加载崩 | §4.7 迁移幂等兜底 |
| MCP 删 `get_task_diff` 后 claude 仍调 → unknown tool | 工具声明同步删；提示用户更新 claude 的 MCP 工具列表 |
| cmd/wsl/pwsh 路径编码或启动命令不一致 → resume 失败 | pathCodec 单一来源；三 shell 启动命令本机实测后再定稿 |
| wsl `--cd`+`start`、pwsh 引号转义行为未知 | Phase 1 先本机逐条手测命令，定稿再写进策略映射 |
| cmd/pwsh 共享同一编码目录 → 历史混在一起难区分 | 列表项用 cwd + env 徽标区分；不强行合并 |
| TerminalLauncher 唤起新窗口在某些 shell 失败 | openClaude 失败时降级为「仅返回指令到剪贴板」并 toast 提示手动开终端 |
