# 知识库写入 / Agent 调研落库 / 划词翻译 技术方案

> 日期:2026-07-09 · 状态:**待评审** · 作者:协作产出

## 一、背景与目标

三个需求经调研发现 **①② 共享同一堵点**:后端知识库目前**只能读+删,没有写入能力**;③ 完全独立(扩展侧)。

目标:产出可评审的接口契约 + 文件清单 + 任务拆解,评审通过后进入实现。

设计原则:
- **"守规则"靠工具/服务端强制,不靠提示词约束 LLM**(防御式编程);
- **更优实现优先**:在成本可控前提下采用业界推荐做法(MCP Resources 双暴露、Chrome Translator API、floating-ui);
- 操作路径最短化、复用系统原生能力。

## 四、共享基础:后端写知识库能力(P0)

### 4.1 REST 接口契约

**创建文档**
```
POST /api/knowledge/doc
Body: { title: string; content: string; tags?: string[]; dir?: string }
成功 201: { path: string }            // 相对 knowledge-base/ 的 posix 路径
错误 400: { error }  // 标题非法/为空
错误 403: { error }  // dir 路径越界
```
- 文件名**由服务端生成**,调用方不传 → 命名规则集中受控。

**更新文档(覆盖写)**
```
PUT /api/knowledge/doc?path=xxx
Body: { content: string }
成功 200: { ok: true, path: string }
错误 403: { error }  // 路径越界
错误 404: { error }  // 文件不存在
```

### 4.2 KnowledgeService 新增方法
```ts
async createDoc(input: { title: string; content: string; tags?: string[]; dir?: string }): Promise<{ path: string }>;
async saveDoc(relPath: string, content: string): Promise<{ path: string }>;
```
两者写完均调 `refreshManifest()`,保证前端 manifest 即时一致。**原子写**:先写临时文件再 `rename`,防中途崩溃产生半截文件。

### 4.3 命名规则强制(核心设计)
```
文件名 = <YYYYMMDDHHMMSS>_<sanitizedTitle>.md
```
- 时间戳服务端生成(本地时区,与 Scanner 展示一致),与 `stripPrefix` 兼容;
- `sanitizeTitle`:去文件系统非法字符(`/ \ : * ? " < > |`)、压缩空白、长度截断(≤ 60 字);
- frontmatter `---\ntags: [..]\n---\n` 由服务端按 `tags` 拼装;
- **调用方(前端 / MCP / agent)只传语义字段,物理文件名无法干预** → 想违反命名规则也违反不了。

### 4.4 shared 类型(`shared/src/types/knowledge.ts` 新增)
```ts
export interface KnowledgeCreateRequest { title: string; content: string; tags?: string[]; dir?: string; }
export interface KnowledgeSaveRequest { content: string; }
export interface KnowledgeDocWriteResponse { path: string; }
```

## 五、模块 A:前端 md 编辑(问题①)

### 5.1 交互
- `KnowledgeViewer` 对 `kind==='md'` 增加「编辑/预览」切换;编辑态 `Ctrl+S` → `PUT /api/knowledge/doc?path=...`。
- 顶部「新建笔记」按钮 → 标题输入 → `POST /api/knowledge/doc` → 打开新文件编辑态。
- 保存成功 toast + 刷新 manifest(`KnowledgeTree` 重拉)。

### 5.2 编辑器:CodeMirror 6(及「为何不用 WYSIWYG」)
采用 **CodeMirror 6**(`@codemirror/lang-markdown` + 主题):md 高亮/行号/折叠,基础 tree-shake 后约 50KB gzip、带 markdown 约 80–130KB,远轻于 Monaco(2.4MB)。

> **为何不用 Milkdown / Tiptap 等 WYSIWYG**:本场景 md 由 **agent(MCP)与人共同读写**,且含 frontmatter + `[[wiki]]` 双链(非标准 md)。WYSIWYG 解析 md 时可能改写/破坏这些自定义语法;CodeMirror 把 md 当纯文本,**零破坏、保真、agent 友好**。仅当未来「人工撰写长笔记」成为主场景,再评估 Milkdown。

### 5.3 文件清单
| 文件 | 改动 |
|------|------|
| `frontend/src/components/knowledge/KnowledgeViewer.tsx` | 编辑/预览切换、保存、新建 |
| `frontend/src/api/knowledge.ts`(无则新建) | `createDoc` / `saveDoc` 封装 |
| `frontend/src/components/knowledge/MdEditor.tsx`(新) | CodeMirror 6 封装组件 |

## 六、模块 B:MCP 知识库接入 —— Tool 写入 + Resources 只读(问题②)

按 2025 MCP 最佳实践:**静态/半静态内容用 Resources 暴露、动态查询/动作用 Tool**。本项目知识库两者都给,让 agent「先读后写」。

### 6.1 写入 Tool:`save_to_knowledge`
```jsonc
{
  "name": "save_to_knowledge",
  "description": "把调研结果/笔记以 markdown 写入知识库(自动按命名规则落盘,无需关心文件名)",
  "inputSchema": {
    "type": "object",
    "properties": {
      "title":   { "type": "string", "description": "笔记标题(文件名由系统生成)" },
      "content": { "type": "string", "description": "markdown 正文" },
      "tags":    { "type": "array", "items": { "type": "string" } },
      "dir":     { "type": "string", "description": "可选子目录,如 '调研笔记'" }
    },
    "required": ["title", "content"]
  }
}
```
handler 参照 `handleRecordResult`(`server.ts:293`)→ `container.resolve('KnowledgeService').createDoc(...)`。

### 6.2 只读 Resources:知识库文件
- `listResources`:返回 `knowledge-base/` 下所有 md 为 `{ uri: "knowledge://<relPath>", name: <title> }`;
- `readResource(uri)`:解析 relPath → `KnowledgeService.getDoc` → 返回 md 文本。
- agent 调研前可先 `readResource` 读已有笔记,**避免重复落库**;调研结论再 `save_to_knowledge` 追加。
- SDK 支撑:`@modelcontextprotocol/sdk` 已有 Resources capability,接入模式与 Tools 平行。

### 6.3 守规则机制(为何不会乱)
- agent 只能传 `title/content/tags/dir` 四个语义字段;文件名/前缀/frontmatter/越界校验**全在 `createDoc` 内**;
- 即便 agent 传 `../../etc/passwd` 之类 `dir`,`safeResolve` 直接拒;
- 不依赖「在 prompt 里叮嘱 agent 遵守规则」——工具边界即规则。

### 6.4 进阶(可选)
- research 对话页加「把这段结论存进知识库」按钮:前端把选中对话内容 → `POST /api/knowledge/doc`(人工挑内容固化,质量更可控)。
- 知识库变大后加 `search_knowledge` Tool(全文/语义搜)。

| 文件 | 改动 |
|------|------|
| `backend/src/interfaces/mcp/server.ts` | 新增 `save_to_knowledge` 工具 + `resources` 能力(list/read handler) |
| `backend/src/interfaces/mcp/__tests__/save-to-knowledge.test.ts` | 写入工具单测(命名/越界/tags) |

## 七、模块 C:划词弹窗翻译 + 生词本(问题③,独立)

### 7.1 整体链路
```
网页选词 → content script(mouseup)
        → floating-ui 定位 → Shadow DOM 浮层(选区附近)
        → 即时翻译优先 Chrome Translator API(离线);
          不可用/需丰富释义 → 回退 PROXY_FETCH→POST /api/vocab/translate(LLM)
        → 译文回浮层;点「存生词本」→ PROXY_FETCH→POST /api/vocab
```
全程复用现有 `PROXY_FETCH` + 现有 vocab 接口,**翻译环节新增浏览器内置 API 作为快速路径**,后端仅作降级/补全。

> PNA 说明:localhost 在 Chrome 属「可信任源」,https 页面 fetch `http://localhost` 的**混合内容**一般放行;真正会拦的是 **PNA**——需 preflight 返回 `Access-Control-Allow-Private-Network: true`。走 service worker 代理可同时绕过 PNA 与页面 CSP,一劳永逸。

### 7.2 即时翻译:Chrome Translator API 优先 + 后端降级
- Chrome 138+ 内置 `Translator` Web API(on-device AI,离线、零网络往返、不耗后端 LLM 额度),与「划词即时反馈」天然契合。
- 浮层即时翻译走 `Translator`(feature-detect:`typeof Translator !== 'undefined'`;首次调用异步下载小模型,UI 显示 loading);
- 需更丰富释义(词性/定义/例句)或 API 不可用时,回退 `PROXY_FETCH`→`/api/vocab/translate`(LLM)。
- 权衡:仅 Chromium 138+,必须有降级路径;翻译质量对「看懂划词」足够。存生词本时若想要词典级释义,后台再调后端补全。

### 7.3 content script 与浮层(新增 `extension/src/content/selection-translate.ts`)
- 监听 `mouseup`:延迟 ~250ms 防抖,确认 `window.getSelection().toString().trim()` 非空且长度合理(如 ≤ 200 字,避免整段翻译)。
- **定位用 floating-ui**(`@floating-ui/dom`,~6KB):`computePosition` + `flip` + `shift`,选区贴近视口边缘时自动翻转/避让,比手写 `getBoundingClientRect` 稳。
- **Shadow DOM 隔离**:浮层挂 `attachShadow({mode:'open'})`,样式全内联 shadow 内 `<style>`;**`:host { all: initial }` 重置宿主继承的所有样式**(单靠固定字号不够,`font-family`/`color`/`line-height` 仍会漏进 shadow)。
- 关闭:点击浮层外、`ESC`、选区清空。

> 扩展框架:本项目扩展基于 **crxjs + Vite**,不引入 WXT(换构建链不划算);手写时对齐 WXT `createShadowRootUi` 的关键点(默认 `all:initial`、挂载/卸载 cleanup、CSS 内联 shadow)。

### 7.4 权限流程(避免默认高权限警告)
manifest 不静态注入划词脚本;`optional_host_permissions` 已声明 http/https。
- side panel 设置加「全局划词翻译」开关(**`chrome.permissions.request` 必须在用户手势—点击—回调内调用,脱离手势会静默失败**):
  - 开启:`chrome.permissions.request({origins:['<all_urls>']})` → 授权后 `chrome.scripting.registerContentScripts({id, matches:['<all_urls>'], js:[...]})` 注册常驻脚本;
  - 关闭:`unregisterContentScripts({ids:[id]})` + `chrome.permissions.remove`。
- 依赖:manifest 已有 `"scripting"` 权限 ✅;MV3 下动态注册是 2025+ 推荐做法,优于旧 tab 监听 + executeScript。
- 用户主动授权才注入,默认安装无高权限警告(沙拉查词等同类扩展同款)。

### 7.5 已知限制(文档注明,不阻塞 MVP)
- `chrome://`、`edge://`、Chrome 应用商店等浏览器内建页**无法注入**(扩展机制限制)。
- 跨 `iframe` 的选区先不支持(复杂,后续按需)。
- PDF/特殊渲染页选区可能取不到。

## 九、风险与待决策点

| # | 待决策 | 建议 |
|---|--------|------|
| 1 | 划词权限默认态 | **opt-in**(设置开关 + 运行时授权),避免安装时高权限警告 |
| 2 | 前端是否要「新建笔记」 | 建议要(闭环);agent 落库走 MCP 不经前端新建 |
| 3 | MCP 是否支持更新已有笔记 | MVP 只新建;后续可加 `update_knowledge(path,content)` |
| 4 | Chrome Translator API 默认开关 | 建议默认开(有 feature-detect 降级,无副作用) |

**主要风险**:
- 模块 C 浮层在各类网页兼容性(样式污染、选区定位)→ Shadow DOM + `all:initial` + floating-ui 缓解,需多真实页面回归。
- agent 落库内容质量 → 由「人工/research 对话页挑选后存」兜底,而非全自动灌库。
- Chrome Translator API 模型首次下载延迟 → UI 明确 loading,降级路径保障可用。
