# 网页剪藏扩展联调踩坑记录

> 日期：2026-06-17
> 关联：网页剪藏扩展（MV3）Task 1-8 端到端联调
> 状态：✅ 已解决，端到端跑通（采集 → 落盘 → AI 拆解 → 建任务）

## 一、背景与最终结论

网页剪藏扩展（浏览器侧栏）抓取网页内容 → 后端 AI 拆解 → 生成看板任务草案 → 用户确认后建任务。
联调时侧栏点「抓取本页」后，POST 到本地后端持续报错，经历了 **4 个串联坑**：

1. `Failed to fetch`（请求不到达后端）
2. `Payload Too Large`（413）
3. 草案**无图片**

**一句话结论**：扩展 → localhost 的请求被 Chrome **Local Network Access（PNA）**拦截（PNA 只拦预检请求），改 `text/plain` 简单请求绕过；但 base64 图片又触发 Fastify **413**（提 bodyLimit）；落盘正则只认 `image/*` mime，跨域空 mime 全丢（放宽正则）。三层叠加，逐层暴露。

```
侧栏 ClipView
  └─ chrome.scripting.executeScript 注入 content script（网页上下文）
       └─ capturePageContext：划词优先 / Readability 正文 + collectImages（img→base64）

  └─ captureToDrafts → backendPost

  └─ DraftCard 渲染：text block + <img src="localhost:3000/api/uploads/xxx">
```

## 四、坑一：CORS `credentials` 与 `origin:*` 违规

- **现象**：扩展 POST 触发 CORS 预检，预检响应被浏览器拒绝。
- **根因**：`@fastify/cors` 同时配了 `credentials: true` + `origin: '*'`。CORS 规范规定——**带 credentials 时，`access-control-allow-origin` 必须是具体值，不能是通配 `*`**，否则浏览器拒绝预检。
- **方案**：本项目跨域调用方（扩展、vite 代理后的前端）都不带 cookie，无需凭证 → `credentials: false`。
- **代码**：`backend/src/interfaces/http/server.ts`（CORS 注册块）

---

## 五、坑二：PNA 拦截预检（text/plain 绕过）—— 最耗时

- **现象**：CORS 修好后，**GET 通，POST 仍 `Failed to fetch`**（请求不到达后端）。
- **根因**：Chrome 142+ 的 **Local Network Access（PNA）**——扩展页（`chrome-extension://`）→ `localhost`（私有网络）的请求，**若触发 CORS 预检，会被 PNA 门控拦截**。`application/json` 的 POST 触发预检 → 被拦；而 GET 是简单请求（无预检）→ 不被 PNA 门控 → 能过（这是定位的关键铁证）。
- **方案**：POST 改用 `text/plain`（CORS 简单请求，不触发预检 → 不被 PNA 拦）。后端注册 `text/plain` → JSON 解析器；扩展经 service worker 代理（`host_permissions` 授权）。
- **关键依据**：Chrome 官方文档——**PNA 只门控触发预检的请求**。
- **代码**：
  - 后端 `text/plain` 解析器：`server.ts`（`addContentTypeParser`）
  - PNA 响应头 hook：`server.ts`（`onSend` 补 `access-control-allow-private-network`）
  - 扩展 service worker 代理：`extension/src/background/index.ts`
  - 扩展 `text/plain` POST：`extension/src/sidepanel/api/backend.ts`

> **注意**：把 fetch 移到 service worker（特权上下文）**并不能**免除 PNA——实测 SW 代理的 `application/json` POST 仍被拦。根因是请求方法/类型触发预检，不是上下文。

---

## 六、坑三：Fastify `bodyLimit` 1MB（base64 图片 413）

- **现象**：`text/plain` POST 通了，但报 **`Payload Too Large`（413）**。
- **根因**：Fastify 默认 `bodyLimit` 1MB。5 张图 base64 编码后（每张 150~440KB）轻松超 1MB，被拒。
- **方案**：Fastify 构造选项加 `bodyLimit: 25 * 1024 * 1024`（25MB，覆盖多图场景；图片传输优化后可调小）。
- **代码**：`backend/src/interfaces/http/server.ts`（`BODY_LIMIT_BYTES` 常量 + Fastify 构造）

---

## 七、坑四：`saveBase64Image` 正则只认 `image/*` mime（跨域空 mime 全失败）

- **现象**：413 修好后，扩展日志显示 `images: 5`（采到图），但草案无图。后端日志：`Invalid base64 image data URL`，**5 张全部落盘失败**。
- **根因**：`saveBase64Image` 原正则 `/^data:(image\/[a-z+]+);base64,(.+)$/i` 要求 mime **必须是 `image/*`**。但**跨域 fetch 页面图片时，服务器常不返回 `Content-Type`** → `blob.type` 为空 → `FileReader.readAsDataURL` 产出 `data:;base64,...`（**空 mime**）→ 正则不匹配 → 全部判"非法"丢弃。典型表现：**「采到图（images:5）但草案无图」**。
- **方案**：放宽正则，兼容空 mime / 非 image mime / 裸 base64；mime 非 `image/*` 或缺失时统一兜底为 `png`。
- **代码**：`backend/src/infrastructure/storage/saveBase64Image.ts`

---

## 八、诊断方法论：日志散落三个上下文（最值得记的教训）

定位之所以耗时，是因为**日志分散在三个互不可见的控制台**，后端根本够不着：

| 日志 | 所在控制台 | 后端可查 |
|---|---|---|
| `[clip] 采集完成`（content script） | **网页** F12（content script 跑在网页上下文） | ❌ |
| `[clip] backendPost`（侧栏） | 侧栏 DevTools（渲染进程） | ❌ |
| `imageCount`（后端路由） | 后端日志，但 `request.log.info` 是 pino info，**被默认 warn 级过滤** | ❌ |

**教训**：跨上下文系统，关键链路必须在**后端可查**的地方记录。最终在 `WebClipService` 用 `FileLogger`（不受 pino 级别控制、写文件 + 镜像 console）记录完整链路，一行日志定位所有坑：

```
[webclip] 收到剪藏请求  {imageCount:5, images:[{name,bytes,prefix}]}   ← 采集几张 + 实际格式前缀
[webclip] 图片落盘完成  {saved:5, total:5}                              ← 落盘几张
[webclip] 草案图片引用  {saved:5, referencedBlocks:5, uniqueUrls:5}     ← LLM 引用几张
```

对比前后日志，根因一目了然：
```
修复前：落盘 {saved:0, total:5}   ← 5 张全失败 → 草案无图
修复后：落盘 {saved:5, total:5}   ← 全成功    → 草案有图
```

## 十、经验教训

1. **`Failed to fetch` 不一定是网络问题** —— CORS / PNA 都表现为这个。要分清：请求没发出？被浏览器拦？还是没到达后端？看后端日志有没有收到是第一刀。
2. **PNA 只拦预检请求** —— 简单请求（GET、`text/plain` POST）绕过。这是 MV3 扩展访问 localhost 的关键。把 fetch 挪到 service worker **不能**免除 PNA。
3. **CORS `credentials:true` + `origin:*` 是非法组合** —— 带凭证必须返回具体 origin。
4. **跨域 fetch 的 `blob.type` 不可靠** —— 解析/落盘图片不要硬依赖 mime 前缀，要兜底。
5. **跨上下文系统，诊断日志要落到后端可查处** —— 扩展端日志在网页/SW 控制台，后端够不着；pino 的 info 还会被 warn 过滤。
6. **诊断代码定位后要清理** —— 本次的 SW 连通性探测（`probeSwAccess`）已达诊断目的，定稿前移除。
